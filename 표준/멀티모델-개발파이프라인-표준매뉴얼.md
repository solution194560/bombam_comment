# 멀티모델 개발 파이프라인 — 표준 매뉴얼

> 이 문서는 특정 프로젝트에 종속되지 않는다. 새 프로젝트에 다중 모델(Claude+Codex 등)
> 개발 파이프라인을 심을 때 그대로 복사해 쓴다. §5의 소스 템플릿은 봄밤 텔레그램
> 알림봇 프로젝트에서 동작 확인된 서브에이전트 7개 + 커맨드 2개를 일반화한 것이다.
> 다만 새 프로젝트에서는 권한 범위, 테스트 명령, 외부 서비스 접근 정책, 가용 모델,
> 운영 환경 차이를 반드시 재검증해야 한다 — "여기서 됐다"가 "어디서나 안전하다"는
> 뜻은 아니다.

---

## 1. 핵심 개념 — 게이트형 6단계 파이프라인

```
설계 → 검토 → 최종설계 → 구현 → 테스트 → 판정
(Fable)  (Codex)  (Fable)     (Opus)  (Codex)   (Fable)
```

- **게이트 방식** — 각 단계는 이전 단계 산출물의 판정이 통과일 때만 진행한다. 미통과면
  작업하지 않고 종료(또는 앞 단계로 되돌림).
- **문서 릴레이** — 단계 간 인계는 대화 맥락이 아니라 **파일**로 한다. 서브에이전트는
  이전 대화를 보지 못하므로 필요한 정보는 전부 지시문 또는 파일에 있어야 한다.
- **외부 모델이 실제 판정자** — 검토·테스트 단계는 오케스트레이터가 아니라 **Codex
  CLI**가 실제로 수행한다. 오케스트레이터(서브에이전트)는 구동·저장·검증만 담당한다.
- **판정은 항상 고정 모델로** — "배포 가능한가"를 세션 모델이 직접 판단하지 않고,
  판정 전용 에이전트(가벼운 모델로 고정)에 위임한다. 오케스트레이터의 낙관적 자체
  판정을 구조적으로 막는다.

### 단계별 실제 모델 배정 (기존 검증 구성 그대로 참고)

| 단계 | 에이전트 | 모델 | 산출물 | 게이트 조건 |
|---|---|---|---|---|
| 1 설계 | design-architect | **Fable** | `01_설계.md` | — |
| 2 검토 | codex-design-reviewer | **haiku(오케스트레이터) + Codex CLI(실제 검토자)** | `02_검토결과.md` | 승인 |
| 3 최종설계 | final-designer | **Fable** | `03_최종설계.md` | — |
| 4 구현 | implementer | **Opus** | 코드 | 컴파일·자체회귀 PASS |
| 5 테스트 | codex-tester | **haiku(오케스트레이터) + Codex CLI(실제 테스터)** | `04_테스트결과.md` | 통과 |
| 6 판정 | pipeline-judge | **Fable(고정)** | 판정 텍스트 | 배포가능/수정필요/판정불가 |
| (에러 시) | error-analyst | **Fable** | `05_에러분석.md` | — |

이 배정 논리는 **작업 성격에 맞춰 결정**됐다 — 설계·판단(Fable, 신중/저비용), 실제
코드 작성(Opus, 실력), 비판적 외부 검증(Codex, 독립 시각), 오케스트레이션만 하는
껍데기 단계(haiku, 저비용). 새 프로젝트에서 가용 모델이 다르면 **"이 역할에 왜 이
모델을 썼는가"의 논리만 유지**하고 실제 모델명은 교체한다.

**경량/정식 2모드**를 둔다. 파일 1~2개·스키마 변경 없는 소규모 변경은 2·3단계(검토·
최종설계)를 생략하고 `01_설계.md`를 그대로 스펙으로 넘긴다. 구조 변경·다중 파일·
스키마 변경은 전 단계를 수행한다. **구현 직전 사용자 확인 게이트는 두 모드 모두
필수**로 둔다 — 이게 없으면 파이프라인이 아무리 정교해도 사람이 통제권을 잃는다.

## 2. 각 단계 에이전트 설계 시 반드시 넣을 장치

실제 운용에서 빠지기 쉽고, 빠지면 파이프라인의 신뢰도가 크게 떨어지는 항목들이다.
아래 §5 템플릿에는 이미 반영되어 있다.

### 2-1. 설계 단계 — "확인 요청 사항"을 [사용자]/[검토자]로 구분

설계 문서 끝에 미해결 질문 절을 두되, 사용자가 결정할 것과 검토자(2단계 모델)가
사실 대조로 답할 수 있는 것을 구분해 표시한다. 이렇게 하면 검토 단계가 "칭찬/통과"에
그치지 않고 실제로 답을 내야 하는 일감을 갖게 된다.

### 2-2. 검토 단계 — 질문에 명시적으로 답하게 강제

검토 지시문에 "설계 문서의 미해결 질문 각각에 명시적으로 답하라"는 관점을 반드시
포함한다. 안 그러면 검토자가 질문을 무시하고 일반론만 쓰는 경우가 잦다.

### 2-3. 테스트 단계 — scope creep(범위 밖 무단 변경)을 결함으로 취급

구현자의 보고를 신뢰하지 않는다. 테스트/검증 단계에서 **변경 파일 목록을 스펙 문서의
"변경 범위"와 직접 대조**해, 스펙에 없는 파일 수정이나 요청 밖 변경이 있으면 결함으로
보고하게 한다.

### 2-4. 판정 단계 — 철칙 2개

- **테스트가 실패하면, Codex가 자체적으로 "통과"라고 적었더라도 최종 판정은 실패다.**
  하위 모델의 낙관적 자기평가를 상위 판정이 뒤집을 수 있어야 한다.
- **판정 모델은 Codex의 "종합 판정" 한 줄을 그대로 베끼지 않는다.** 시나리오별
  통과/실패와 원래 성공 기준을 직접 대조해 독립적으로 판정한다. 애매하면 "수정 필요"
  쪽으로 판단한다(낙관적 반올림 금지).

### 2-5. 실패 시 흐름

되돌림 방향을 명시해둔다 — "검토 반려 → 설계로", "테스트 실패 → 구현으로", "판정
불가 → 테스트 재실행". 실패 원인 분석을 담당하는 에이전트(error-analyst)를 별도로
두면 재작업 방향이 더 명확해진다. 이 에이전트는 개발 파이프라인 실패뿐 아니라
**실서비스 운영 중 에러**에도 공용으로 쓸 수 있게 설계한다(§5의 fix-error 참조).

### 2-6. 경량 모드에서도 생략하면 안 되는 최소 검증

경량 모드는 검토·최종설계 단계를 생략하는 것이지, 검증 자체를 생략하는 게 아니다.
검토가 빠지는 만큼 아래는 오히려 더 엄격히 지킨다.

- 설계 문서(`01_설계.md`)에 변경 파일 목록과 성공 기준을 명시한다.
- 구현 전 사용자가 설계 요약을 확인한다(구현 승인 게이트).
- 구현 후 문법/컴파일 검증을 수행한다.
- 테스트 단계에서 성공 기준을 최소 1개 이상 **직접** 검증한다(간접 추정 금지).
- 판정 단계에서 `pipeline-judge`가 테스트 결과와 성공 기준을 대조한다 — 경량이라고
  판정 단계까지 생략하지 않는다.

### 2-7. Codex 실패 시 폴백 보고 형식

검토·테스트 담당(codex-design-reviewer, codex-tester)이 Codex를 1회 재시도하고도
실패하면(미설치·인증 만료·타임아웃 등), 조용히 자체 판단으로 넘어가지 않는다.
결과 문서 맨 위에 아래 형식을 반드시 적어, 이 산출물의 신뢰도가 Codex 정상 실행 때와
다르다는 것을 다음 단계(사용자·판정 에이전트)가 알 수 있게 한다.

```markdown
> Codex 상태: 실패
> 실패 사유: <명령 실패 / 인증 실패 / 타임아웃 등>
> 폴백 수행자: <이 서브에이전트가 §2의 점검 관점으로 직접 수행>
> 신뢰도: 제한적 (외부 독립 검증 없이 자체 판단)
```

이 표시가 있으면 오케스트레이터는 사용자에게 "이번 검토/테스트는 Codex가 아니라
자체 폴백으로 수행됐다"는 사실을 반드시 알린다.

## 3. 참고 도구 설명 (필요해지면 도입)

파이프라인 자체와는 별개로, Claude+Codex 등 페어 운용 시 도움이 될 수 있는 도구.
지금 당장 설치할 필요는 없고, 아래 도입 조건이 실제로 성립할 때 검토한다.

### knot — 벤더 중립 지식 vault

Claude·Codex 등 서로 다른 CLI 도구가 같은 지식을 공유하기 위한 평문 마크다운 폴더
체계다. **데이터 층(vault 폴더 자체)은 도구 무관하게 그냥 공유된다** — 별도 변환이
필요 없는 순수 마크다운이기 때문. 하지만 **지시 층은 도구별로 각각 배선해야 한다** —
CLAUDE.md에만 자동 설치되므로, Codex 등 다른 CLI를 페어로 쓴다면 그 도구의 지시 파일
(예: AGENTS.md)에도 "`$KNOT_VAULT/wiki/` 참고, `ingest.md` 규약대로 저장" 같은 문구를
직접 추가해야 양쪽이 같은 지식을 실제로 활용한다.

**도입 조건**: vault 폴더를 실제로 만들고 `$KNOT_VAULT` 환경변수를 설정할 계획이 있을 때.

### codexbar + usage-coach — 사용량 계측·자율 루프 정지

codexbar(macOS 메뉴바 앱+CLI)가 Claude·Codex 등 57개 이상 서비스의 사용량·리셋
시각을 수집하고, usage-coach(`coach`)가 그 데이터로 "지금 큰 작업을 돌려도 되는지"를
판단해 Stop 훅으로 자율 루프를 멈춘다. `coach guard on/off/status`로 런타임 전환
가능하며 미설치 시 fail-open.

**도입 조건**: `/goal`류로 수 시간~수 일 걸리는 **무인 자율 작업**을 실제로 돌리기
시작할 때. 사람이 매 단계 승인하는 대화형 파이프라인에는 낮은 우선순위.

## 4. 새 프로젝트 적용 체크리스트

1. §5의 서브에이전트 7개 + 커맨드 2개를 대상 프로젝트의 `.claude/agents/`,
   `.claude/commands/`로 복사한다.
2. 토큰 치환표(§5-0)대로 모든 복사본에서 `{{TOKEN}}`을 찾기·바꾸기 한다.
3. §2의 6가지 장치(확인요청 구분, 검토자 답변 강제, scope creep 감사, 판정 철칙 2개,
   경량모드 최소검증, Codex 폴백 보고 형식)가 전부 살아있는지 확인한다 — 템플릿에 이미
   반영돼 있으니 토큰 치환 중 실수로 지우지 않도록 주의. `{{SAFETY_RULES}}`는 특히
   "프로젝트의 위험 지점에 맞게 다시 쓰는" 항목이라 복사만 하고 채우지 않으면 무의미하다.
4. 경량/정식 2모드와 구현 직전 사용자 확인 게이트는 어떤 경우에도 생략하지 않는다.
5. Codex CLI 설치·로그인 여부(`codex --version`, `codex login status`)를 확인한다
   (2·5단계 전제). 없으면 해당 단계는 세션 모델이 직접 수행하는 폴백으로 대체하되
   품질 저하를 사용자에게 알린다.
6. knot·codexbar/usage-coach는 §3의 도입 조건이 실제로 성립할 때만 추가한다.

## 5. 전체 소스 템플릿 (복사용)

### 5-0. 토큰 치환표

| 토큰 | 의미 | 이 프로젝트(봄밤)의 실제 값 |
|---|---|---|
| `{{PROJECT_NAME}}` | 프로젝트 이름 | 봄밤 텔레그램 알림봇 |
| `{{KEY_FILES}}` | 설계·구현·에러분석 시 참고할 핵심 소스 파일 목록 | `ridi_collector.py`, `bot.py`, `notify.py`, `settings.json`, `Dockerfile`, `entrypoint.sh` |
| `{{PITFALL_SECTION}}` | CLAUDE.md 내 "알려진 함정" 절 이름 | 알고 있어야 할 함정들 (재발 방지) |
| `{{COMPILE_CMD}}` | 문법/컴파일 검증 명령 | `python3 -m py_compile <파일>` |
| `{{LOCAL_TEST_SECTION}}` | CLAUDE.md 내 로컬 테스트 절차 절 이름 | 로컬 테스트 실행법 |
| `{{CODEX_BIN}}` | Codex CLI 실행 파일 경로 | `/Applications/Codex.app/Contents/Resources/codex` |
| `{{RUNTIME_CONFLICT_NOTE}}` | [선택] 로컬 테스트 시 공유 자원 충돌 주의사항(없으면 해당 문구 삭제) | 로컬 테스트 컨테이너와 NAS가 같은 텔레그램 토큰 사용 → 동시 실행 시 409 Conflict |
| `{{DEPLOY_ARTIFACT_BUILD}}` | [선택] 배포 산출물 생성 단계(1차 승인 — 로컬 산출물만 만듦) | `bombam_synology.zip` 재생성 |
| `{{DEPLOY_ARTIFACT_APPLY}}` | [선택] 실제 운영 환경 반영 단계(2차 승인 — 여기서부터 되돌리기 어려움) | 시놀로지 NAS Container Manager에서 재빌드·재시작 |
| `{{PROD_ERROR_SOURCE_NOTE}}` | [선택] 실서비스 에러 정보 수집 방법(운영 환경이 없으면 삭제) | SSH 불가 — Synology Container Manager GUI 로그 화면만 가능 |
| `{{COMMENT_LANGUAGE}}` | 코드 주석/로그 메시지 언어 | 한국어 |
| `{{SAFETY_RULES}}` | 전 단계 공통 안전 제약(프로젝트 위험 지점에 맞게 다시 쓸 것) | 아래 예시 |

**`{{SAFETY_RULES}}` 예시 (봄밤 프로젝트)**
```markdown
- 로그인 계정 정보(account.json)·텔레그램 토큰(telegram.json)을 출력하거나 커밋하지 않는다.
- 실제 시놀로지 NAS 재배포, 컨테이너 재시작, 텔레그램 실제 발송은 사용자 승인 없이 하지 않는다.
- 리디북스 등 외부 사이트 접근은 저빈도·단일 세션으로 하고, 병렬 스크래핑·짧은 간격
  반복 요청·실패 직후 무한 재시도를 하지 않는다.
```
일반 프로젝트 예 — "프로덕션 DB 접근/마이그레이션 자동 적용 금지", "비밀키·API 토큰

  명시적으로 허용한 3·4·5단계에 한한다(그 외 임의 생략 금지). 서브에이전트가 실패하면 원인을
  파악해 같은 단계를 재시도하고, 2회 실패 시 사용자에게 상황을 보고하고 지시를 기다린다.
- 각 단계 시작 시 사용자에게 "N단계: ○○ 진행 중 (모델: ○○)"을 한 줄로 알린다.
  판정 단계 모델은 `pipeline-judge`다 — 세션 모델이 무엇이든 판정은 항상 고정 모델로 실행된다.
- 모든 서브에이전트는 `{{SAFETY_RULES}}`를 준수한다. 오케스트레이터도 배포 관련
  실제 조치는 위 2단계 승인 없이는 수행하지 않는다.

## 단계 (반드시 순차 실행)

1. **에러 분석** — `error-analyst` 서브에이전트 실행 → `05_에러분석.md` 생성 확인.
2. **⛔ 사용자 확인 게이트** — 근본 원인과 수정 방향을 요약해 보여주고
   진행 여부를 묻는다 (AskUserQuestion). CLAUDE.md의 "작업 전 리뷰" 규칙이 이 게이트다.
   - 분석 결과 "코드 수정 불필요"(설정 정리, 재시작 등으로 끝나는 경우)면 구현 단계로
     가지 않고, 사용자가 할 조치를 단계별로 안내하고 종료.
   - 원인이 불확실하면(6장 미해결 질문 있음) 사용자에게 추가 정보를 요청하고 1단계 재실행.
3. **수정 스펙 정리** — 분석의 "4. 수정 방향"을 implementer가 그대로 구현할 수 있도록
   `03_최종설계.md` 형식(변경 파일 목록 + 파일별 상세 + 테스트 계획)으로 메인 세션이 직접
   정리해 저장한다. (규모가 크면 `final-designer` 서브에이전트에 위임해도 된다.)
4. **구현** — `implementer` 서브에이전트 실행. 변경 파일 목록을 사용자에게 요약.
5. **테스트** — [선택] {{RUNTIME_CONFLICT_NOTE}}가 있다면 실행 전 사용자에게 상기시키고
   확인을 받은 뒤, `codex-tester` 서브에이전트 실행 → `04_테스트결과.md` 확인. 테스트
   시나리오에는 **원래 에러의 재현 조건이 반드시 포함**되어야 한다(고쳐졌는지 확인이 목적이므로).
6. **판정 & 마무리** — 판정은 세션이 직접 하지 말고 `pipeline-judge` 서브에이전트에
   맡긴다(작업 폴더·`04_테스트결과.md`·`03_최종설계.md` 경로 전달). 받은 판정으로 산출물 링크 정리.

   - "배포 가능"이면: [선택] **1차 승인** {{DEPLOY_ARTIFACT_BUILD}} → **2차 승인**
