## 1. 전체 평가

개선본은 이전 `agent-pipeline` 문서보다 실사용성이 좋아졌다. 특히 검토 단계의 `read-only` 샌드박스, 경량/정식 모드 분리, 사용자 확인 게이트, scope creep 감사, 최종 판정 전용 모델은 좋은 개선이다.

다만 아직 세 가지 구조적 보완이 필요하다.

1. 게이트가 실제 실행 레벨에서 강제되지 않고 문서 규칙에 머문다.
2. 경량 모드 단계 번호가 실제 단계 목록과 충돌한다.
3. 테스트 단계에서 `danger-full-access`를 기본 호출 예시로 둔 점은 위험하다.

### [P1] 테스트 단계 `danger-full-access` 기본값 낮추기

위치:
- 원본 355-364행: `codex-tester` 실행 명령

문제:
- 테스트 단계가 기본 예시에서 `--sandbox danger-full-access`를 사용한다.
- 테스트가 빌드 산출물이나 임시 파일을 만들 수는 있지만, 전체 파일 시스템 접근을 기본으로 두면 위험하다.
- 특히 로그인 세션, 토큰, 외부 서비스, 스크래핑 관련 프로젝트에서는 과한 권한이다.

권장 수정:

```bash
{{CODEX_BIN}} exec \
  --skip-git-repo-check \
  --sandbox workspace-write \
  -C "<프로젝트 루트 절대경로>" \
  -o "<테스트결과 저장 절대경로>" \
  "<테스트 프롬프트>" </dev/null
```

권장 문구 추가:

```markdown
- 기본 샌드박스는 `workspace-write`로 둔다.
- 프로젝트 루트 밖 쓰기, 외부 배포, 프로덕션 DB 접근, 실제 메시지/댓글/티켓 쓰기는
  테스트 단계에서도 금지한다.
- `danger-full-access`는 테스트가 프로젝트 밖 시스템 자원에 접근해야 하는 예외 상황에서만
  사용자에게 이유를 설명하고 승인받은 뒤 사용한다.
```

문제:

- 이전 문서의 `{{SAFETY_RULES}}` 개념이 빠지면서 외부 쓰기, 비밀키, 프로덕션 접근, 스크래핑 빈도 같은 위험 제어가 약해졌다.

권장 수정:

토큰 치환표에 추가:

```markdown
| `{{SAFETY_RULES}}` | 전 단계 공통 안전 제약 | 외부 쓰기 금지, 비밀키 출력 금지, 프로덕션 DB/배포 금지 등 |
```

각 에이전트 규칙에 추가:

```markdown
## 공통 안전 제약
{{SAFETY_RULES}}
```

봄밤/댓글 수집 계열 프로젝트 예시:

```markdown
- 로그인 쿠키, 토큰, 계정 정보, 개인 메시지 내용을 출력하거나 커밋하지 않는다.
- 실제 댓글 작성, 메시지 발송, 알림 발송, 배포는 사용자 승인 없이는 하지 않는다.
- 리디 등 외부 사이트 접근은 저빈도·단일 세션 중심으로 수행한다.
- 병렬 브라우저, 짧은 간격 반복 요청, 실패 직후 무한 재시도, 전체 히스토리 반복 재수집을 금지한다.
- 테스트는 가능하면 저장된 샘플 데이터와 오프라인 fixtures로 먼저 수행한다.
```

문제:
- 게이트 규칙은 좋아졌지만 여전히 자연어 지시다.
- 산출물이 실제로 승인/통과 상태인지 사람이 읽고 판단해야 한다.
- 오래된 파일, 잘못된 폴더, 다른 작업의 `04_테스트결과.md`를 참조하는 실수가 생길 수 있다.

권장 추가:

각 산출물 상단에 다음 frontmatter를 붙인다.

```yaml
---
pipeline:
  task_id: "YYYYMMDD_slug"
  stage: "test"
  status: "passed"
  spec_path: "/absolute/path/to/03_최종설계.md"
  created_at: "2026-07-08T09:00:00+09:00"
  model: "codex"
---
```

상태값 예시:

```markdown
- 설계: drafted
- 검토: approved / revise_required / failed_to_review
- 최종설계: finalized
- 구현: implemented / compile_failed
- 테스트: passed / failed / skipped / invalid
- 판정: deployable / needs_fix / inconclusive
```

## 3. 원본에 바로 넣기 좋은 요약 패치 문구

아래 문단은 원본 §2 뒤나 §4 체크리스트 앞에 추가하기 좋다.

```markdown
### 공통 안전 및 강제 규칙

- 각 산출물은 가능하면 YAML frontmatter로 `task_id`, `stage`, `status`, `spec_path`,
  `created_at`을 기록한다. 다음 단계는 이전 단계 status가 기대값이 아니면 진행하지 않는다.

- 모든 에이전트는 `{{SAFETY_RULES}}`를 준수한다. 외부 쓰기, 프로덕션 DB/배포, 비밀키 출력,

  판정은 생략하지 않는다.
```

## 4. 최종 판단

현재 개선본은 이전 버전보다 한 단계 좋아졌다. 그대로 써도 “AI가 무작정 구현하는 문제”는 상당히 줄일 수 있다.

다만 표준 매뉴얼로 배포하려면 다음 3개는 먼저 반영하는 것을 권장한다.

1. 경량 모드 단계 번호 충돌 수정
2. 테스트 단계 기본 샌드박스를 `danger-full-access`에서 낮추기
3. `{{SAFETY_RULES}}`와 산출물 상태 frontmatter 추가
