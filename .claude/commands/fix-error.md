다음 에러 수정 파이프라인을 순서대로 실행하라. 에러 정보: $ARGUMENTS

## 준비
- 오늘 날짜와 에러 요약으로 작업 폴더를 정한다: `docs/설계/YYYYMMDD_fix_<짧은-영문-슬러그>/`
- 산출물 경로: `05_에러분석.md`, `03_최종설계.md`(분석의 "수정 방향"을 스펙으로 정리),
  `04_테스트결과.md` (모두 이 폴더 안)
- 각 서브에이전트에게는 **절대경로**와 에러 정보 전문을 지시문에 포함해 넘긴다
  (서브에이전트는 이 대화를 못 본다).
- 에러 정보가 로그 없이 증상뿐이면, 분석 전에 사용자에게 NAS Container Manager 로그
  확인 방법(컨테이너 선택 → 세부 정보 → 로그 탭)을 안내하고 로그를 받는 것을 우선 시도한다.

## 단계 (반드시 순차 실행)

1. **에러 분석** — `error-analyst` 서브에이전트 실행 → `05_에러분석.md` 생성 확인.
2. **⛔ 사용자 확인 게이트** — 근본 원인과 수정 방향을 초보자 눈높이로 요약해 보여주고
   진행 여부를 묻는다 (AskUserQuestion). CLAUDE.md의 "작업 전 리뷰" 규칙이 이 게이트다.
   - 분석 결과 "코드 수정 불필요"(예: 설정 파일 정리, 컨테이너 재시작, Singleton 락 삭제로
     끝나는 경우)면 구현 단계로 가지 않고, 사용자가 GUI에서 할 조치를 단계별로 안내하고 종료.
   - 원인이 불확실하면(6장 미해결 질문 있음) 사용자에게 추가 정보를 요청하고 1단계 재실행.
3. **수정 스펙 정리** — 분석의 "4. 수정 방향"을 implementer가 그대로 구현할 수 있도록
   `03_최종설계.md` 형식(변경 파일 목록 + 파일별 상세 + 테스트 계획)으로 메인 세션이 직접
   정리해 저장한다. (규모가 크면 `final-designer` 서브에이전트에 위임해도 된다.)
4. **구현** — `implementer` 서브에이전트 실행. 변경 파일 목록을 사용자에게 요약.
5. **테스트** — 로컬 Docker 테스트 전 **NAS 봇 정지 필요**(같은 텔레그램 토큰 → 409 충돌)를
   사용자에게 상기시키고 정지 확인을 받은 뒤, `codex-tester` 서브에이전트 실행 →
   `04_테스트결과.md` 확인. 테스트 시나리오에는 **원래 에러의 재현 조건이 반드시 포함**되어야
   한다(고쳐졌는지 확인이 목적이므로).
6. **마무리 보고** — 종합 판정과 산출물 링크 정리.
   - "수정 필요"면: 원인 요약 후 4~5단계 재실행 제안. 2회 실패 시 사용자에게 보고하고 대기.
   - "배포 가능"이면: `bombam_synology.zip` 재생성 → 시놀로지 재배포 절차(CLAUDE.md 참조)
     진행 여부를 묻는다. zip 재생성은 사용자가 승인한 경우에만.
   - CLAUDE.md "함정 목록"에 추가할 항목이 분석에서 제안됐으면, 사용자 승인 후 CLAUDE.md에 반영.

## 규칙
- 단계를 건너뛰거나 순서를 바꾸지 않는다 (2단계에서 "코드 수정 불필요" 분기 제외).
- 각 단계 시작 시 사용자에게 "N단계: ○○ 진행 중 (모델: ○○)"을 한 줄로 알린다.
