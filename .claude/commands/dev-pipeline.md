다음 개발 파이프라인을 순서대로 실행하라. 요구사항: $ARGUMENTS

## 실행 모드 (0단계: 규모를 판단해 하나 선택)
요구사항의 규모·위험도를 보고 모드를 정한다. 애매하면 사용자에게 한 줄로 물어 확정한다.

- **경량 모드** — 버그 수정·소규모 변경(대략 파일 1~2개, settings 스키마 변경 없음, 기존
  함정 재발 대응 등). **설계 검토(3단계)·최종설계(4단계)와 그에 딸린 사용자 확인
  게이트 2(5단계)를 생략**한다. 대신 2단계(사용자 확인 게이트 1)를 "구현 승인 게이트"로
  겸용해, `1 설계 → 2 구현 승인 게이트 → 6 구현 → 7 테스트 → 8 판정` 순으로 진행한다.
  구현(6)에는 `03_최종설계.md`가 없으므로 **`01_설계.md`를 스펙 문서로** 넘기고, 테스트(7)의
  codex-tester 지시문에는 `reasoning effort=low`를 명시한다.
- **정식 모드** — 구조 변경·다중 파일·settings 스키마 변경·위험도 높음. 아래 1~8단계를 **전부** 수행한다.

두 모드 모두 **구현 직전 게이트는 반드시 유지**한다(CLAUDE.md "작업 전 리뷰" 규칙). 경량 모드는
그 게이트를 1단계 직후에 둔다(설계 요약 → 구현 승인). 선택한 모드를 첫 알림에 함께 통지한다.

## 준비
- 오늘 날짜와 요구사항으로 작업 폴더를 정한다: `docs/설계/YYYYMMDD_<짧은-영문-슬러그>/`
- 산출물 경로: `01_설계.md`, `02_검토결과.md`, `03_최종설계.md`, `04_테스트결과.md`,
  (테스트 실패 시) `05_에러분석.md` (모두 이 폴더 안)
- 각 서브에이전트에게는 **절대경로**와 요구사항 전문을 지시문에 포함해 넘긴다 (서브에이전트는 이 대화를 못 본다).

## 단계 (반드시 순차 실행, 각 단계 산출물 확인 후 다음 진행)

1. **설계** — `design-architect` 서브에이전트 실행 → `01_설계.md` 생성 확인.
2. **⛔ 사용자 확인 게이트 1** — 설계 요약과 미해결 질문을 사용자에게 보여주고
   진행 여부를 묻는다 (AskUserQuestion). 사용자가 수정을 원하면 반영 후 재확인.
   (경량 모드에서는 이 게이트가 곧 "구현 승인 게이트"이며, 통과 시 3~5단계를
   건너뛰고 바로 6단계로 간다.)
3. **설계 검토** — `codex-design-reviewer` 서브에이전트 실행 → `02_검토결과.md` 생성 확인.
4. **최종설계** — `final-designer` 서브에이전트 실행 → `03_최종설계.md` 생성 확인.
5. **⛔ 사용자 확인 게이트 2** — 검토 반영 내역(수용/반려)과 최종설계 요약을 보여주고
   구현 진행 여부를 묻는다. CLAUDE.md의 "작업 전 리뷰" 규칙이 이 게이트에 해당한다.
6. **구현** — `implementer` 서브에이전트 실행. **스펙 문서 경로**로 정식 모드는 `03_최종설계.md`,
   경량 모드는 `01_설계.md`를 넘긴다. 보고받은 변경 파일 목록을 사용자에게 요약.
7. **테스트** — 로컬 Docker 테스트 전 NAS 봇과의 텔레그램 409 충돌 가능성을 사용자에게
   상기시킨 뒤, `codex-tester` 서브에이전트 실행 → `04_테스트결과.md` 확인.
   (경량 모드면 지시문에 `reasoning effort=low`를 명시한다.)
8. **판정 & 마무리** — 판정은 **세션이 직접 하지 말고** `pipeline-judge` 서브에이전트(Fable)에
   맡긴다. 작업 폴더 절대경로와 `04_테스트결과.md`(있으면 `03_최종설계.md`/`01_설계.md`) 경로를
   넘겨 **배포가능/수정필요/판정불가**를 받는다. 받은 판정으로 종합 보고(산출물 파일 링크 포함):
   - "수정 필요"면: `error-analyst` 서브에이전트를 실행해 `05_에러분석.md`를 생성하고,
     근본 원인과 수정 방향을 사용자에게 요약한 뒤, 수정 후 6~7단계 재실행을 제안한다.
     (에이전트에게는 04·저장 경로·변경 파일 목록과 pipeline-judge가 준 실패 힌트를 넘긴다.)
   - "배포 가능"이면: **1차 승인** — `bombam_synology.zip` 재생성 진행 여부를 사용자에게
     묻는다(아직 로컬 산출물만 만드는 단계). 이어서 **2차 승인** — 시놀로지 NAS 재배포 절차
     (CLAUDE.md 참조, 여기부터 실서비스에 반영되어 되돌리기 어려움) 진행 여부를 **별도로**
     다시 묻는다. zip 재생성 승인이 NAS 배포까지 자동으로 허가하지 않는다.
   - "판정 불가"면: 테스트가 유효하지 않은 것이므로 7단계를 다시 수행한다.

## 규칙
- **정식 모드**에서는 단계를 건너뛰거나 순서를 바꾸지 않는다. 단계 생략은 오직 경량 모드가
  명시적으로 허용한 3·4·5단계에 한한다(그 외 임의 생략 금지). 서브에이전트가 실패하면 원인을
  파악해 같은 단계를 재시도하고, 2회 실패 시 사용자에게 상황을 보고하고 지시를 기다린다.
- 각 단계 시작 시 사용자에게 "N단계: ○○ 진행 중 (모델: ○○)"을 한 줄로 알린다.
  판정 단계 모델은 `pipeline-judge`(Fable)다 — 세션 모델이 무엇이든 판정은 항상 Fable로 고정된다.
- 모든 서브에이전트는 아래 "공통 안전 제약"을 준수한다. 오케스트레이터도 배포 관련 실제
  조치는 위 2단계 승인 없이는 수행하지 않는다.

## 공통 안전 제약 (전 단계 공통)
- 로그인 계정 정보(`account.json`)·텔레그램 토큰(`telegram.json`)을 출력하거나 커밋하지 않는다.
- 실제 시놀로지 NAS 재배포, 컨테이너 재시작, 텔레그램 실제 발송(테스트 목적 이외)은
  사용자 승인 없이 하지 않는다.
- 리디북스 등 외부 사이트 접근은 저빈도·단일 세션으로 하고, 병렬 스크래핑·짧은 간격
  반복 요청·실패 직후 무한 재시도를 하지 않는다.
