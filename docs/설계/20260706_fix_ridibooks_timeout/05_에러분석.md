## 1. 증상

발생 환경: **시놀로지 NAS(DS224+, Celeron J4125, RAM 2GB)의 bombam 컨테이너** (실서비스).

```

```
- 2026/07/04 약 18:00 ~ 21:41 반복

※ SSL 핸드셰이크(handshake) = 암호화 통신(HTTPS)을 시작하기 전 서버와 나누는 첫 인사 절차.
"핸드셰이크 타임아웃"은 서버 주소는 찾았고(TCP 연결까지 됨) 그 다음 암호화 협상 단계에서

- 2026/07/05 11:15~11:46 — **31개 작품 전부** 타임아웃 (동일 패턴)
- 2026/07/04 17:57~18:00 — 일부 작품만 타임아웃

## 2. 근본 원인

### 결론: 코드 버그가 아니라 **NAS 쪽 외부 네트워크 품질의 일시적 저하**가 근본 원인

근거:

1. **서로 무관한 두 목적지가 동시에 실패했다.**
   - 에러 1은 `api.telegram.org`(bot.py `_api()`, bot.py:40~51, urllib 사용)
   - 에러 2는 `ridibooks.com`(ridi_collector.py `_open_book_and_scroll()`, ridi_collector.py:323
     의 `page.goto(..., timeout=60000)`, Playwright/Chromium 사용)
   - 통신 라이브러리도 다르고(urllib vs Chromium) 목적지 서버도 다른데 같은 시간대에 함께
     실패했다 → 공통 구간은 "NAS에서 인터넷으로 나가는 길"뿐이다.
2. **같은 코드가 13:31에는 완벽히 성공했다.** 07/05 08:12 실패 → 11:15 실패 → 13:31 성공
   사이에 코드/설정 변경이 없었으므로, 변한 것은 환경(네트워크 상태)이다.
3. **핸드셰이크 타임아웃이라는 실패 지점이 특징적이다.** DNS(주소 찾기)와 TCP 연결은 됐는데
   그 다음 단계에서 멈췄다 → 회선 혼잡·패킷 손실(데이터 조각이 중간에 유실되는 현상) 시
   전형적으로 나타나는 증상이다. 봇 감지/차단(Cloudflare)이라면 타임아웃이 아니라
   차단 페이지가 로드되는 형태로 나타난다.
4. **로컬 요인 배제 확인 완료.** 맥의 로컬 테스트 컨테이너(bombam_test)는
   07/04 15:27~15:52 KST에만 돌았고(docker inspect로 확인) 에러 시간대(07/04 18:00 이후,
   07/05 전일)에는 꺼져 있었다. 로컬 컨테이너 로그의 텔레그램 오류도 `409 Conflict`(토큰
   중복 폴링)뿐, SSL 타임아웃은 0건이다. → 함정 목록의 409 문제와는 별개 사안.

### 보조 가설 (중첩 가능): NAS 리소스 포화가 증상을 악화

- 수집 중에는 Chromium(실제 창 모드 + Xvfb)이 2GB RAM NAS에서 메모리·CPU를 크게 쓴다.
  스왑(RAM 부족 시 디스크를 메모리처럼 쓰는 것)이 심해지면 모든 통신이 함께 느려져
  SSL 타임아웃이 동반될 수 있다.
- 다만 **07/04 저녁 18:00~21:41(수집이 돌지 않던 시간)에도 SSL 오류가 났으므로 리소스만으로는
  설명이 안 된다.** 이 시간대에 NAS의 다른 예약 작업(Hyper Backup, 미디어 인덱싱 등)이
  돌았는지는 6장에서 사용자 확인이 필요하다.

### 코드가 원인은 아니지만, 피해를 키운 코드 약점 3가지

1. **실패해도 끝까지 도는 구조 (fail-fast 없음)** — `gather_recent()`(ridi_collector.py:698~731)는
   작품별 예외를 잡고 다음 작품으로 넘어가기만 한다. 네트워크가 완전히 죽어 있어도
   31개 × 60초 = **31분을 헛돌고**, 그동안 Chromium이 메모리를 계속 점유한다.
2. **조용한 실패 (silent failure)** — 전부 실패하면 `rows`가 빈 목록이 되고,
   4_매일알림.py:44~45에서 `NOTIFY_WHEN_EMPTY=False`라 **아무 알림도 안 간다.**
   사용자 입장에선 "새 댓글이 없는 날"과 "수집이 통째로 실패한 날"을 구별할 수 없다.
3. **getUpdates 실패 시 백오프 없음** — bot.py:203~237 메인 루프는 `_api()`가 None을 돌려주면
   `time.sleep(1)` 후 곧바로 재시도한다. 네트워크 장애 중에 오류 로그가 수백 줄씩 쌓이고
   불필요한 재접속을 반복한다. (백오프 = 실패할수록 재시도 간격을 점점 늘리는 기법)

CLAUDE.md 함정 목록 1~10번 대조 결과: **해당 항목 없음(재발 아님).** 새 유형이므로
5장에서 신규 항목을 제안한다.

## 3. 재현 조건

- **원 조건**: NAS의 인터넷 회선 품질이 일시적으로 나빠진 상태(패킷 손실/혼잡)에서
  봇 폴링·수집이 돌면 발생. 회선 문제라 개발자가 임의로 재현할 수는 없다.
- **로컬 모의 재현 방법** (수정안 검증용): Docker의 `--add-host`로 리디북스 주소를
  응답 없는 가짜 IP로 돌리면 goto 타임아웃을 그대로 흉내낼 수 있다.
  ```bash
  docker run ... --add-host ridibooks.com:10.255.255.1 --add-host api.telegram.org:10.255.255.1 ...
  ```
  (10.255.255.1 = 어디에도 연결되지 않는 내부용 주소 → 접속 시도가 하염없이 기다리다 타임아웃)
- 검증 시나리오는 4장 끝의 "테스트 계획" 참고.

## 4. 수정 방향

근본 원인이 환경(회선)이므로 "에러를 0으로" 만드는 코드는 불가능하다. 목표는
**(a) 장애 때 낭비를 줄이고, (b) 실패를 사용자에게 알리고, (c) 회복되면 스스로 정상화**되게
만드는 것이다.

### 수정 1 — 연속 실패 시 조기 중단 (ridi_collector.py `gather_recent`)
- 작품 수집 루프에서 **연속 실패 카운터**를 두고, 예: 4작품 연속 `Page.goto` 타임아웃이면
  루프를 중단하고 "네트워크 장애로 중단" 상태를 반환한다.
- 반환값 확장: 현재 `return rows, cutoff` → `return rows, cutoff, stats`
  (stats = {"total": N, "failed": M, "aborted": bool}) 형태 권장.
  기존 호출부(4_매일알림.py:23, `run_recent_days`)도 함께 수정.
- 장점: 31분 낭비 → 약 4분으로 단축. 단점: 특정 작품 몇 개만 일시적으로 느린 경우도
  중단될 수 있으나, "연속" 조건이라 실제로는 드묾. **권장.**

### 수정 2 — 작품별 1회 재시도 (ridi_collector.py `_open_book_and_scroll` 또는 `_collect_reviews`/`gather_recent`)
- `page.goto` 타임아웃 시 10초 쉬고 1회만 재시도. 첫 시도는 timeout을 60초 → 30초로 줄여
  "30초 + 재시도 30초"로 총 소요는 비슷하게 유지하면서 일시적 지연에는 강해진다.
- 수정 1과 병행 권장 (재시도까지 실패한 것만 '연속 실패'로 집계).

### 수정 3 — 수집 실패 알림 (4_매일알림.py)
- 수정 1의 stats를 받아, `aborted`이거나 실패율이 예: 50% 이상이면
  `"⚠️ 수집 실패: 31개 중 31개 접속 실패(네트워크 문제 추정). 나중에 /run 으로 다시 시도해 주세요."`
  를 소유자(자동 실행) 또는 /run 누른 사람에게 전송.
- 자동 실행 실패 시 `last_run_date` 처리: 현재 bot.py:179~181은 성공/실패와 무관하게
  기록해 **그날 자동 재시도가 없다.** 무한 재시도 루프 위험이 있으므로 자동 재시도보다는
  "실패 알림 + 사용자가 /run"이 단순하고 안전하다. **실패 알림 방식 권장.**
  (대안: 실패 시 1시간 뒤 1회만 자동 재시도 — 구현 복잡도가 올라가므로 2순위.)

### 수정 4 — getUpdates 실패 시 지수 백오프 (bot.py `main` 루프)
- `_api("getUpdates", ...)`가 None을 돌려준 횟수를 세어, 실패가 이어지면 대기 시간을
  5초 → 10초 → 20초 → 최대 60초로 늘리고, 성공하면 즉시 초기화.
- 같은 오류 로그는 예: 10회에 1번만 찍어 로그 오염을 줄인다.
- 텔레그램 명령 반응이 최대 1분 늦어질 수 있으나 장애 중에만 그렇고, 평상시 영향 없음.

### 수정 5 (선택) — 브라우저 띄우기 전 사전 연결 점검 (ridi_collector.py `gather_recent` 서두)
- Chromium을 띄우기 전에 urllib로 `https://ridibooks.com`에 짧은 타임아웃(10초) HEAD/GET을
  2~3회(간격 20초) 시도, 전부 실패면 브라우저를 아예 띄우지 않고 실패 반환.
- 장점: 장애 중 Chromium 메모리 점유(2GB NAS에서 치명적)를 원천 차단.
- 단점: 리디 서버가 urllib 요청을 다르게 취급할 수 있음(403 등) → "타임아웃만 실패로 간주,
  403 등 응답이 오면 통과"로 구현하면 안전. **여유가 되면 추가 권장.**

### 운영(코드 외) 개선
- NAS 제어판 → 네트워크 → 일반에서 DNS 서버를 `1.1.1.1` / `8.8.8.8`로 지정(공유기 DNS 의존 제거).
- 07/04 저녁 시간대에 Hyper Backup·인덱싱 등 무거운 예약 작업이 있다면 수집 시각과 겹치지
  않게 조정.

### 테스트 계획 (로컬 Docker, 배포 전 필수)
1. **정상 동작 회귀**: 기존 방식대로 빌드·실행 → `/run` → 31개 수집·알림 정상 확인.
2. **전면 장애 모의**: `--add-host ridibooks.com:10.255.255.1` 추가 실행 → `/run` →
   (수정 1·2) 약 4작품 시도 후 조기 중단, (수정 3) "⚠️ 수집 실패" 알림 수신 확인.
   31분이 아니라 수 분 내에 끝나는지 확인.
3. **텔레그램 장애 모의**: `--add-host api.telegram.org:10.255.255.1` → (수정 4) 로그에
   백오프 간격이 늘어나는지, 오류 로그가 도배되지 않는지 확인 → 컨테이너 재실행(호스트
   차단 해제) 후 폴링 자동 회복 확인.
4. ⚠️ 로컬 테스트 전 NAS 봇과의 409 충돌 주의 — 시나리오 3처럼 텔레그램을 차단한 채
   테스트하면 실토큰 폴링이 안 나가므로 안전. 시나리오 1·2는 CLAUDE.md 절차대로
   NAS 쪽을 먼저 멈추고 진행.
5. 통과 후 `bombam_synology.zip` 재생성 → NAS 배포(다시 빌드) → `/run` 실동작 확인.
