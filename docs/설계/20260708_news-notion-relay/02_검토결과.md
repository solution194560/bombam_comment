# 설계 검토 결과

## [심각도: 높음] Notion 저장 계층·제목 규칙이 실제 산출물 경로와 어긋날 가능성

문제 설명: 설계는 `AI NEWs` 부모 페이지 바로 아래에서 `child_page.title`에 `통합 보고서_YYYY-MM-DD`가 포함된 페이지를 찾습니다. 그런데 이전 검증된 Notion 저장 규칙 기준으로는 `AI NEWs -> YYYY-MM-DD -> codex news_YYYY-MM-DD` 계층이 성공한 경로였습니다. 이 설계가 그 산출물을 중계하려는 것이라면, 현재 탐색은 날짜 페이지까지만 보거나 아예 제목이 달라서 보고서를 못 찾습니다.

개선 제안: 목표 산출물 제목과 계층을 먼저 확정하세요. 기존 `news_CODEX` 산출물 중계가 목적이면 `AI NEWs` 하위 `YYYY-MM-DD` 페이지를 찾은 뒤 그 자식에서 `codex news_YYYY-MM-DD` 또는 새로 정한 `통합 보고서_YYYY-MM-DD`를 찾는 2단계 탐색이 필요합니다. 반대로 새 게시 경로가 정말 `AI NEWs` 직하위 `통합 보고서_YYYY-MM-DD`라면, 생산자 쪽 저장 규칙도 같은 이름으로 고정해야 합니다.

## [심각도: 중간] Notion 오류 분기가 403·529·일시 장애를 빠뜨림

문제 설명: 설계는 401/404/429/타임아웃을 주요 실패로 다루지만, Notion 공식 상태 코드에는 403 `restricted_resource`, 502/503/504, 529 `service_overload`도 있습니다. 특히 block children 엔드포인트는 읽기 권한 부족 시 403을 반환할 수 있고, 접근 불가 또는 미공유 리소스는 404일 수 있습니다. 현재 설계대로 `RuntimeError("HTTP 401 — ...")` 문자열만 단순 판별하면 사용자에게 “내부 오류”로 뭉개질 가능성이 큽니다.

개선 제안: 새 예외 클래스를 금지할 이유가 크지 않습니다. 최소 구현을 유지하더라도 `_get_json`에서 `RuntimeError` 대신 `NotionApiError(status, body, retry_after)` 같은 작은 예외를 두는 편이 더 안전합니다. 403은 “통합 권한/읽기 권한 부족”, 404는 “페이지 미공유 또는 ID 오류”, 429와 529는 `Retry-After` 기반 1회 재시도로 묶어 처리하세요. 참고: [Notion status codes](https://developers.notion.com/reference/status-codes), [request limits](https://developers.notion.com/reference/request-limits).

## [심각도: 중간] `Notion-Version: 2022-06-28`은 동작 가능성이 높지만 신규 구현 기본값으로는 애매함

문제 설명: Notion 버전 헤더는 필수입니다. 공식 문서는 현재 최신 버전을 `2026-03-11`로 안내하며, versioning 문서에는 구버전 지원 중단 계획은 없다고 설명합니다. 즉 `2022-06-28`이 즉시 틀렸다고 보기는 어렵지만, 새 구현에서 “안정 버전이라 충분”이라고만 쓰면 검증 근거가 약합니다.

개선 제안: 외부 SDK를 쓰지 않고 HTTP만 호출하므로 최신 버전인 `2026-03-11`로 맞추는 쪽이 깔끔합니다. 굳이 `2022-06-28`을 유지한다면 로컬 실호출 테스트 항목에 “2022-06-28에서 child_page title, block rich_text, pagination 응답 확인”을 명시하세요. 참고: [Notion versioning](https://developers.notion.com/reference/versioning), [Retrieve block children](https://developers.notion.com/reference/get-block-children).

## [심각도: 중간] 핵심 요약 추출은 정상 구조에는 충분하지만 중첩·지원 블록에 약함

문제 설명: heading 레벨 비교 규칙은 맞습니다. 시작 heading과 같거나 더 높은 heading에서 종료하는 방식은 `heading_2` 아래 `heading_3` 세부항목을 포함하고 다음 `heading_2`에서 멈추므로 타당합니다. 그러나 Notion block children API는 한 번에 첫 단계 자식만 반환합니다. 설계처럼 toggle/callout/paragraph 내부 자식을 재귀 조회하지 않으면, 요약이 toggle이나 callout 안에 들어간 문서에서는 정상 heading을 찾지 못하거나 폴백도 빈 텍스트가 됩니다. 공식 문서도 완전한 표현을 위해 재귀 조회가 필요할 수 있다고 안내합니다.

개선 제안: 전면 재귀 대신 제한적 재귀가 적절합니다. `has_children=True`인 `toggle`, `callout`, `paragraph`, list item만 최대 깊이 1~2, 최대 블록 수 200 정도로 펼치면 NAS 부담은 거의 없고 실패율은 크게 줄어듭니다. 최상위만 유지하려면 생산자 문서 규칙에 “핵심 요약 heading과 본문은 최상위 블록”을 명시해야 합니다. 참고: [Working with page content](https://developers.notion.com/guides/data-apis/working-with-page-content).

## [심각도: 중간] 페이지네이션 커서 처리와 10페이지 상한의 실패 방식이 불명확함

문제 설명: `urllib.parse.quote(cursor)`는 기본적으로 `/`를 인코딩하지 않습니다. Notion은 커서를 불투명 값으로 취급하라고 하므로 직접 붙이기보다 `urlencode`로 쿼리 전체를 만드는 편이 안전합니다. 또한 10페이지 상한은 NAS 보호 측면에서는 좋지만, 1,000개 이후 `has_more=True`인 상태로 조용히 반환하면 “없음” 또는 요약 누락으로 오진할 수 있습니다.

개선 제안: `urllib.parse.urlencode({"page_size": 100, "start_cursor": cursor})` 방식으로 URL을 구성하세요. 10페이지 상한에 걸리면 로그와 실패 알림 중 하나는 남겨야 합니다. 부모 페이지 목록은 10페이지 상한 초과 시 보고서 탐색 정확성이 깨지므로 `RuntimeError("Notion pagination limit exceeded")`가 더 낫습니다.

## [심각도: 낮음] `find_today_report`의 “첫 번째 매치” 규칙은 중복 페이지에 취약함

문제 설명: 같은 날짜 제목이 여러 개 있으면 첫 번째를 사용한다고 되어 있습니다. Notion 페이지 내 배치 순서가 사용자가 기대한 최신본이라는 보장은 약합니다. 공식 versioning 문서도 정렬을 보장하지 않는 엔드포인트의 결과 순서는 개선될 수 있다고 안내합니다.

개선 제안: 중복이 있으면 `last_edited_time`이 가장 최신인 블록을 선택하거나, 중복 감지 로그를 남기고 제목이 정확히 일치하는 페이지를 우선하세요. 최소 구현을 유지하려면 “중복 제목은 운영자가 만들지 않는다”를 운영 규칙으로 문서화해야 합니다.

## [심각도: 낮음] `last_run_date` 선기록 정책은 설계 의도와 일치하지만 제품 요구를 좁게 해석함

문제 설명: `news_bot.py`의 현재 루프는 `daily_time` 이후이고 `last_run_date`가 오늘이 아니면 실행 시작 직후 오늘 날짜를 저장합니다. 설계는 이를 유지하므로 중복 방지와 비용 방지에는 맞습니다. 다만 오늘 10시 이후 늦게 게시되는 보고서를 그날 다시 확인하지 않는 것은 기능 요구가 아니라 운영 정책입니다.

개선 제안: “늦게 게시되면 그날은 링크가 오지 않는다”를 사용자 승인 사항으로 남기세요. 상습 지연이면 polling을 넣기보다 `daily_time`을 늦추는 방식이 이 프로젝트의 최소 구현 원칙과 NAS 제약에 맞습니다.

## [심각도: 낮음] 삭제 범위와 기존 코드 충돌 위험은 대체로 잘 통제됨

문제 설명: `news_llm.py`의 런타임 사용처는 현재 `news_bot.py`뿐이고, `send_long`은 실제로 `notify.py`가 아니라 `news_bot.py`에 있습니다. `entrypoint.sh`는 `news_bot.py`를 별도 재기동 루프로 띄우고, `bot.py`의 댓글봇 설정 파일은 `settings.json`, 뉴스봇은 `news_settings.json`이라 직접 충돌은 없습니다.

개선 제안: 삭제 시 `.claude/settings.local.json` 같은 개발 허용 목록에 `news_llm.py`가 남아도 런타임 문제는 아니지만, 테스트 명령이 혼동될 수 있으니 후속 정리 대상으로만 표시하세요. `DEFAULT_KEYS = {"notion": ""}` 축소는 안전합니다. 기존 `api_keys.json`에 LLM 키가 남아도 로딩 루프가 `DEFAULT_KEYS` 키만 읽으므로 무시됩니다.

## [심각도: 낮음] 더 단순한 대안

문제 설명: `news_notion.py` 분리는 테스트 가능성과 비용 없는 몽키패치 측면에서 타당합니다. 다만 기능이 “오늘 페이지 찾기, 요약 추출, URL 만들기”뿐이라 초기 구현은 파일 하나로도 가능합니다.

개선 제안: 유지보수 기준으로는 설계안처럼 `news_notion.py`를 두는 편이 낫습니다. 더 단순하게 하려면 설정 스키마 확장 없이 부모 ID 상수만 두되, 탐색 경로만 실제 저장 계층에 맞추세요. 외부 패키지 0개, Xvfb/Chromium 미사용, Docker 설정 무변경이므로 2GB NAS 제약은 위반하지 않습니다.

## 미해결 질문 답변

1. 핵심 요약 heading 문구는 생산자 계약으로 고정해야 합니다. 고정할 수 없다면 현재 폴백은 “빈 문자열 방지”에는 충분하지만 “핵심 요약 전송” 목표에는 부족합니다.

2. 수신자 1인은 이 뉴스 중계 목적에는 적절합니다. 댓글봇의 `subscribers` 규칙과 섞으면 범위가 커지므로 이번 변경에서는 소유자 1인 유지가 맞습니다.

3. 당일 늦게 게시해도 재확인하지 않는 정책은 현재 `last_run_date` 무변경 원칙과 일치합니다. 운영상 늦은 게시가 잦으면 재시도 루프보다 `daily_time`을 늦추는 것이 더 단순합니다.

4. 기존 LLM 키 항목은 코드상 무시되므로 방치해도 동작에는 문제 없습니다. 다만 더 이상 쓰지 않는 실키는 보안상 운영자가 파일에서 제거하는 편이 좋습니다.

5. `2022-06-28`은 구버전 지원 관점에서 동작 가능성이 높지만, 새 코드 기본값으로는 최신 `2026-03-11`이 더 명확합니다. `2022-06-28`을 유지하려면 실호출 검증 결과를 테스트 기록에 남기세요.

6. 최상위 블록만 읽는 판단은 현재 리포트 구조가 정말 최상위 heading+paragraph라면 타당합니다. 구조가 조금이라도 바뀔 수 있으면 제한적 재귀 조회를 넣는 편이 안전합니다.

7. “없음”은 정보 알림, 오류는 실패 알림으로 구분하는 설계가 맞습니다. `_notify_failure`로 합치면 코드는 조금 줄지만 사용자가 게시 지연과 시스템 장애를 혼동할 가능성이 커집니다.
