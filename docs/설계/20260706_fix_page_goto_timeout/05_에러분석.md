# 에러 분석: Page.goto 30초 타임아웃 연속 4회 → 수집 조기 중단 (07/06 09:36)

## 1. 증상

발생 환경: **시놀로지 NAS(DS224+, Celeron J4125, RAM 2GB)의 bombam 컨테이너** (실서비스).

- 시점: 2026-07-06 09:36:11 수집 실행 중
- 결과: 작품 31개 중 6개 시도 → **2개 성공, 4개(#3~#6) 연속 실패 → 조기 중단** ("수집 2/6 성공(조기 중단)")
- 같은 시간대에 봇 프로세스(bot.py)의 텔레그램 통신도 실패:

```
[ 3~6/31] 상세페이지 바로가기 | ❌ Page.goto: Timeout 30000ms exceeded.
  [시도 2/2] Page.goto 실패: Page.goto: Timeout 30000ms exceeded.

[getUpdates 실패 1회] 1초 대기 후 재시도...
  [api] getUpdates 오류: <urlopen error _ssl.c:993: The handshake operation timed out>
  [api] getUpdates 오류: The read operation timed out
```

중요한 배경: **이 로그는 전부 새 코드(커밋 f408206, 07/06 02:46 zip 재생성분)의 출력이다.**
- `Timeout 30000ms`(30초) — f408206에서 기존 60초를 30초 + 1회 재시도로 바꾼 값
- `[시도 2/2] Page.goto 실패` — f408206에서 추가한 작품별 재시도 로직 (`ridi_collector.py:322~336`, `_open_book_and_scroll`)
- `연속 4개 작품 실패 → 조기 중단` — f408206에서 추가한 조기 중단 로직 (`ridi_collector.py:812~816`, `gather_recent`)
- `[getUpdates 실패 1회] 1초 대기` — f408206에서 추가한 지수 백오프 로그 (`bot.py:237~243`)

즉 **새 방어 로직이 배포된 상태에서, 방어 로직이 설계 그대로 작동한 이벤트**다.
(예전 코드였다면 31개 × 60초 ≈ 31분을 헛돌았을 상황이, 약 5~6분 만에 중단 + "⚠️ 수집 조기
중단" 알림 + 수집분 2개 전송으로 끝났다.)

## 2. 근본 원인

### 결론: 07/04~05와 동일한 **NAS 외부 네트워크 회선 품질 저하의 재발**. 코드 버그 아님.

근거 (이전 분석 `docs/설계/20260706_fix_ridibooks_timeout/05_에러분석.md`과 동일 논리):

1. **서로 무관한 두 목적지가 같은 시간대에 함께 실패했다.**
   - `ridibooks.com` — Playwright/Chromium의 `page.goto` (`ridi_collector.py:327`)
   - `api.telegram.org` — bot.py의 urllib `getUpdates` (`bot.py:40~51`)
   - 통신 라이브러리도, 상대 서버도 다른데 동시에 실패 → 공통 구간은 "NAS에서 인터넷으로
     나가는 회선"뿐이다. SSL 핸드셰이크 타임아웃(암호화 통신 시작 전 첫 인사 단계에서 멈춤)은
     회선 혼잡·패킷 손실의 전형적 증상이다.
2. **장애가 '부분적'이었다는 점이 이번 사건의 특징이다.** 사전 연결 점검
   (`_check_network_connectivity`, ridi_collector.py:711)을 통과했고, 로그인과 작품 #1~#2
   수집도 성공했다. 그 직후 #3부터 막혔다 → 실행 도중에 회선 상태가 나빠진 것.
   07/05에도 "08:12 전면 실패 → 13:31 완벽 성공" 패턴이 있었으므로, 이런 간헐적 등락은
   이 회선의 확인된 행동 양식이다.
3. **Cloudflare 차단이 아니다.** Cloudflare에 막히면 페이지 자체는 로드되고 "Just a moment"
   화면이 뜬다(→ `_wait_cloudflare`가 대기). `Page.goto` 타임아웃은 페이지 응답 자체가
   30초 안에 안 온 것으로, 차단이 아니라 연결 문제다. HEADLESS=0/Xvfb는 정상이었다는 뜻
   (로그인·#1~#2 성공이 증거).
4. **메모리(OOM) 단독 원인 가능성 낮음.** Chromium 구동 중 2GB RAM 압박이 통신을 느리게
   만들 수는 있으나(보조 요인), 07/04 저녁 수집이 돌지 않던 시간에도 SSL 타임아웃이 났던
   전례가 있어 회선이 1차 원인이다. 다만 이번 건은 수집 중에만 관찰됐으므로 리소스 포화
   중첩 여부는 6장에서 확인이 필요하다.

CLAUDE.md 함정 1~10번 대조: 해당 없음. 단, **이전 분석에서 제안한 "함정 11번(동시 타임아웃 =
회선 장애)"이 아직 CLAUDE.md에 추가되지 않은 상태에서 같은 유형이 재발**했다. 5장 참조.

## 3. 재현 조건

- **원 조건**: NAS 외부 회선이 실행 도중 일시적으로 나빠지면 발생. 회선 문제라 임의 재현 불가.
- **로컬 모의 재현** (수정안 검증용): 수집을 시작시킨 뒤 도중에 리디북스 주소를 막으면
  "일부 성공 → 연속 실패 → 조기 중단"을 흉내낼 수 있다.
  ```bash
  # 컨테이너 실행 중 (root 권한으로) hosts를 바꿔 도중 차단을 모의
  docker exec -u root bombam_test sh -c 'echo "10.255.255.1 ridibooks.com" >> /etc/hosts'
  ```
  (10.255.255.1 = 어디에도 연결되지 않는 주소 → goto가 하염없이 기다리다 타임아웃)
  차단 해제는 hosts에서 해당 줄 제거 또는 컨테이너 재시작.
- 전면 장애 모의는 이전과 동일: `docker run ... --add-host ridibooks.com:10.255.255.1`.
- ⚠️ 로컬 테스트 전 NAS 봇 정지 필수 (같은 토큰 → 409 Conflict, CLAUDE.md 참고).

## 4. 수정 방향

전제: 에러 자체(회선 등락)를 코드로 없앨 수는 없다. 목표는 **"일시 장애 후 자동 복구"**다.
우선순위 순서로 제시한다.

### 1순위 — 조기 중단 시 지연 자동 재시도 1회 (영향도: 큼 / 리스크: 낮음)
- **구현 위치**: `4_매일알림.py` (main 블록) — `gather_recent()` 결과 `stats["aborted"]`이면
  20~30분 `time.sleep` 후 `gather_recent()`를 **딱 1회** 다시 호출. 2차 결과가 더 좋으면
  (succeeded 더 많으면) 그것으로 알림, 2차도 실패면 기존 실패 알림 유지.
- 근거: 07/05 실측 패턴(오전 실패 → 오후 성공)상 회복 대기 후 재시도의 성공 확률이 높다.
- 주의점: `bot.py:172`의 `subprocess.run(..., timeout=3600)` 안에서 도는 것이므로
  "대기 30분 + 수집 2회"가 1시간을 넘지 않게 대기시간을 20분 정도로 잡거나 timeout을
  7200으로 늘릴 것. 재시도 중에도 `_job_running`이 set 상태라 /run 중복 실행은 자동 방지됨.
  재시도 시작 전 "⏳ 20분 후 자동 재시도합니다" 안내를 실패 알림 문구에 덧붙이면 사용자
  혼란이 없다.
- 대안(비권장): bot.py에서 실패 감지 후 재스케줄 — 4_매일알림.py의 결과(stats)를 프로세스
  경계 너머로 전달해야 해서(파일 or exit code) 구조가 복잡해진다.

### 2순위 — 조기 중단 전에 '쿨다운 + 연결 재점검' 한 번 (영향도: 중간 / 리스크: 낮음)
- **구현 위치**: `ridi_collector.py` `gather_recent()`의 연속 실패 분기(812~816행 부근).
- 연속 4회 실패 도달 시 즉시 중단하지 말고: ① 90~120초 대기 → ② 기존
  `_check_network_connectivity()` 재호출 → ③ 통과하면 실패 카운터를 리셋하고 다음 작품부터
  계속(이때 실패했던 4개는 `failed_works` 목록에 넣어 루프 끝에서 한 번 더 시도), ④ 그래도
  불통이면 그때 진짜 중단.
- 효과: 2~5분짜리 짧은 등락에는 수집을 살려내고, 진짜 장애면 기존처럼 빠르게 중단.
  1순위와 병행하면 "짧은 등락 → 2순위가 흡수, 긴 장애 → 1순위가 복구"로 역할이 나뉜다.

### 3순위 — 재시도(2차 시도) 타임아웃을 60초로 차등화 (영향도: 경미 / 리스크: 거의 없음)
- **구현 위치**: `ridi_collector.py` `_open_book_and_scroll()` (322~336행).
- 현재 두 시도 모두 30초. 1차 30초는 유지하되 **2차 시도만 60초**로: 느리지만 살아있는
  회선에서 35~50초 걸리는 페이지를 건질 수 있다. 최악 소요는 작품당 70초 → 100초로 늘지만
  조기 중단 판정이 4.7분 → 6.7분이 되는 정도라 감내 가능. 코드 두 줄 수준의 저비용 수정.

### 4순위 — 운영(코드 외): 회선/DNS 점검 (영향도: 근본적일 수 있음 / 리스크: 없음)
- 이전 분석(20260706_fix_ridibooks_timeout) 6장의 확인 요청이 **아직 미회신** 상태다.
  같은 회선 문제가 3일 연속(07/04, 05, 06) 재발했으므로 이제는 환경 쪽 조치가 필요하다:
  - DSM → 제어판 → 네트워크 → 일반: DNS 서버를 `1.1.1.1` / `8.8.8.8`로 수동 지정
  - 공유기 재부팅, 랜선/포트 교체 시도
  - 며칠 더 반복되면 통신사 회선 품질 점검 요청
- 코드가 아무리 방어해도 회선이 하루 수 회 끊기면 알림 품질은 계속 저하된다.

### 5순위(참고) — 메모리 절약: 수집 중 이미지 로딩 차단 (영향도: 중간 / 리스크: 중간)
- `page.route()`로 이미지/폰트 요청을 차단하면 페이지 로드가 가벼워져 혼잡 회선에서
  타임아웃 확률과 2GB RAM 부담이 함께 준다. 다만 Cloudflare가 리소스 차단을 봇 신호로
  볼 가능성(함정 1번과 같은 계열의 위험)이 있어 **로컬에서 Cloudflare 통과를 충분히 검증한
  뒤에만** 도입할 것. 1~3순위 적용 후에도 문제가 잦으면 검토.

### 테스트 계획 (로컬 Docker, 배포 전 필수 — CLAUDE.md 워크플로우)
1. 정상 회귀: 기존 절차로 31개 완주 + 알림 확인.
2. 도중 차단 모의(3장 방법): #N부터 실패 → (2순위) 쿨다운·재점검 후 차단 해제 시 수집 계속
   재개되는지 / 계속 차단 시 (1순위) 지연 재시도 1회 후 실패 알림이 오는지 확인.
3. 텔레그램 백오프 회귀(기존 시나리오 3) 재확인.
4. 통과 후 `bombam_synology.zip` 재생성 → NAS 배포(다시 빌드) → `/run` 검증.

## 6. 미해결 질문

1. **09:36 실행이 자동이었나 수동(/run)이었나?** 로컬 settings.json의 daily_time은 11:00이라
   09:36은 수동 /run(새벽 배포 후 검증 절차 4단계?)으로 추정되지만, NAS의
   `docker/bombam/work/settings.json`은 봇이 바꿀 수 있어 확정 불가.
   → File Station에서 `docker/bombam/work/settings.json`을 열어 `daily_time`과
   `last_run_date` 값을 알려주시면 확정된다. (수동이었다면 11:00 자동 수집은 그날 정상
   시도됐을 것 — 그 결과도 궁금하다.)
2. **09:36 조기 중단 이후 같은 날 재실행(11:00 자동 또는 수동 /run) 결과** — 성공했다면
   "일시 등락 후 회복" 패턴이 한 번 더 확인되어 1순위 수정안의 근거가 강해진다.
   → Container Manager → bombam → 로그에서 07/06 09:36 이후 구간 화면 캡처 부탁.
3. **09:36 수집 중 NAS 자원 상태** — DSM → 리소스 모니터 → 성능(과거 기록)에서 07/06

4. **이전 분석(20260706_fix_ridibooks_timeout) 6장의 5개 확인 항목** — 특히 NAS DNS 설정
   값과 저녁 시간대 예약 작업 여부가 아직 미회신. 회선 장애가 3일 연속이므로 이제 환경
