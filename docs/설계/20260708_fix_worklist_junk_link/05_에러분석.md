# 에러 분석: 작품목록 캐시가 "상세페이지 바로가기" 제목으로 오염 + 수집 조기 중단 반복

## 1. 증상

### 텔레그램 알림
```
▶️ 수집 시작! 끝나면 결과를 보낼게요. (몇 분~)
⚠️ 수집 조기 중단
작품 5개 중 1개만 수집됨 (네트워크 문제로 추정)
현재까지 수집한 댓글을 이하 전송합니다.
```

### NAS 로그 (`nas_log_원문.txt`, 2026-07-08, 컨테이너=NAS 실서비스)
하루 동안 수집이 5회 실행됐고 **전부 조기 중단**됐다.

| 시각 | 결과 | 상세 |
|---|---|---|
| 09:32 (정기 자동) | 0/4 성공 | 1~4번 전부 `상세페이지 바로가기` 제목으로 goto 타임아웃 |
| 10:02 (자동 재시도) | 0/4 성공 | 동일 패턴 |
| 20:57 | 0/4 성공 | 동일 패턴 |
| 21:27 (자동 재시도) | 2/6 성공 | **1번 "파고드는 사랑", 2번 "웨딩데이" 성공** → 3~6번 타임아웃 |
| 21:33 | 1/5 성공 | 1번 성공 → 2~5번 타임아웃 (텔레그램 증상의 그 실행) |

공통 로그 패턴 (로그 665~699행 등).
```
[최근댓글] 기존 작품목록 31개 사용

  [시도 2/2] Page.goto 실패: Page.goto: Timeout 30000ms exceeded.

  연속 4개 작품 실패 → 조기 중단

```

- **실패한 작품은 예외 없이 제목이 "상세페이지 바로가기"**로 찍힌다. 성공한 작품만 정상 제목이 찍힌다.
- 25분 뒤 자동 재시도(`retry_at`)가 돌아도 같은 캐시 목록을 다시 쓰므로 같은 지점 부근에서 반복 중단된다.
- 로그 전반의 `getUpdates 오류: HTTP Error 409: Conflict` 다수는 **별개 이슈**(같은 봇 토큰을 다른 곳에서도 폴링 — CLAUDE.md 함정의 로컬/NAS 이중 폴링 케이스)로, 이번 분석 대상이 아니다. 다만 같은 로그에 텔레그램 API SSL handshake 타임아웃도 5회 이상 보여(02:37, 07:04, 09:12, 09:43, 20:53) **NAS의 바깥 네트워크가 간헐적으로 불안정**하다는 관찰 근거는 된다.

## 2. 근본 원인

두 층위의 문제가 겹쳐 있다. (1)은 코드로 확정, (2)는 로그로 일부 확정 + 일부 추정.

### 2-1. [확정] 작품목록 수집기가 제목 유효성 검사 없이 "첫 번째 앵커"를 채택한다 → 캐시 제목 오염

`ridi_collector.py`의 작품목록 추출 JS `WORKS_JS`(133~154행)는 작가 페이지의
`a[href*="/books/"]` 앵커(링크)를 무차별로 순회하며, **같은 book_id의 첫 앵커가 이긴다.**

```js
document.querySelectorAll('a[href*="/books/"]').forEach(a => {
  ...
  if (seen.has(id)) return;                       // ← id 중복만 차단
  let title = (a.innerText || a.textContent || '').trim();
  if (!title || title.length < 2) return;          // ← 길이 검사뿐, 내용 검사 없음
  seen.add(id);                                    // ← 나쁜 제목도 id를 '선점'
  out.push({ title, url: 'https://ridibooks.com/books/' + id, book_id: id });
```

리디 작가 페이지에는 책 하나당 링크가 여러 개 있고(표지·제목·접근성용 "상세페이지 바로가기" 등),
DOM에서 **"상세페이지 바로가기" 텍스트 링크가 진짜 제목 링크보다 먼저** 나오면 그 텍스트가
title로 저장되고, 뒤에 오는 진짜 제목 링크는 `seen`(중복 방지 집합)에 막혀 버려진다.

프로젝트에는 이미 이런 가짜 제목을 걸러내는 함수 `_good_title()`(229~235행,
`bad=("상세페이지 바로가기","바로가기","구독불가","19세")`)이 있지만, 이 함수는
**수집 성공 후 "표시용 제목 보정"에만** 쓰인다(`gather_recent` 799행, `_collect_reviews` 585행,
`run_latest_only` 626행). 정작 목록을 만드는 `collect_work_list()`(412~441행)와
목록을 로드하는 `_load_or_collect_works()`(382~409행)에는 **한 번도 적용되지 않는다.**

`_load_or_collect_works()`의 방어는 "0개면 재수집"(399행 `if works: return works`)뿐이라,
**항목 수만 맞으면(31개) 제목이 전부 쓰레기여도 그대로 통과**한다. CLAUDE.md 함정 6번
("빈 목록이 캐시로 굳음")과 같은 구조의 문제가 '개수'가 아니라 '품질' 축으로 재발한 것이다
(함정 6번의 변형 재발).

이렇게 오염된 목록이 언젠가 NAS의 `work/봄밤_작품목록.json`(OUTPUT_DIR=/app/out)에 저장돼
캐시로 굳었고, 이후 모든 실행이 그 파일을 로드한다("기존 작품목록 31개 사용").
이미지에 동봉된 seed(`봄밤_작품목록.json`)는 31개 전부 정상 제목임을 확인했다(오염 0개).
로컬 테스트가 31/31 성공한 이유도 로컬 `work/`에 캐시 파일이 없어 이 깨끗한 seed를 썼기 때문이다.

### 2-2. [부분 확정] goto 타임아웃 자체는 "오염 URL" 단정 불가 — 간헐적 접속 장애 성분이 확실히 섞여 있다

당초 가설은 "오염 항목의 URL이 유효하지 않아 타임아웃"이었으나, 로그가 이를 그대로 지지하지는 않는다.

- `WORKS_JS`는 URL을 앵커의 href에서 뽑은 book_id로 `https://ridibooks.com/books/<id>` 형태로
  **재조립**하므로, 제목이 쓰레기여도 URL 형식 자체는 정상이다.
- 결정적으로 **같은 캐시의 같은 위치 항목이 성공과 실패를 오간다.** 21:27 실행에서 1·2번 항목은
  성공했고 성공 시 제목 보정 로직(799~800행)이 "파고드는 사랑"/"웨딩데이"로 바꿔 출력했다.
  즉 **캐시의 1·2번 제목도 "상세페이지 바로가기"였는데 URL은 유효한 실제 작품 페이지**였다는 뜻이다.
  같은 2번 항목이 6분 뒤 21:33 실행에서는 30초 타임아웃 2회로 실패했다.
- 따라서 타임아웃의 상당 부분은 NAS(저사양 Celeron + 불안정 네트워크 + Cloudflare)에서
  `page.goto(..., timeout=30000)`(`_open_book_and_scroll`, 323~337행)이 30초를 못 지키는
  **간헐적 접속 장애**다. 한 번 타임아웃이 시작되면 그 실행 안에서는 연속으로 실패해
  4연속 실패 조기 중단(813~817행)에 걸리는 패턴이 5회 실행 모두에서 반복됐다.
- 다만 **3번 이후 항목은 관찰된 어떤 실행에서도 성공한 적이 없어**, 목록 뒤쪽 일부 항목의
  book_id가 실제 접근 불가한 페이지(작가 페이지의 다른 성격 링크에서 뽑힌 id)일 가능성은
  배제하지 못한다. 이는 NAS 캐시 파일을 직접 확인해야 확정된다(6장 참조).

### 2-3. 오염이 실제로 만든 피해 (확정)

URL 유효성과 무관하게, 제목 오염은 다음 실피해를 확정적으로 만들었다.

1. **진단 불능** — 실패 로그가 전부 "상세페이지 바로가기"라 어느 작품이 실패했는지 식별 불가.
   이번 원인 규명이 늦어진 직접 원인.
2. **오염의 영속화** — 캐시가 "31개"라는 이유만으로 매 실행 통과되고, 자동 재시도(`retry_at`,
   bot.py 207~219행)까지 같은 목록으로 돌아 실패 알림이 반복 발송됨.
3. (미확정이나 가능성) 뒤쪽 항목 일부가 무효 URL이라면, 무효 항목이 연속 배치된 구간에서
   4연속 실패 조기 중단을 **구조적으로 유발**해 뒤의 정상 작품 수십 개가 매번 수집 기회를 잃는다.

## 3. 영향 범위

오염된 `WORKS_JSON` 캐시를 로드하는 모든 경로가 영향을 받는다.

| 경로 | 코드 | 영향 |
|---|---|---|
| 최근 N일 댓글(매일 자동 알림·`/run`) | `gather_recent()` 789행 → `_load_or_collect_works()` | **주 피해 경로.** 조기 중단·재시도 반복 |
| 작품별 최신댓글 | `run_latest_only()` 620행 → `_load_or_collect_works()` | 동일 오염 목록 사용 |
| 댓글만 수집 | `run_comments_only()` 568행 → `WORKS_JSON` 직접 로드 | 동일 |
| 전체 수집 | `run_full()` 551행 → `collect_work_list()` | 캐시를 안 쓰지만 **재수집 결과를 필터 없이 556행에서 저장** → 오염을 새로 만드는 발생원 |

## 4. 수정 방향 (implementer 작업 지시 수준)

근본(코드) 2건 + 즉효(NAS 캐시 정화) 1건의 두 층으로 제안한다.

### 4-A. [근본 1] `WORKS_JS`: 나쁜 제목 앵커는 book_id를 '소비'하지 않고 건너뛰기
- 파일·위치: `ridi_collector.py` 133~154행 `WORKS_JS`.
- 변경 요지: 앵커 순회 시 제목이 나쁜 패턴(`상세페이지 바로가기`, `바로가기`, `구독불가`,
  `19세` — Python `_good_title()`의 `bad` 튜플과 동일 기준)에 걸리면 **`seen.add(id)`를 하지 말고
  `return`**(스킵)한다. 그러면 같은 책의 진짜 제목 앵커가 뒤에서 정상적으로 잡힌다.
- 주의: 나쁜 제목 필터 문자열이 JS와 Python(`_good_title`) 두 곳에 존재하게 되므로, 주석으로
  상호 참조를 남겨 한쪽만 고치는 실수를 방지할 것(최소 구현 원칙상 공유 상수화까지는 불요).
- 부작용 검토: 어떤 책이 "상세페이지 바로가기" 앵커만 갖고 제목 앵커가 없으면 그 책이 목록에서
  빠질 수 있다. 이 경우는 4-B의 로드 시 필터와 동일한 결과(오염 항목 제외)이므로 허용 가능.

### 4-B. [근본 2] `_load_or_collect_works()`: 로드한 캐시의 오염 항목 자동 정화 (굳은 NAS 캐시의 자가 치유)
- 파일·위치: `ridi_collector.py` 382~409행.
- 변경 요지 (권장안).
  1. `WORKS_JSON`/seed에서 로드한 `works`에 `_good_title(w["title"])` 필터를 적용해
     `clean`(정상 제목 항목)을 만든다.
  2. `clean`이 충분하면(예: 원본의 절반 이상 && 1개 이상) `clean`을 사용하고,
     오염 항목이 있었다면 정화된 목록을 `WORKS_JSON`에 다시 저장한다 → **NAS 캐시가
     다음 실행 한 번으로 자동 정화**된다.
  3. `clean`이 크게 줄었으면(오염이 대부분) 캐시를 버리고 기존 흐름대로
     seed → 작가 페이지 재수집(4-A가 적용된 `collect_work_list`) 순으로 진행한다.
  - 지금 NAS 캐시는 제목만 오염이고 URL은 (최소 일부) 유효하므로, "오염 항목 제외" 대신
    "재수집으로 교체"가 되는 3번 경로로 떨어져도 안전하다.
- 대안 비교.
  - (대안 1) 필터 없이 "오염 발견 시 무조건 재수집": 단순하지만 작가 페이지 접속이 막힌 날엔
    목록을 통째로 잃는다. 권장안은 정상 항목을 보존한다.
  - (대안 2) 성공 시 보정된 `page_title`을 캐시에 역기록: 점진 치유는 되지만 조기 중단으로
    뒤 항목까지 못 가는 현 상황에선 치유 속도가 느리다. 이번 범위에서는 제외 권장(최소 구현).

### 4-C. [근본 3] `collect_work_list()` 저장 전 필터
- 파일·위치: `ridi_collector.py` 412~441행(및 저장 지점인 `run_full()` 556행,
  `_load_or_collect_works()` 406행).
- 변경 요지: 4-A가 1차 방어이므로, 수집 결과를 반환/저장하기 전에 `_good_title` 필터를 한 번 더
  통과시키는 2차 방어를 둔다(리디 DOM이 또 바뀌어 새로운 형태의 쓰레기 텍스트가 들어와도
  "저장 전 최종 관문"이 있게).

### 4-D. [즉효] NAS 캐시 수동 정화 (코드 배포 전이라도 당장 복구 가능, GUI만으로)
1. DSM **File Station** → `docker` → `bombam` → `work` 폴더로 이동.
2. `봄밤_작품목록.json` 파일을 우클릭 → **이름 바꾸기**로 `봄밤_작품목록.json.bak` 등으로 변경
   (삭제보다 이름 변경 권장 — 6장의 원인 확정용 증거로 보존).
3. 다음 실행부터 `_load_or_collect_works()`가 캐시 없음 → 이미지 동봉 seed(깨끗한 31개)를
   사용한다(393~398행). seed 사용 시에는 파일을 다시 저장하지 않으므로 재오염 없음.
4. 텔레그램에서 `/run`으로 즉시 검증.

### 4-E. [별개 후속 이슈로 분리 권장] 간헐적 goto 타임아웃 내성
2-2에서 본 대로 타임아웃에는 환경성 성분이 있다. 이번 수정 범위에는 넣지 말고(최소 구현),
캐시 정화 후에도 조기 중단이 재발하면 별도 건으로 검토할 것 — 예: 연속 실패 2회째에
`page`를 새로 만들거나 대기 후 재개, goto 타임아웃 상향(30→45초) 등. **지금 함께 고치면
이번 수정의 효과 검증이 흐려진다.**

## 5. 재현 조건·검증 계획

### 재현 (수정 전, 로컬 Docker)
1. 깨끗한 seed를 복사해 일부(연속 4개 이상) 항목의 `title`을 `"상세페이지 바로가기"`로,
   `url`의 book_id를 존재하지 않는 값(예: `9999999999`)으로 바꾼 오염 파일을
   `local_test/work/봄밤_작품목록.json`으로 배치.
2. 컨테이너 실행 → `/run` → NAS와 동일하게 "상세페이지 바로가기 | ❌ Page.goto: Timeout"
   연속 4회 → "조기 중단" 재현 확인.
   (⚠️ 로컬 테스트 전 NAS 봇과의 409 충돌 규칙 준수 — CLAUDE.md 로컬 테스트 절차.)

### 검증 (수정 후)
1. 같은 오염 파일 상태에서 실행 → `_load_or_collect_works()`가 오염을 감지·정화(로그로 확인)하고
   31개 전 작품 수집이 완주되는지.
2. 정화 후 `work/봄밤_작품목록.json`에 "상세페이지 바로가기" 항목이 0개인지.
3. `work` 목록 파일을 지운 상태에서 작가 페이지 재수집 경로(`collect_work_list`)를 태워
   새로 만든 목록에도 오염 항목이 0개인지(4-A·4-C 검증).
4. NAS 배포 후 `/run` 실행, 로그에서 정상 제목 31개와 "실제 창 모드" 확인.

## 6. 재발 방지 (CLAUDE.md 함정 목록 추가 제안)

함정 13번으로 추가 제안.

> 13. **작품목록 캐시는 '개수'만이 아니라 '제목 품질'도 검증해야 한다.** 작가 페이지에는 책 하나당
>     앵커가 여러 개("상세페이지 바로가기" 등 접근성/버튼 텍스트 포함) 있어, 첫 앵커를 무차별
>     채택하면 쓰레기 제목이 book_id를 선점해 캐시로 굳는다(함정 6번의 '품질판' 재발). 수집
>     JS에서 나쁜 제목 앵커는 book_id를 소비하지 않고 스킵하고, 저장·로드 시 `_good_title()`로
>     걸러 자동 정화한다. 실패 로그에 "상세페이지 바로가기"가 보이면 이 오염을 의심할 것.

## 7. 미해결 질문 (사용자 확인 요청)

1. **NAS 캐시 원본 확인 (가장 중요).** File Station에서
   `docker/bombam/work/봄밤_작품목록.json`을 열어(또는 다운로드해서 전달)
   ① "상세페이지 바로가기" 제목 항목이 몇 개인지, ② 각 항목의 `url`(book_id)이 seed의
   31개 id와 일치하는지 확인이 필요하다. **일치하면** 타임아웃은 순수 환경성(4-E 후속 검토 필요
   가능성 큼), **불일치/미지의 id가 섞여 있으면** 무효 URL이 조기 중단의 직접 원인이었다는
   가설이 확정된다. 4-D에서 삭제 대신 이름 변경을 권한 이유.
2. **오염 발생 시점.** NAS에서 이 캐시가 언제 만들어졌는지(파일 수정 날짜를 File Station에서
   확인 가능)를 알면, 리디 작가 페이지 DOM이 언제부터 "상세페이지 바로가기" 앵커를 먼저
   렌더링하게 바뀌었는지 추정할 수 있다. 수정의 긴급도 판단 참고용이며 필수는 아님.
3. **409 Conflict 발생원.** 로그 전반의 409는 같은 토큰을 폴링하는 다른 프로세스가 존재한다는
   뜻이다(별건). 로컬 맥에 `bombam_test` 컨테이너가 남아 있지 않은지 확인 권장 — 이번 로컬
   진단에서는 컨테이너 상태를 변경하지 않았다.
