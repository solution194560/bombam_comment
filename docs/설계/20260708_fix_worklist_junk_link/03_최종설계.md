# 03 최종설계 — 작품목록 "상세페이지 바로가기" 오염 수정

/ 이 문서가 implementer의 유일한 구현 기준이다. 근거는 `05_에러분석.md` 참조.

## 문제 요약
작가 페이지에는 책 1권당 `/books/{id}` 앵커가 여러 개(표지·제목·"상세페이지 바로가기" 버튼)
붙어 있다. `WORKS_JS`가 "book_id당 첫 앵커 승리"로 무차별 수집해, "상세페이지 바로가기" 같은
버튼 텍스트가 제목으로 저장되고 진짜 제목 앵커는 스킵된다. 이 오염 목록이 NAS
`work/봄밤_작품목록.json`에 캐시로 굳어, 매 수집·자동재시도마다 goto 타임아웃 → 조기 중단이
반복된다. (함정 6번 '개수 검증'의 '품질 검증' 누락판)

## 변경 파일 목록
- `ridi_collector.py` — 3곳 수정 (아래 상세). **다른 파일 변경 없음.**
- 기존 `_good_title(t)`(229~235행), `bad` 튜플(234행)을 재사용한다. 새 상수/함수/설정 옵션을
  만들지 않는다(최소 구현).

## 변경 1 — `WORKS_JS`: 나쁜 제목 앵커는 book_id를 소비하지 않고 스킵
**위치**: `ridi_collector.py` 133~154행 `WORKS_JS`

**요지**: 앵커 순회 중 title이 나쁜 패턴에 걸리면 `seen.add(id)` **전에** `return`(스킵)한다.
그러면 같은 book_id의 진짜 제목 앵커가 뒤에서 정상적으로 잡힌다. 현재 로직은
`if (!title || title.length < 2) return;` 다음에 곧바로 `seen.add(id)`를 하므로, 그 사이에
나쁜 제목 필터를 추가하면 된다.

**나쁜 패턴**: Python `_good_title()`의 `bad` 튜플과 **동일 기준** — `상세페이지 바로가기`,
`바로가기`, `구독불가`, `19세`. JS 배열로 하드코딩하되, 한쪽만 고치는 실수를 막기 위해
JS·Python 양쪽에 상호 참조 주석을 남긴다(공유 상수화는 하지 않음 — 최소 구현).

**구현 예시** (144행과 145행 사이에 삽입):
```js
    if (!title || title.length < 2) return;
    // ⚠️ 나쁜 제목(버튼/접근성 텍스트)은 book_id를 소비하지 않고 스킵 → 같은 책의
    //    진짜 제목 앵커가 뒤에서 잡히도록. Python _good_title()의 bad 튜플과 동일 기준(둘 다 수정할 것).
    const BAD = ['상세페이지 바로가기', '바로가기', '구독불가', '19세'];
    if (BAD.some(b => title.includes(b))) return;
    seen.add(id);
```
- `_good_title`(234행)에도 주석 한 줄 추가: `# ⚠️ WORKS_JS의 BAD 배열과 동일 기준(한쪽만 고치지 말 것).`

## 변경 2 — `_load_or_collect_works()`: 로드한 캐시 자동 정화 (굳은 NAS 캐시 자가 치유)
**위치**: `ridi_collector.py` 382~409행

**요지**: `WORKS_JSON`/seed에서 로드한 `works`에 `_good_title` 필터를 적용해 오염 항목을
걸러낸다. 정상 항목이 충분하면 그것을 쓰고, 오염이 있었으면 정화된 목록을 `WORKS_JSON`에 다시
저장한다(→ NAS 캐시가 다음 실행 한 번으로 자동 정화). 정상 항목이 대부분 사라졌으면 캐시를
버리고 기존 흐름(seed → 작가 페이지 재수집)으로 넘어간다.

**구현 지침** (현재 385~401행의 로드·판정 로직을 아래로 대체):
1. `works`를 로드한 뒤(현재 386~398행 로직 유지), 비어 있지 않으면:
   - `clean = [w for w in works if _good_title(w.get("title"))]`
   - `removed = len(works) - len(clean)`
   - **판정**: `len(clean) >= 1` 이면서 `len(clean) >= len(works) * 0.5` (정상이 절반 이상) →
     `clean` 채택.
     - `removed > 0` 이고 로드 출처가 `WORKS_JSON`(seed 아님)이면 정화본을 `WORKS_JSON`에 재저장.
       (seed에서 로드했다면 seed는 이미지 소유라 덮어쓰지 않는다 — 어느 파일에서 로드했는지
       플래그로 구분.)
     - 로그: `print(f"[{tag}] 작품목록 {len(clean)}개 사용(오염 {removed}개 정화)")` (removed>0일 때),
       아니면 기존 `기존 작품목록 N개 사용` 로그 유지.
     - `return clean`
   - 정상이 절반 미만이면(오염이 대부분) → `works=[]`로 두고 아래 재수집 경로로 진행하며
     `print(f"[{tag}] 캐시 오염 과다(정상 {len(clean)}/{len(works)}) → 재수집")`.
2. 재수집 경로(현재 402~409행)는 유지. `collect_work_list`가 변경 1·3으로 이미 필터되므로
   새 목록엔 오염이 없다.

**주의**: seed에서 로드한 경우 정화본을 seed 파일에 쓰지 말 것(WORKS_JSON에만 저장 가능).
현재 코드는 seed와 WORKS_JSON을 구분해 로드하므로(391~398행), 어느 쪽에서 채웠는지
불리언 변수(예: `from_cache = os.path.exists(WORKS_JSON) and works loaded from it`)로 구분한다.

## 변경 3 — `collect_work_list()` 저장 전 최종 필터 (2차 방어)
**위치**: `ridi_collector.py` 412~441행(`collect_work_list` 반환부) — 저장 지점은
`_load_or_collect_works()` 406행과 `run_full()` 556행.

**요지**: 변경 1이 1차 방어지만, 리디 DOM이 또 바뀌어 새로운 형태의 쓰레기 텍스트가 들어와도
막히도록 `collect_work_list`가 `all_works`를 반환하기 전에 `_good_title` 필터를 한 번 더
통과시킨다(441행 `return all_works` 직전).
```python
    all_works = [w for w in all_works if _good_title(w.get("title"))]
    return all_works
```
- 이렇게 하면 406행·556행의 저장은 이미 필터된 목록을 저장하므로 별도 수정 불필요.

## 구현하지 않을 것 (범위 밖 — 최소 구현)
- 간헐적 goto 타임아웃 내성(재개 전략·타임아웃 상향 등)은 **이번 범위 제외**. 캐시 정화 후에도
  조기 중단이 재발하면 별건으로 검토(분석 4-E).
- 필터 문자열의 공유 상수화/설정 옵션화 안 함(주석 상호참조로 충분).
- 성공 시 보정된 `page_title`을 캐시에 역기록하는 점진 치유 안 함(분석 4-B 대안2, 제외).

### T1. 재현 (수정 전 동작 확인 — 참고용, 이미 로그로 확인됨)

  `"상세페이지 바로가기"`로, `url`의 book_id를 `9999999999`(무효)로 바꾼 오염 파일을
  `local_test/work/봄밤_작품목록.json`으로 배치.

### T2. 자동 정화 검증 (변경 2)
- T1 오염 파일을 둔 상태로 컨테이너 실행 → `/run`.

### T3. 오염 과다 시 재수집 경로 (변경 2 분기)

### T4. 수집 JS 필터 (변경 1·3)
- (가능 범위에서) `collect_work_list` 결과에 `_good_title` 필터가 걸리는지 코드/유닛 관점 확인.
  실제 페이지 수집이 어려우면 `_good_title`이 "상세페이지 바로가기"를 False로 판정하는지, 반환부
  필터가 적용됐는지 정적으로 검증.

### T5. 회귀
- 깨끗한 캐시(오염 0)를 두고 실행 → 기존과 동일하게 `기존 작품목록 31개 사용`, 정화 로그 없이
