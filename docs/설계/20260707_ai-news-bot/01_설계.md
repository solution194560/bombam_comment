# 설계: AI 뉴스 리포트 봇 (4개 LLM 교차검증 → 텔레그램 일일 리포트)

## 1. 목표와 배경

### 1.1 무엇을 만드는가
Grok(xAI)·GPT(OpenAI)·Claude(Anthropic)·GLM-5.2(Z.AI) 4개 LLM API를 각자의 웹검색
기능과 함께 병렬 호출해 "AI / IT / 클라우드 인프라 / 사이버보안" 뉴스를 조사시키고,
그 4개 결과를 Claude API 1회 호출로 비교·교차검증·종합해 하나의 한국어 리포트로 만든 뒤,
텔레그램으로 소유자(chat_id 276954538)에게 전송하는 **별도 파이썬 상주 프로그램**을
기존 봄밤 도커 컨테이너에 추가한다.

- 기존 `bot.py`(봄밤 댓글 알림봇)와는 **완전히 별도의 프로세스**로 상주한다(사용자 명시 요구).
- 매일 정해진 시각(기본 **08:00** 제안)에 자동 실행. 시각은 설정 파일로 변경 가능.
- 사용자는 4개 API 키 중 **일부만 보유** — 키가 없는 제공자는 건너뛰고 리포트에
  "미참여"로 표기한다. 전부 없거나 전부 실패하면 실패 알림을 보낸다.

### 1.2 성공 기준 (무엇이 되면 완료인가)
1. 컨테이너 시작 시 `bot.py`(댓글봇)와 `news_bot.py`(뉴스봇)가 함께 뜨고, 서로 간섭 없이
   각자 동작한다(댓글봇의 텔레그램 명령 응답이 평소와 동일, 409 Conflict 없음).
2. 매일 설정 시각에 뉴스 리포트가 소유자에게 텔레그램으로 도착한다(4096자 초과 시 자동 분할).
3. 키가 없거나 호출에 실패한 제공자는 리포트 서두에 "미참여"로 표기되고, 나머지 제공자만으로
   리포트가 완성된다.
4. 전 제공자 실패(또는 키 전무) 시 "⚠️ 뉴스 리포트 실패" 알림이 소유자에게 도착한다.
5. 뉴스봇에 어떤 오류가 나도 댓글봇과 컨테이너 자체는 죽지 않는다.

## 3. 상세 설계

### 3.1 아키텍처 (프로세스 구조)

```
컨테이너 (python:3.12-slim, TZ=Asia/Seoul)
│
├─ entrypoint.sh
│    ├─ Xvfb :99 구동 + DISPLAY=:99          (기존 그대로)
│    ├─ python -u /app/news_bot.py &         ★ 추가 (백그라운드)
│    └─ exec python -u bot.py                (기존 그대로, PID 1)
│
├─ [프로세스 A] bot.py — 댓글봇 (기존)
│    └─ 텔레그램 getUpdates 롱폴링 + 매일 11:00 댓글 수집
│
└─ [프로세스 B] news_bot.py — 뉴스봇 (신규)
     └─ 시계만 보는 단순 루프 (텔레그램 폴링 안 함!)
          매일 daily_time 도달 → run_report()
            1) api_keys.json 로드 → 키 있는 제공자 선별
            2) 조사 프롬프트 생성(오늘 날짜 주입)
            3) 스레드 4개 병렬 호출 (Grok/GPT/GLM/Claude, 각자 웹검색)
            4) Claude 종합 호출 1회 (교차검증·병합)
            5) 리포트 서두에 참여/미참여 헤더 부착
            6) 텔레그램 분할 전송 → 소유자 1명
```

**핵심 격리 원칙 — 뉴스봇은 텔레그램 `getUpdates`를 절대 호출하지 않는다.**
같은 봇 토큰으로 두 프로세스가 동시에 폴링하면 409 Conflict가 나는데(CLAUDE.md 로컬 테스트
주의사항과 동일 원리), `sendMessage`(발신 전용)는 여러 프로세스가 써도 충돌이 없다.
따라서 뉴스봇에는 텔레그램 명령 인터페이스를 두지 않고, 설정 변경은 File Station으로
`work/news_settings.json`을 직접 수정하는 방식으로 한다(§3.8).

두 프로세스는 파일도 공유하지 않는다 — 댓글봇은 `settings.json`, 뉴스봇은
`news_settings.json`만 읽고 쓰므로 동시 쓰기 충돌이 원천적으로 없다.

### 3.2 entrypoint.sh 변경 (유일한 기존 파일 수정)

변경 전(현재) 마지막 부분.

```bash
# Xvfb 가 뜰 때까지 잠깐 대기
sleep 2

exec "$@"
```

변경 후.

```bash
# Xvfb 가 뜰 때까지 잠깐 대기
sleep 2

# AI 뉴스 리포트 봇 — 별도 상주 프로세스로 백그라운드 기동 (로그는 컨테이너 로그에 함께 출력)
python -u /app/news_bot.py &

exec "$@"
```

- `exec "$@"` 구조를 깨지 않는다(함정 2번). PID 1은 여전히 `bot.py`이고,
  백그라운드 자식의 stdout/stderr는 컨테이너 로그에 그대로 섞여 나온다
  (뉴스봇 로그는 전부 `[news]` 접두사를 붙여 구분 — §3.9).
- `set -e` 환경이지만 백그라운드 기동(`&`)은 즉시 성공으로 처리되므로 컨테이너 시작을
  막을 수 없다.
- 알려진 한계 1 — 뉴스봇 프로세스가 죽어도 컨테이너는 재시작되지 않는다(PID 1이 아니므로).
  대응: `news_bot.py`의 메인 루프를 최상위 try/except로 감싸 **어떤 예외에도 프로세스가
  종료되지 않게** 설계한다(§3.9). import 시점 문법 오류 등은 로컬 Docker 테스트에서 걸러진다.
- 알려진 한계 2 — 만약 뉴스봇이 죽으면 PID 1(bot.py)이 자식 프로세스를 수거(reap)하지
  않으므로 좀비 프로세스 항목이 1개 남는다. 실행 중인 코드가 아니라 프로세스 테이블의
  흔적일 뿐이라 2GB RAM에도 실질 영향은 없다(참고로만 기록).

### 3.3 API 키 보관 방식 — `api_keys.json` (이미지에 COPY) 로 결정

**결정: 이미지에 COPY되는 별도 `api_keys.json` 파일. docker-compose 환경변수 방식은 채택하지 않음.**

근거.
1. **기존 패턴과 동일** — `account.json`(리디 계정)·`telegram.json`(봇 토큰)이 이미
   "비밀값은 이미지에 COPY되는 json" 방식이다. 초보 사용자에게 새 관리 방식을 하나 더
   가르치지 않아도 된다.
2. **함정 5번 회피** — 시놀로지에서 단일 파일 볼륨 마운트는 컨테이너가 시작조차 안 되는
   원인이므로 금지. 이미지 COPY 방식이면 마운트가 필요 없다.
3. **docker-compose 환경변수 방식의 단점** — 키 4개를 compose 파일에 평문으로 넣으면
   Container Manager UI의 프로젝트 설정 화면에 그대로 노출되고, compose 파일은 배포
   zip에 항상 포함되므로 비밀 분리 효과도 없다. 또 어느 쪽이든 시놀로지에서는
   "다시 빌드"가 필요해 편의성 차이도 없다.
4. **키 변경 절차가 기존 배포 절차와 동일** — File Station에서 `api_keys.json` 덮어쓰기
   → 프로젝트 "다시 빌드". 새로 배울 것이 없다.

파일 스키마 (`api_keys.sample.json`으로 커밋, 실제 키는 `api_keys.json`에 — .gitignore 등록).

```json
{
  "xai": "",
  "openai": "",
  "zai": "",
  "anthropic": ""
}
```

- 빈 문자열("") 또는 키 누락 또는 파일 자체 부재 = "키 없음" = 해당 제공자 미참여.
- `.dockerignore`는 이 파일을 제외하지 않으므로(확인 완료) `COPY . .`로 이미지에 들어간다.
  파일이 없어도 빌드는 성공하며, 그 경우 전 제공자 미참여 → 실패 알림 경로로 동작한다.

### 3.4 새 파일: `news_llm.py` — 4개 제공자 호출 모듈

첫 줄에 한국어 역할 주석 필수(예: `# 4개 LLM(Grok/GPT/GLM/Claude) 뉴스 조사 API 호출 + Claude 종합 호출 — urllib만 사용`).

**의존성 결정: urllib.request만 사용 (SDK 미설치).** 근거.
- 4개 SDK(xai-sdk, openai, anthropic, zhipuai)를 설치하면 이미지 크기·메모리·의존성 충돌
  리스크가 늘어난다. 2GB RAM NAS + "최소 구현" 수칙에 반한다.
- 4개 호출 모두 "JSON POST 1번 → JSON 응답 파싱"이라 urllib로 충분하다.
  기존 `notify.py`도 같은 이유로 urllib만 쓴다.
- 스트리밍이 필요 없다(하루 1회 배치 실행이라 응답을 통째로 받으면 됨).

#### 공통 헬퍼

```python
def _post_json(url: str, headers: dict, body: dict, timeout: int = 600) -> dict
```
- `urllib.request.Request(url, data=json.dumps(body).encode(), headers=..., method="POST")`
  → `urlopen(req, timeout=timeout)` → `json.loads`.
- HTTP 오류(4xx/5xx) 시 `urllib.error.HTTPError`의 응답 본문을 읽어 예외 메시지에 포함
  (에러 원문 확인 수칙 — 상태코드만 남기지 않고 서버가 준 오류 JSON을 로그에 남김).
- 타임아웃은 넉넉히 600초(웹검색 도구 호출은 서버 측에서 수 분 걸릴 수 있음).

#### 제공자별 호출 함수 (아래 스펙은 2026-07 공식 문서 확인 완료분 — 추측 금지, 이대로 구현)

모든 함수는 `(prompt: str, api_key: str) -> str` 시그니처로 통일한다.
성공 시 조사 결과 텍스트를 반환하고, 실패(HTTP 오류·타임아웃·응답 구조 이상·빈 텍스트) 시
예외를 던진다. 호출자(`news_bot.py`)가 예외를 잡아 "미참여" 처리한다.

**1) `call_grok(prompt, key)` — xAI Grok**
- 구 Live Search(search_parameters)는 2026-01-12 폐기(410 Gone). 현행 Responses API 사용.
- POST `https://api.x.ai/v1/responses`
- 헤더: `Authorization: Bearer <key>`, `Content-Type: application/json`
- 요청 본문 (필드명이 `messages`가 아니라 **`input`** 임에 주의).
```json
{"model": "grok-4.3",
 "input": [{"role": "user", "content": "<조사 프롬프트>"}],
 "tools": [{"type": "web_search"}]}
```
- 응답 파싱: `output[]` 중 `type=="message"` 항목의 `content[]` 중 `type=="output_text"`의
  `.text`를 이어붙임.
- 인용: 루트 `citations` 배열(+ output_text의 `annotations[]` url_citation). 존재하면
  본문 뒤에 `\n\n[검색 출처]\nURL…` 형태로 덧붙여 종합 입력에 포함(§3.6 병합 규칙의
  링크 유지에 활용).

**2) `call_openai(prompt, key)` — OpenAI GPT**
- POST `https://api.openai.com/v1/responses`
- 헤더: `Authorization: Bearer <key>`, `Content-Type: application/json`
- 요청 본문 (`input`은 문자열 가능).
```json
{"model": "gpt-5.5",
 "tools": [{"type": "web_search"}],
 "input": "<조사 프롬프트>"}
```
- 응답 파싱: `output[]`에는 `web_search_call` 항목이 섞여 있으므로 `type=="message"`만
  걸러 `content[0].text`를 취함.
- 인용: `content[0].annotations[]`의 url_citation → Grok과 동일하게 `[검색 출처]` 부록.

**3) `call_zai(prompt, key)` — Z.AI GLM-5.2**
- POST `https://api.z.ai/api/paas/v4/chat/completions` (OpenAI 호환)
- 헤더: `Authorization: Bearer <key>`, `Content-Type: application/json`
- 요청 본문.
```json
{"model": "glm-5.2",
 "messages": [{"role": "user", "content": "<조사 프롬프트>"}],
 "tools": [{"type": "web_search",
            "web_search": {"enable": true,
                           "search_result": true,
                           "search_recency_filter": "oneWeek"}}]}
```
  - `search_recency_filter`는 `"oneWeek"` 사용 — 프롬프트가 "24시간 우선, 부족 시 3일까지
    확장"이므로 `oneDay`는 3일 확장을 막는다. `count`는 기본값(10) 사용.
  - `search_engine` 값은 문서 간 상이(search_pro_jina vs search-prime)하므로 **보내지 않고**
    실호출로 검증한다(테스트 계획 §5 참고).
- 응답 파싱: `choices[0].message.content`. 루트 `web_search[]`(title/link/publish_date)가
  있으면 `[검색 출처]` 부록으로 덧붙임(GLM은 본문에 링크가 빠질 수 있어 특히 중요).

**4) `call_claude_research(prompt, key)` — Anthropic Claude 조사 호출**
- POST `https://api.anthropic.com/v1/messages`
- 헤더: `x-api-key: <key>`, `anthropic-version: 2023-06-01`, `Content-Type: application/json`
- 요청 본문.
```json
{"model": "claude-opus-4-8",
 "max_tokens": 16000,
 "tools": [{"type": "web_search_20260209", "name": "web_search", "max_uses": 15}],
 "messages": [{"role": "user", "content": "<조사 프롬프트>"}]}
```
- 응답 파싱: `content[]` 블록 배열에서 `type=="text"` 블록들의 `.text`를 이어붙임
  (`web_search_tool_result` 블록이 섞여 있으므로 필터 필수).
- **`stop_reason` 처리 (필수)**
  - `"pause_turn"` — 서버측 도구 루프 일시정지. 기존 user 메시지 + 이번 응답의 assistant
    `content` 전체를 messages에 붙여 재요청하면 이어서 진행. **최대 5회 상한**,
    초과 시 그때까지 모인 텍스트가 있으면 그것을 반환, 없으면 예외.
  - `"refusal"` — 예외 처리(미참여 사유 로그에 stop_reason 명기).
  - `"max_tokens"` — 잘린 텍스트라도 반환(로그에 경고).
- temperature/top_p 파라미터는 이 모델에서 400 에러 — **보내지 않는다**.
- max_tokens는 16000 이하 유지(비스트리밍 타임아웃 예방), HTTP 타임아웃 600초.

**5) `call_claude_synthesize(prompt, key)` — Claude 종합 호출 (1회)**
- 같은 엔드포인트·헤더·모델. **tools 없음.** `max_tokens: 16000`.
- messages의 user content에 종합 프롬프트(4개 결과 텍스트 포함, §3.6)를 넣어 1회 호출.
- 응답 파싱·stop_reason 처리는 조사 호출과 동일(도구가 없으므로 pause_turn은 사실상
  없지만 방어적으로 동일 처리).

#### 병렬 실행 (표준 라이브러리 threading 채택)

- 조사 호출 4개는 각각 수 분씩 걸릴 수 있으므로 직렬이면 최악 40분. `threading.Thread`
  4개로 병렬화한다(대기 시간의 99%가 네트워크 I/O라 GIL 영향 없음, 스레드당 메모리
  수 MB 수준 — 2GB NAS에 부담 없음).
- 각 스레드는 결과를 `results[제공자명]`, 실패 사유를 `errors[제공자명]`에 기록
  (딕셔너리 키 단순 대입은 파이썬에서 원자적이라 락 불필요).
- `join(timeout=900)` — 15분이 지나도 안 끝난 스레드는 "미참여(시간 초과)" 처리하고
  진행(daemon 스레드라 프로세스 종료를 막지 않음).
- 제공자별 재시도: 예외 발생 시 10초 대기 후 **1회만** 재시도, 그래도 실패하면 미참여.

### 3.7 새 파일: `news_bot.py` — 상주 프로세스

첫 줄 한국어 역할 주석(예: `# AI 뉴스 리포트 봇 — 매일 정해진 시각에 4개 LLM 조사→종합→텔레그램 전송 (bot.py와 별도 상주 프로세스)`).

주요 함수와 데이터 흐름.

| 함수 | 시그니처 | 동작 |
|---|---|---|
| `load_news_settings` | `() -> dict` | `OUTPUT_DIR/news_settings.json` 로드. 없거나 파싱 실패 시 기본값(`ridi_collector.load_settings`와 동일 패턴 — 알 수 없는 키 무시, 기본값 병합) |
| `save_news_settings` | `(s: dict) -> None` | 같은 경로에 indent=2로 저장 |
| `load_api_keys` | `() -> dict` | `/app/api_keys.json`(스크립트 옆 경로) 로드. 없으면 전부 빈 값 |
| `build_research_prompt` | `(today: str) -> str` | §3.5 프롬프트 생성 |
| `build_synthesis_prompt` | `(today, results, absent) -> str` | §3.6 프롬프트 생성 |
| `send_long` | `(text: str, chat_id: str) -> int` | 4096자 제한 대응 분할 전송(아래 §3.10). 성공 조각 수 반환 |
| `run_report` | `() -> None` | 파이프라인 본체(아래) |
| `main` | `() -> None` | 시작 로그 → 무한 루프(아래) |

`run_report()` 데이터 흐름.
1. `keys = load_api_keys()` — 빈 키 제공자는 `absent[이름] = "키 없음"`.
2. `prompt = build_research_prompt(today)`.
3. 키 있는 제공자만 스레드 생성 → `news_llm.call_grok / call_openai / call_zai /
   call_claude_research` 병렬 실행(모듈 속성 경유 호출 — 테스트에서 몽키패치 가능하도록
   `news_llm.call_grok(...)` 형태 유지). 실패는 `absent[이름] = "호출 실패"` 등으로.
4. `results`가 비어 있으면 → 실패 알림(§3.11) 전송 후 종료.
5. `anthropic` 키가 없으면 종합 불가 → 실패 알림("Anthropic 키 없음 — 종합 단계 수행 불가")
   후 종료. ※ Anthropic 키는 이 설계에서 **필수 키**다(종합이 Claude 1회 호출로 확정).
6. `call_claude_synthesize()` 1회(실패 시 10초 후 1회 재시도) → 실패하면 실패 알림.
7. 헤더 부착 → `send_long(report, owner_chat_id)`.

`main()` 루프 (bot.py의 스케줄 판정 규칙을 그대로 따름 — 함정 8번).

```python
while True:
    try:
        s = load_news_settings()
        now = datetime.now()
        if now.strftime("%H:%M") >= s["daily_time"] and \
           s["last_run_date"] != now.strftime("%Y-%m-%d"):
            s["last_run_date"] = now.strftime("%Y-%m-%d")
            save_news_settings(s)          # ★ 실행 '시작 직후' 기록 (아래 근거)
            run_report()
    except Exception:
        traceback 로그                      # 프로세스는 절대 죽지 않음
    time.sleep(20)
```

- **시각 비교는 `==`가 아니라 `>=`** — 함정 8번 그대로. 20초 주기 루프가 정확한 1분을
  놓쳐도 그 이후 실행된다.
- **`last_run_date` 기록 시점은 bot.py와 달리 '실행 시작 직후'로 한다.** bot.py는 종료 시
  기록하지만, 뉴스봇의 작업은 **유료 API 호출**이라 NAS 재부팅 등으로 실행 도중 컨테이너가
  재시작되면 "매번 처음부터 재호출 → 과금 반복" 위험이 있다. 시작 직후 기록하면 최악의
  경우 "그날 리포트 1회 유실"로 끝난다(과금 반복보다 안전). 재시도는 두지 않는다 —
  다음 날 정상 실행되고, 이 트레이드오프는 §6 미해결 질문에도 명시.
- 뉴스봇은 조사~전송을 루프 스레드에서 동기(순차)로 수행한다 — 하루 1회 실행이므로
  bot.py처럼 백그라운드 스레드로 뺄 이유가 없다(중복 실행 자체가 불가능한 구조).

### 3.8 설정 파일 스키마: `news_settings.json`

위치: `OUTPUT_DIR/news_settings.json` (= 컨테이너 `/app/out/`, NAS에선
`docker/bombam/work/` 마운트 폴더). **폴더 마운트 안의 파일이므로 함정 5번(단일 파일
마운트 금지)과 무관하고**, 재빌드 없이 File Station에서 수정할 수 있으며 컨테이너를
다시 만들어도 값이 보존된다.

최초 실행 시 파일이 없으면 뉴스봇이 아래 기본값으로 자동 생성.

```json
{
  "daily_time": "08:00",
  "last_run_date": ""
}
```

| 키 | 의미 | 변경 방법 |
|---|---|---|
| `daily_time` | 매일 리포트 실행 시각(HH:MM, 한국시간 — 컨테이너 TZ=Asia/Seoul) | File Station에서 파일 열어 수정. 루프가 매회 새로 읽으므로 저장 즉시 반영(재시작 불필요) |
| `last_run_date` | 오늘 이미 실행했는지 기록(YYYY-MM-DD). 뉴스봇이 자동 기록 | 당일 재실행 테스트 시 `""`로 비우면 됨(기존 settings.json의 `last_run_date` 요령과 동일) |

기존 `settings.json`은 이전/이후 모두 **변화 없음** (스키마 비변경 근거로 명시).

### 3.9 로그·생존성

- 모든 출력은 `print(..., flush=True)` + 접두사 `[news]` — 댓글봇 로그와 한 컨테이너
  로그에 섞이므로 접두사로 구분한다.
- 시작 시 1회: `[news] === AI 뉴스봇 시작 === daily_time=08:00 키보유: openai, anthropic`
  (키 값 자체는 절대 로그에 남기지 않고 보유 여부만).
- `main()` 최상위와 루프 내부 이중 try/except — 뉴스봇의 어떤 오류도 프로세스 종료로
  이어지지 않게 한다(entrypoint가 재기동해 주지 않으므로).

### 3.10 텔레그램 전송 방식 (분할 전송)

- 확인 결과 **`notify.py`에는 분할 로직이 없다** (4096자 초과 시 텔레그램 API가 400을
  반환해 전송 실패). 종합 리포트는 수천~수만 자가 예상되므로 분할이 필수다.
- `notify.py`는 수정하지 않고(수술적 수정 원칙), `news_bot.py`에 `send_long()`을 둔다.

`send_long(text, chat_id)` 동작.
1. 텍스트를 **3500자** 기준으로 자르되, 경계는 가장 가까운 줄바꿈(`\n`)에서 끊는다
   (한 항목이 두 메시지에 걸쳐 찢기는 것 최소화). 3500자로 여유를 두는 이유는
   조각 머리표시(`[2/5]`) 추가분과 안전 마진.
2. 조각이 2개 이상이면 각 조각 앞에 `[i/N]\n`을 붙인다.
3. 각 조각을 `notify_telegram(조각, chat_id=chat_id)`(기존 notify.py 재사용)로 순차 전송,
   조각 사이 `time.sleep(0.7)` — 텔레그램 초당 전송 제한 회피.
4. 중간 조각이 실패해도 로그만 남기고 나머지 조각은 계속 전송(부분이라도 도착하는 게 낫다).

수신자 결정 — **소유자 1명 고정.**
- `notify._load_telegram()["chat_id"]`(= telegram.json의 276954538)를 **명시적으로**
  `chat_id` 인자로 넘긴다.
- 함정 9번 검토: `notify.py`는 `TELEGRAM_CHAT_ID` 환경변수를 telegram.json보다 우선하지만,
  그 환경변수는 bot.py가 `4_매일알림.py` **자식 프로세스에만** 주입하는 값이라 뉴스봇
  프로세스에는 존재하지 않는다. 게다가 `notify_telegram(chat_id=...)`처럼 인자를 명시하면
  환경변수·파일 어느 쪽도 타지 않으므로 이중으로 안전하다. `subscribers` 목록은 뉴스봇과
  무관(사용 안 함).

| 함정 # | 관련성 | 대응 |

| 3. 무인 컨테이너에서 input() 금지 | 직접 | 뉴스봇은 stdin을 전혀 읽지 않음. 모든 실패는 로그+텔레그램 알림으로 처리 |

| 11. 네트워크 복원력 | **직접** | 제공자별 1회 재시도, 스레드 join 타임아웃, 뉴스봇 루프 최상위 예외 방어(§3.9). 뉴스봇은 getUpdates를 안 쓰므로 백오프 로직은 불필요 |
| (로컬 테스트 주의) 409 Conflict | **직접** | 뉴스봇은 sendMessage만 사용 — 폴링 충돌 원천 배제(§3.1). 단, 로컬 테스트 시 bot.py 폴링 충돌 주의는 기존과 동일 |

추가 엣지케이스.
- **컨테이너 시작 시각이 daily_time 이후인 경우** — `>=` 비교라 그날 아직 실행 전이면
  시작 직후 곧바로 리포트가 1회 실행된다(bot.py와 동일 동작). 의도된 동작으로 문서화.
- **뉴스 실행(08:00)과 댓글 수집(11:00)의 겹침** — 기본값이 달라 겹치지 않지만, 겹쳐도
  프로세스·파일이 분리돼 있어 문제없다(뉴스봇은 CPU를 거의 안 씀 — 네트워크 대기가 전부).
- **리포트가 극단적으로 길 때** — 종합 max_tokens 16000 상한 + 3500자 분할이므로
  최대 약 15~20조각. 조각 간 0.7초 대기로 레이트리밋 회피(§3.10).

## 6. 미해결 질문 (사용자 결정 필요)

1. **실제 보유한 API 키가 무엇인가?** (xAI / OpenAI / Z.AI / Anthropic 중)
   특히 **Anthropic 키는 이 설계에서 필수**다 — 종합 단계가 Claude 1회 호출로 확정돼
   있어, Anthropic 키가 없으면 매일 실패 알림만 온다. 없다면 종합 담당 모델을 보유 키
   중 하나로 바꾸는 설계 변경이 필요하다.
2. **기본 실행 시각 08:00으로 좋은가?** (댓글 수집 11:00과 겹치지 않게 제안한 값.
   출근 전 확인 용도면 07:00~08:00, 다른 선호 시각이 있으면 지정)
3. **실행 도중 컨테이너가 재시작되면 그날 리포트는 건너뛰는 정책(과금 반복 방지 우선)으로
   확정해도 되는가?** 리포트 유실이 더 싫다면 "실패 시에만 25분 뒤 1회 재시도"(댓글봇의
   retry_at 방식)를 추가할 수 있으나, 유료 API 중복 과금 가능성이 생긴다.
4. **일일 API 비용 상한에 대한 우려가 있는가?** 웹검색 포함 4개 조사 + Claude 종합
   1회는 하루 수백 원~수천 원 수준으로 예상된다(모델·검색 횟수에 따라 변동).
   비용이 부담이면 참여 모델 수를 줄이는 것이 가장 확실한 절감책이다.
5. **테스트 시나리오 7(실 API 1회 호출, 소액 과금)을 진행해도 되는가?** 구현 완료 후
   실행 전에 다시 확인받는다(작업 전 리뷰 규칙).
