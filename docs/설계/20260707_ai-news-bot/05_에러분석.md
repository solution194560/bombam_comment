# 에러 분석: 뉴스봇 설정 파일 교정 저장 누락 (시나리오 7) + 테스트 시나리오 7건 미수행

## 1. 증상

### 1-1. 버그 — 깨진 JSON이 파일에 교정 저장되지 않음
- **환경**: 로컬 Docker 테스트(`bombam:test`), 04_테스트결과.md 시나리오 7.
- **현상**: `news_settings.json`을 깨진 JSON(형식이 망가진 파일)으로 만들어 두면,
  봇은 메모리(실행 중인 프로그램 안)에서는 기본값 `10:00`으로 잘 대체하고 경고 로그도
  남기지만, **디스크의 파일은 깨진 상태 그대로 남는다.** 03_최종설계.md §3.5의
  "파일에는 교정값을 다시 저장해 상태를 눈에 보이게 한다" 요구 위반.
- **로그 원문** (이번 분석에서 함수 단위로 재현한 실측값. 케이스 B가 문제 지점):

  ```
  --- 케이스 B: 깨진 JSON ('{"daily_time": "10:00", ')
  [news] news_settings.json 읽기 실패(Expecting property name ...) → 기본값 사용
    반환값: {'daily_time': '10:00', 'last_run_date': ''}
    디스크 파일 내용: '{"daily_time": "10:00", '   ← 깨진 채 그대로 (버그)

  --- 케이스 C: 유효 JSON + daily_time="aa:bb"
  [news] daily_time 형식 오류(입력값: aa:bb) → 기본 10:00 사용
    디스크 파일 내용: 정상 JSON으로 재저장됨 ✓ (이 경로는 정상)
  ```

- **부가 증상**: 메인 루프(`news_bot.py:437`)가 20초마다 설정을 다시 읽으므로, 깨진
  파일이 방치되면 `읽기 실패` 경고 로그가 **20초마다 무한 반복**된다 (로그 오염).

### 1-2. 테스트 공백
04_테스트결과.md에 03_최종설계.md §4가 요구한 11개 시나리오 중 **1, 2, 7, 11(제외)만
기록**되어 있고, 시나리오 3~6(성공 경로 모의·URL 보존 분할·부분 실패 표기·내부 오류
알림)과 8~10(크래시 복구·재시작 후 스케줄·하루 1회 보장)은 수행/기록 자체가 없다.

## 2. 근본 원인

### 2-1. 버그의 구조적 원인 (`news_bot.py` `load_news_settings()`, 53~96행)

`load_news_settings()`에는 "저장이 필요한 상태"가 세 종류 있는데, 그중 **두 종류만**
저장 조건에 연결돼 있다.

| 상태 | 감지 위치 | 저장 트리거 | 결과 |
|---|---|---|---|
| 파일이 아예 없음 | 93행 `not os.path.exists(...)` | 있음 ✓ | 기본값 생성됨 (케이스 A 정상) |
| 값 수준 오류 (`daily_time` 형식 등) | 75~78행 / 86~88행에서 `corrected = True` | 있음 ✓ | 교정 저장됨 (케이스 C·D 정상) |
| **파일은 있으나 파싱 실패 (깨진 JSON)** | 60~61행 except 블록 | **없음 ✗** | **깨진 채 방치 (케이스 B 버그)** |

동작 흐름을 따라가면 이렇다.

1. 58행 `json.load()`가 깨진 JSON에서 예외를 던져 **60~61행** except 블록으로 진입
   — 여기서는 로그만 남기고 아무 플래그도 세우지 않는다. `raw`는 빈 dict(`{}`)로 남는다.
2. 62행 `s.update(raw)` — 빈 dict라 `s`는 기본값 그대로다.
3. **64행에서야 `corrected = False`가 선언**된다. 즉 60행 except 시점에는 `corrected`
   변수가 아직 존재하지도 않아, 그 자리에서 `corrected = True`를 세우려 해도 64행이
   곧바로 `False`로 덮어쓰는 구조다 (선언 순서 자체가 함정).
4. 69~78행 `daily_time` 검증 — `s["daily_time"]`은 기본값 `"10:00"`이라 **정상 통과**,
   `corrected`는 `False` 유지. 81~90행 `last_run_date`도 `""`라 통과.
5. 93행 `if not os.path.exists(NEWS_SETTINGS_FILE) or corrected:` — 파일은 **존재**하고
   (`False`) `corrected`도 `False` → **저장이 실행되지 않는다.**

**왜 한쪽만 플래그를 세우는가** — `corrected`는 "개별 값을 교정했다"는 추적용으로 설계됐고,
"파일 자체가 못 읽힘"은 93행의 `not os.path.exists()`(파일 없음)로만 커버하려 했다.
그런데 "파일은 존재하지만 내용을 못 읽는" 상태는 두 분류 사이의 **사각지대**에 떨어진다.
깨진 JSON은 기본값으로 대체되는 순간 모든 값 검증을 '정상'으로 통과해 버리기 때문에
`corrected`가 켜질 기회가 없다.

**04_테스트결과.md의 원인 서술 정정** — 테스터는 "파일에 저장하는 로직이 함수 말미에
누락"이라고 썼으나 부정확하다. 저장 로직은 **93~94행에 이미 존재**하며, 문제는 깨진 JSON
경로에서 저장 **조건이 참이 되지 않는 것**이다. 판정(pipeline-judge)의 진단(60~61행에서
`corrected = True` 미설정)이 정확하다. 수정 시 함수 말미에 저장 블록을 또 추가하면
중복 코드가 되므로 주의.

**CLAUDE.md 함정 목록 대조** — 함정 1~11 모두 해당 없음 (브라우저·Xvfb·볼륨 마운트·
409·OOM·수신자·별점과 무관). 기존 함정의 재발이 아닌 **신규 유형**(설정 파일 자가 교정
로직의 경로 누락)이다.

### 2-2. 테스트 공백의 원인 (추정)

04_테스트결과.md의 기록 순서(1 → 2 → 7 → 11 제외 → 종합 판정)와 문서 내용으로 볼 때,
가장 개연성 높은 시나리오는 다음과 같다.

- **주원인 추정 — 시나리오 7 실패 후 조기 중단.** 테스터는 정적 코드 리뷰 단계에서 이미
  "교정 저장 로직 문제"를 발견했고(04 문서 "코드 리뷰 주요 포인트"에 명시), 시나리오 7로
  이를 실증하자마자 "배포 불가 — 수정 필요" 판정을 내리고 잔여 시나리오를 생략한 것으로
  보인다. 문서 말미의 "시나리오 7 재실행 확인 → 재시도 통과 후 배포 진행" 권고와, 테스트
  환경까지 이미 정리(docker stop/rm/rmi)한 점이 이를 뒷받침한다.
- **보조 요인 추정** — 시나리오 3~6은 `docker exec`로 몽키패치(테스트를 위해 함수를
  가짜로 바꿔치기) 스크립트를 작성해야 하는 고비용 시나리오라 뒤로 미뤄졌을 가능성,
  또는 세션 시간/토큰 한도에 걸렸을 가능성이 있다. 테스터 세션 로그가 없어 확정은 불가
  (→ 8장 미해결 질문).
- **판단** — 실패를 발견했더라도 시나리오 8~10(크래시 복구·재시작·하루 1회)은 시나리오 7과
  **독립적**이라 수행 가능했다. 스펙(§4)은 "전 시나리오 통과 후에만 zip 재생성"을 요구하므로
  전 시나리오의 수행·기록은 필수였다. 재실행 시 지시문으로 강제해야 한다 (§5-2 문구 참조).

## 3. 재현 조건

`news_settings.json` 파일이 **존재하면서** `json.load()`가 실패하는 모든 경우에 100%
재현된다. 확인된 트리거는 다음과 같다.

- 깨진 JSON 문법 (예: `{"daily_time": "10:00", ` — 이번 분석에서 재현 확정).
- 루트가 dict가 아닌 유효 JSON (예: `[1,2]` — 59행 `data.items()`에서 AttributeError,
  같은 except로 진입하므로 동일 증상).
- 파일 인코딩 파손 등 읽기 자체 실패.

반면 "유효 JSON + 잘못된 값"(`"aa:bb"`, `"8:00"`, 잘못된 `last_run_date`)은 정상적으로
교정 저장된다 — 버그는 오직 **파싱 실패 경로**에 국한된다.

## 4. 수정 방향 (버그 1)

### 권장안 — except 블록을 `corrected` 플래그에 통합 (최소 수정)

`news_bot.py` `load_news_settings()` 한 함수만 수정한다. 변경 요지는 두 가지다.

1. **`corrected = False` 선언을 파일 읽기 앞으로 이동** — 현재 64행에 있는 선언을
   55행(`raw = {}`) 부근으로 올린다. except 블록보다 먼저 선언돼야 그 안에서 값을
   세울 수 있고, 이후 덮어써지지 않는다.
2. **60~61행 except 블록에 `corrected = True` 추가** — "파일은 있는데 못 읽음"을
   "저장 필요 상태"로 편입시킨다. 93행의 저장 조건(`not os.path.exists(...) or corrected`)은
   그대로 두면 된다 — 저장 블록을 새로 추가하지 않는다 (04 문서의 제안 코드처럼 말미에
   `if corrected: save...`를 또 넣으면 93~94행과 중복).

수정 후 기대 동작 — 깨진 파일은 다음 로드(최대 20초 내)에 정상 JSON으로 덮어써지고,
`읽기 실패` 로그의 20초 간격 무한 반복도 1회로 끝난다. 사용자가 File Station에서 파일을
열면 교정된 상태가 보인다 (§3.5 취지 충족).

대안 (참고) — 별도 플래그(`broken = True`)를 두고 93행 조건에 or로 추가하는 방법도
동작은 같다. 로그를 구분할 수 있다는 장점이 있으나 변수가 하나 늘고, 이미 61행 로그로
원인이 구분되므로 이득이 없다. **권장안(플래그 통합)이 최소 구현 원칙에 부합한다.**

### 함께 점검할 인접 로직 (회귀 방지)

- **`last_run_date` 검증(81~90행)** — `corrected = True`를 이미 세우고 있어 정상.
  수정 불필요.
- **`load_api_keys()`(107~125행)** — 같은 패턴의 사각지대가 있다. `api_keys.json`이
  깨진 JSON이면 로그만 남기고 빈 키로 진행하며 파일을 재생성하지 않는다. 다만 §3.8
  스펙은 "파일이 **없으면** 생성"만 요구하고, 깨진 키 파일을 빈 스키마로 **덮어쓰면
  사용자가 입력해 둔 실키가 날아갈 위험**이 있으므로 여기는 현행 유지(로그만)가 오히려
  안전하다. **이번 수정 범위에 포함하지 말 것** — 단, 구현자가 "같은 패턴이니 같이
  고치자"고 확장하지 않도록 명시적으로 제외를 지시한다.
- **`daily_time` 키가 아예 없는 유효 JSON**(예: `{}`) — 기본값이 병합되지만
  `corrected`가 켜지지 않아 파일에 키가 채워지지 않는다. §3.5는 "교정값 저장"만
  요구하므로 스펙 위반은 아니며, 값이 정상 동작하므로 이번 수정에서 제외 (언급만).
