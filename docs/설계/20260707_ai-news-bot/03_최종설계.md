## 3. 확정 스펙 상세 (변경·신규 부분만)

### 3.1 entrypoint.sh — 크래시 복구 루프

`sleep 2`와 `exec "$@"` 사이에 아래 블록 삽입. 그 외는 무변경.

```bash
# AI 뉴스 리포트 봇 — 죽으면 30초 후 자동 재기동 (별도 상주 프로세스, 로그는 컨테이너 로그에 합류)
(
  while true; do
    python -u /app/news_bot.py || true
    echo "[news] 뉴스봇 프로세스 종료 감지 — 30초 후 재기동"
    sleep 30
  done
) &

exec "$@"
```

- `|| true` **필수** — entrypoint.sh는 `set -e`라서(현재 파일 4행 확인) 이것이 없으면
  뉴스봇이 비정상 종료하는 순간 서브셸의 while 루프 자체가 죽어 재기동이 안 된다.
- exec 구조·PID 1(bot.py)은 그대로(함정 2 유지). import/문법 오류·OOM kill·미상 크래시
  전부 30초 후 자동 재기동. 반복 크래시여도 30초 간격이라 로그 폭주 없음.
- `news_bot.py` 내부의 최상위 + 루프 내부 이중 try/except(01 §3.9)는 그대로 유지(이중 방어).
- 뉴스봇 크래시 시 좀비 항목 1개가 잠시 남을 수 있으나 30초 내 재기동으로 실질 영향 없음.

### 3.2 타임아웃 체계 — 소켓 timeout이 유일한 실제 차단 수단

- **`join(timeout=900)` 삭제 → `join()` 무기한 대기.** 스레드는 강제 중단이 불가능하므로
  타임아웃 처리 후 진행하면 "미참여 처리된 호출이 백그라운드에서 계속 과금"될 수 있다.
  무기한 join이면 늦은 결과 기록·상태 꼬임 문제 자체가 소멸한다.
- 실제 상한은 `urlopen(timeout=600)`이 보장한다. 제공자 스레드 최악 소요 =
  600(1차) + 10(대기) + 600(재시도) = **1210초(약 20분)**.
- **Claude 조사(pause_turn 재개 루프)에만 벽시계 상한 추가** — 재개 포함 총 경과가
  **1800초(30분)** 를 넘으면 남은 재개를 중단하고, 모인 텍스트가 있으면 반환/없으면 예외.
  (pause_turn 최대 5회 상한(01)과 병행 — 둘 중 먼저 걸리는 쪽 적용.)
- 스레드 daemon=True 유지, "미참여(시간 초과)" 분류는 삭제(무기한 join이므로 발생 불가) —
  실패 분류는 "키 없음 / 호출 실패 / 빈 응답"만 남는다.
- 잔여 리스크(문서화): 소켓 timeout은 무응답 구간 기준이라 서버가 찔끔찔끔 응답을 흘리면
  600초를 넘길 수 있다. 하루 1회 배치라 허용한다.

### 3.3 종합 입력 절단 (신설)

`build_synthesis_prompt()` 진입 **전에** 각 제공자 결과에 적용한다.

1. `[검색 출처]` 부록의 URL은 중복 제거 후 **최대 30개**만 유지.
2. 결과 전체(부록 포함)가 **12,000자** 초과 시 12,000자 이내의 마지막 줄바꿈에서 절단하고
   끝에 `\n…(이하 원문 절단됨)` 을 부착 — 종합 모델이 절단 사실을 인지하게 한다.
3. 4개 모두 최대여도 종합 입력 ≈ 48,000자 + 프롬프트로, Claude 컨텍스트·요청 크기에
   안전하고 입력 비용 상한이 예측 가능해진다.

### 3.4 last_run_date 정책 (절충 확정)

- **기록 시점은 '실행 시작 직후' 유지**(01 §3.7 그대로) — 유료 API 과금 반복 방지 우선.
- 대신 **`run_report()` 본체 전체를 try/except로 감싸**, 어떤 예외(구현 버그 포함)에도
  소유자에게 실패 알림을 보낸다. 형식: `⚠️ AI 뉴스 리포트 실패 ({날짜})\n내부 오류: {예외 1줄 요약}` +
  전체 traceback은 `[news]` 로그로. 실패 알림 전송 자체가 실패하면 로그만 남긴다.
- 결과: 어떤 경로로 실패하든 "그날 리포트 유실"이 조용히 지나가지 않는다.
  수동 복구 방법(사용자 안내용): File Station에서 `work/news_settings.json`의
  `last_run_date`를 `""`로 비우면 다음 루프(20초 내)에 즉시 재실행.

### 3.5 news_settings.json 스키마 + 값 검증 (기본 시각 변경)

최초 실행 시 `OUTPUT_DIR/news_settings.json` 자동 생성. 기본값이 **10:00**으로 변경됐다.

```json
{
  "daily_time": "10:00",
  "last_run_date": ""
}
```

`load_news_settings()` 검증 규칙 (직접 편집 파일이므로 필수).
- `daily_time` — `datetime.strptime(v, "%H:%M")` 실패 시(`"8:00"`, `"24:00"`, `"aa:bb"` 등)
  `"10:00"`으로 대체하고 `[news] daily_time 형식 오류(입력값: ...) → 기본 10:00 사용` 경고 로그.
  파일에는 교정값을 다시 저장해 상태를 눈에 보이게 한다.
- `last_run_date` — `strptime(v, "%Y-%m-%d")` 실패 시 `""` 처리(= 오늘 미실행으로 간주).
- 알 수 없는 키 무시·기본값 병합, 매 루프 재로드(저장 즉시 반영)는 01 그대로.

### 3.6 응답 파싱 규칙 (전 제공자 공통 강화)

- **xAI·OpenAI (Responses API)** — 응답 루트에 `output_text` 문자열이 있고 비어 있지
  않으면 그것을 사용. 없으면 `output[]` 중 `type=="message"` 항목들의 `content[]`를
  전부 순회해 `.text` 필드를 가진 블록을 모두 이어붙인다(첫 블록만 보지 않는다).
- **Anthropic** — `content[]` 중 `type=="text"` 블록 전부 이어붙임(01 그대로).
  도구 타입은 **`web_search_20260209` 유지**(검토 지적은 오탐 — 2026-07 공식 최신 타입
  확인 완료, `allowed_callers` 지정 불필요). `stop_reason` 처리(pause_turn/refusal/max_tokens)는
  01 §3.4 그대로 + §3.2의 30분 벽시계 상한.
- **Z.AI** — `choices[0].message.content`(01 그대로). 요청 본문도 01 그대로
  (boolean `enable`/`search_result`, `search_engine` 미전송 — 공식 문서 재확인 완료).
- **공통** — 파싱 직후 응답 루트의 상위 키 목록과 `stop_reason`/`finish_reason` 값을
  `[news]` 로그로 남긴다(본문·키값은 로그 금지, 텍스트는 길이만). 최종 텍스트가 빈
  문자열이면 예외(→ 미참여(빈 응답)).

### 3.7 send_long 분할 규칙 (교체)

1. 텍스트를 줄(`\n`) 단위로 순회하며 누적 버퍼에 담고, "버퍼 + 다음 줄"이 **3500자**를
   초과하는 순간 버퍼를 조각으로 확정한다 — 줄 중간 절단이 원천적으로 없어 URL이 깨지지 않는다.
2. **단일 줄 자체가 3500자를 초과**하면 그 줄 안에서 3500자 이내의 마지막 공백에서 절단,
   공백이 없으면 3500자 하드컷(최후 수단).
3. 조각 머리표시 `[i/N]`, 조각 간 `sleep(0.7)`, 실패 조각 로그 후 계속 진행,
   `notify_telegram(조각, chat_id=소유자)` 명시 인자 — 01 §3.10 그대로.

### 3.9 키 없음 처리 흐름 (초기 상태 = 정상 동작)

사용자는 현재 키 미보유이며 최소 Anthropic 키만 발급 예정 — 아래가 **초기 정상 상태**다.

`run_report()` 순서 (01 §3.7의 1~7단계를 이 순서로 교정).
1. `keys = load_api_keys()` — 빈 키 제공자는 `absent[이름] = "키 없음"`.
2. **`anthropic` 키가 없으면 즉시 실패 알림 후 종료** — 조사 호출을 아예 하지 않는다
   (다른 키가 있어도 종합 불가인데 유료 조사만 낭비하는 것 방지. 01은 조사 후 검사였음 — 변경).
   알림 문구: `⚠️ AI 뉴스 리포트 실패 ({날짜})\nAnthropic 키 없음 — 종합 단계 수행 불가.\nwork/api_keys.json에 anthropic 키를 입력하면 다음 날부터 동작합니다.`
3. 키가 하나도 없으면 전원 "키 없음" 실패 알림(2번 문구에 포함되는 형태로 통합 가능).
4. 이후 조사 병렬 호출 → 종합 → 헤더 부착 → 분할 전송은 01 §3.7 그대로.
   Anthropic 키만 있는 경우 = Claude 단독 조사 + 종합, 리포트 서두에 미참여 3개 표기 — 정상 리포트.
5. 기존 bot.py·컨테이너에는 어떤 경우에도 영향 없음(별도 프로세스, §3.1 재기동 루프 포함).
