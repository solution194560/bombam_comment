# 설계 검토 결과

## [심각도: 높음] `news_bot.py &` 방식은 장애 복구 목표를 달성하지 못함

설계는 `entrypoint.sh`에서 `python -u /app/news_bot.py &` 후 `exec "$@"`를 유지합니다. 현재 `entrypoint.sh`는 Xvfb를 띄운 뒤 `exec "$@"`만 수행하는 구조입니다. 이 방식은 기존 함정은 피하지만, 설계의 성공 기준인 “뉴스봇에 어떤 오류가 나도 댓글봇과 컨테이너 자체는 죽지 않는다”까지만 만족하고, “뉴스봇이 매일 안정적으로 돈다”는 운영 목표는 만족하지 못합니다.

문제는 `news_bot.py`가 죽어도 PID 1인 `bot.py`나 Docker `restart: unless-stopped`가 이를 감지하지 못한다는 점입니다. 설계서도 이 한계를 인정하지만 “try/except로 절대 죽지 않게”만으로는 import 오류, 문법 오류, OOM, SIGKILL, 초기화 전 예외를 막을 수 없습니다. 또한 `bot.py`가 PID 1로 자식을 수거하지 않으므로 크래시 시 좀비 프로세스가 남을 수 있습니다.

개선 제안: `entrypoint.sh` 한 줄 추가 대신 최소 supervisor 역할을 하는 작은 `process_manager.py` 또는 shell wrapper를 PID 1로 두고 `bot.py`와 `news_bot.py`를 각각 자식으로 관리하세요. 댓글봇이 죽으면 컨테이너를 종료해 Docker 재시작을 타게 하고, 뉴스봇만 죽으면 지수 백오프로 재기동하는 구조가 더 명확합니다. 외부 패키지 없이 표준 라이브러리 `subprocess`로 충분합니다.

## [심각도: 높음] 스레드 타임아웃 설계가 실제 중복 과금 방지와 맞지 않음

설계는 제공자별 `urlopen(timeout=600)`, 실패 시 10초 후 1회 재시도, 전체 `join(timeout=900)`을 둔다고 합니다. 하지만 `join(timeout=900)`으로 “시간 초과 처리 후 진행”해도 해당 스레드는 실제로 계속 살아 있을 수 있습니다. Python 스레드는 강제 중단할 수 없고, `urllib` 호출이 DNS/TLS/서버 지연으로 예상보다 오래 붙잡히면 뒤늦게 결과를 쓰거나 재시도를 수행할 수 있습니다.

그 상태에서 종합 호출과 텔레그램 전송이 진행되면 같은 실행 안에서 “미참여 처리된 API 호출”이 백그라운드로 계속 과금될 수 있고, 다음 루프나 종료 시점과도 상태가 꼬입니다. `daemon` 스레드라 프로세스 종료를 막지 않는다는 설명도 상주 프로세스에서는 실질적 보호가 아닙니다.

개선 제안: 스레드별 타임아웃을 `urlopen`의 실제 timeout과 재시도 예산으로만 제어하고, `join`은 “모든 작업 종료”를 기다리게 하세요. 전체 실행 상한이 필요하면 스레드 대신 `multiprocessing` 또는 별도 subprocess로 제공자 호출을 실행해 시간 초과 시 프로세스를 종료할 수 있게 설계해야 합니다.

## [심각도: 높음] 종합 프롬프트 입력 크기 상한과 절단 정책이 없음

4개 모델이 각각 카테고리당 최대 10개, 총 40개 항목을 만들 수 있고, 각 항목에 링크·날짜·CVE·설명·출처 부록이 붙습니다. 이를 4개 제공자 결과 그대로 Claude 종합 입력에 넣으면 입력이 매우 커질 수 있습니다. 설계는 `max_tokens: 16000`을 출력 상한처럼 다루지만, 입력 토큰·검색 결과 토큰·비용 상한·요청 크기 초과에 대한 정책이 없습니다.

Anthropic 문서도 웹 검색 결과가 입력 토큰으로 계산된다고 명시합니다. OpenAI 문서 역시 긴 리포트성 웹 검색은 별도 긴 실행 경로를 권장합니다. 현재 설계대로면 하루 1회 작업이지만 비용과 실패 가능성이 예측 불가능합니다.

개선 제안: 제공자별 결과를 종합 전에 강제로 제한하세요. 예: 제공자별 최대 20,000자, 출처 URL 최대 40개, 카테고리별 상위 5개, URL 중복 제거 후 절단. 절단 시 “이하 원문 절단됨”을 종합 프롬프트에 명시해야 합니다.

## [심각도: 높음] Anthropic `web_search_20260209` 사용 명세가 불완전할 가능성

설계는 Claude 조사 호출에 `web_search_20260209`를 사용합니다. Anthropic 공식 문서 기준으로 `web_search_20260209` 이후 버전은 동적 필터링을 추가하며, `allowed_callers` 기본값이 `direct`가 아니라 `code_execution_20260120` 쪽으로 바뀐다고 설명되어 있습니다. 현재 설계는 code execution 도구도, `allowed_callers`도 지정하지 않습니다.

그 결과 기대한 “Claude가 직접 웹 검색”이 되지 않거나 400 계열 오류가 날 가능성이 있습니다. `pause_turn` 처리는 문서 방향과 맞지만, 도구 정의 자체가 실패하면 전체 Claude 조사 및 종합 품질이 흔들립니다.

개선 제안: 단순 웹 검색이면 `web_search_20250305`를 쓰거나, `web_search_20260209`를 유지한다면 공식 문서에 맞춰 `allowed_callers`를 명시하고 실제 API 1회 검증을 테스트 필수 항목으로 승격하세요.

## [심각도: 중간] `last_run_date`를 시작 직후 기록하는 정책은 알림 유실 범위가 너무 넓음

유료 API 중복 과금을 줄이려는 의도는 타당합니다. 다만 현재 정책은 설정 파싱 성공 직후, 실제 키 검증·API 호출·텔레그램 전송 전에 날짜를 기록합니다. 그러면 네트워크 일시 장애, Telegram 15초 타임아웃, Anthropic 종합 실패, NAS 재부팅뿐 아니라 단순 구현 버그도 그날 리포트 유실로 귀결됩니다.

개선 제안: `last_run_date` 하나로 상태를 표현하지 말고 `run_state`, `started_at`, `completed_date`를 분리하세요. 중복 과금이 걱정되면 `started_at`이 오늘이고 2시간 이내면 재실행 금지, 2시간 이후에는 사용자에게 실패 알림만 보내는 식으로 복구 가능성을 남기는 편이 낫습니다.

## [심각도: 중간] `news_settings.json` 직접 편집 방식인데 값 검증 설계가 부족함

기존 `bot.py`는 `/time` 명령에서 `datetime.strptime(t, "%H:%M")`로 시간을 검증합니다. 반면 뉴스봇은 File Station에서 직접 `news_settings.json`을 수정하는 구조인데 `daily_time` 검증 실패 시 어떻게 처리할지 명확하지 않습니다. 문자열 비교로 `"8:00"`, `"24:00"`, `"aa:bb"`가 들어오면 의도치 않은 실행 또는 영구 미실행이 가능합니다.

개선 제안: `load_news_settings()`에서 `HH:MM` 형식을 검증하고 잘못된 값이면 기본값으로 대체하면서 실패 알림 또는 로그를 남기세요. `last_run_date`도 `YYYY-MM-DD` 형식만 인정하는 편이 안전합니다.

## [심각도: 중간] Z.AI Web Search in Chat 요청 스키마가 공식 예시와 다름

설계는 GLM 호출에서 `web_search.enable`과 `search_result`를 boolean으로 보냅니다. Z.AI 문서의 Web Search in Chat 예시는 `"enable": "True"`, `"search_result": "True"`, `"search_engine": "search-prime"` 형태를 사용합니다. 별도 Web Search API 문서에는 `search_recency_filter`가 `search_pro_jina` 지원 항목으로 표시되어 있어, 설계서의 “search_engine은 보내지 않는다” 판단은 실호출 전에는 위험합니다.

개선 제안: GLM은 반드시 실 API 검증 전까지 “확정 스펙”으로 쓰지 말고, 실패 시 fallback으로 별도 `/web_search` API를 먼저 호출한 뒤 검색 결과를 GLM 채팅에 넣는 단순 경로를 준비하세요.

## [심각도: 중간] API 응답 파싱 경로가 너무 단일 형태에 의존함

OpenAI와 xAI Responses API는 `output_text` 헬퍼가 있고, `output[]`에는 메시지 외 도구 호출이 섞입니다. 설계는 `content[0].text` 또는 `type=="output_text"` 중심으로 파싱하는데, 여러 content 블록, annotation 위치, SDK/REST 응답 차이를 충분히 흡수하지 못합니다.

개선 제안: 파서는 `output_text`가 있으면 우선 사용하고, 없으면 `output[].type=="message"`의 모든 `content` 텍스트 블록을 순회하는 방식으로 작성하세요. 각 제공자별 원본 JSON의 상위 키와 `stop_reason`/`finish_reason`은 로그에 남겨야 합니다.

## [심각도: 중간] 텔레그램 분할 전송은 일반 텍스트라 안전하지만 URL 절단 문제가 남음

`notify.py`는 `parse_mode`를 쓰지 않으므로 Markdown 깨짐은 큰 문제가 아닙니다. 종합 프롬프트도 마크다운 금지를 지시합니다. 다만 3500자 기준 줄바꿈 분할은 긴 URL, 괄호 없는 출처 목록, 한 줄짜리 긴 문단에서 URL을 중간 절단할 수 있습니다. 그러면 클릭 가능한 출처가 깨집니다.

개선 제안: 분할 기준을 “줄” 단위로 누적하고, 한 줄이 3500자를 넘을 때만 강제 절단하세요. URL 패턴 내부에서는 자르지 않도록 마지막 공백 기준 fallback을 두는 것이 좋습니다.

## [심각도: 중간] `api_keys.json` 이미지 COPY는 운영 편의성은 좋지만 보안 경계가 약함

기존 `account.json`, `telegram.json` 패턴과 일관된다는 근거는 맞습니다. 하지만 AI API 키 4개를 이미지에 넣으면 Docker image layer, `bombam_synology.zip`, 백업 파일, NAS 프로젝트 폴더에 키가 장기간 남습니다. 키 교체에도 재빌드가 필요합니다.

개선 제안: 단일 파일 마운트 금지 제약을 피하려면 `/app/out/api_keys.json`처럼 이미 마운트된 work 폴더 안에서 읽는 방식을 우선 검토하세요. 이 경우 재빌드 없이 키 교체가 가능하고, 이미지 레이어에 키가 남지 않습니다. 초보자 UX가 걱정되면 첫 실행 때 `/app/out/api_keys.sample.json`을 생성하면 됩니다.

## [심각도: 낮음] 테스트 계획이 실제 상주 프로세스 위험을 충분히 덮지 못함

몽키패치로 `run_report()`를 직접 호출하는 테스트는 비용을 줄이는 데 유용하지만, 이미 떠 있는 `news_bot.py` 프로세스의 스케줄 루프, entrypoint 기동 순서, 크래시 후 복구, Docker stop/restart, 좀비 프로세스 여부는 검증하지 못합니다.

개선 제안: 테스트에 다음을 추가하세요. `news_bot.py` import 실패 시 컨테이너 로그와 댓글봇 영향 확인, 뉴스봇 프로세스 강제 kill 후 재기동 여부 확인, Docker restart 후 `last_run_date` 동작 확인, `news_settings.json` 파손/잘못된 시각 검증, 텔레그램 400/429 모의 응답 검증.

## [심각도: 낮음] 더 단순한 대안이 충분히 검토되지 않음

4개 LLM 병렬 조사 후 Claude 종합은 품질 목표에는 맞지만, 2GB NAS와 하루 1회 운영에는 과합니다. 웹검색 포함 모델 4개는 비용·장애면·입력 크기를 모두 키웁니다.

개선 제안: 1차 구현은 “OpenAI 또는 Claude 1개 모델 + 웹검색 + 텔레그램 분할”로 끝내고, 이후 교차검증을 2개 모델로 확장하는 단계적 접근이 더 단순합니다. 사용자가 반드시 4개 모델을 원한다면 현재 설계처럼 구현하되, 입력 절단과 비용 상한을 먼저 넣어야 합니다.
