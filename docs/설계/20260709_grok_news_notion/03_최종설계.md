# 최종설계: Grok 뉴스 보고서 자동 생성·Notion 업로드

이 문서는 구현의 유일한 기준이다. 초안(`01_설계.md`)·검토(`02_검토결과.md`)의 결론이
모두 반영된 완결본이며, 구현자는 이 문서만 보고 구현한다.

**확정된 사용자 결정(변경 불가)**
- 로그인은 **맥에서 수동 로그인한 Grok 브라우저 프로필을 NAS로 복사**하는 방식.
  account.json 자동로그인 아님. 세션 만료 시 "실패 알림 → 맥 재로그인 → 프로필 재복사" 운영 수용.
- xAI API 미사용. Playwright로 grok.com 직접 접속, `HEADLESS=0`(함정 1).
- 매일 07:00 실행, 기존 NAS 봄밤 컨테이너에 통합. 범위는 **Notion 업로드까지**
  (텔레그램 발신은 실패 알림 1건뿐).

## 0. 검토 반영 내역

코드로 사실 여부를 검증한 뒤 결정했다(Dockerfile `COPY . .` 확인, `.dockerignore` 원문 확인,
`bot.py`의 `_job_running`·`retry_at` 로직 확인, `news_notion.find_today_report` 존재 확인 등).

| 검토 지적 | 심각도 | 결정 | 사유 |
|---|---|---|---|
| 1. `local_test/grok_profile`이 Docker 이미지에 포함될 수 있음 — `.dockerignore` 보강 필요 | 높음 | **수용** | 사실로 확인. Dockerfile이 `COPY . .`이고 현재 `.dockerignore`에 `local_test/`·`.git/`·`*.zip`이 없다. 지금도 `local_test/work/api_keys.json`(Notion 토큰)·리디 프로필이 이미지 레이어에 들어가며, Grok 세션 쿠키가 추가되면 위험이 커진다. ASCII 패턴만으로 보강(함정 12 준수) |
| 2. 2GB NAS 브라우저 동시 실행을 운영 수칙만으로 막는 것은 부족 — 공유 락 필요 | 높음 | **수용** | 사실로 확인. `/run` 수동 실행은 아무 때나 가능하고 `4_매일알림.py`는 실패 시 `retry_at`(+25분) 자동 재시도를 예약한다. `bot.py`의 `_job_running`은 프로세스 내부 Event라 `news_grok.py`와 공유되지 않는다. 2GB RAM에서 Chromium 2개 동시는 OOM으로 컨테이너 전체를 위협 → `OUTPUT_DIR/browser.lock` **mkdir 원자 락**을 신규 모듈 `browser_lock.py`로 만들고 양쪽이 사용한다. 초안의 "bot.py 수정 없음"은 철회하고 `run_daily_job()`에 획득/해제만 최소 추가(§3.7) |
| 3. Grok 웹 UI 불확실성이 너무 늦게 검증됨 — 스파이크 우선 | 중간 | **수용** | 이 기능의 성패는 grok.com DOM·완료 감지에 달려 있다. `news_grok.py --probe` 모드를 만들어 **구현 1단계에서 로그인 판별·입력·전송·응답 추출만 먼저 검증**하고, 그 결과로 셀렉터·완료 신호를 확정한 뒤 Notion·스케줄을 붙인다(§3.2, §5 0단계). 별도 probe 파일 대신 본 파일의 CLI 모드로 넣어 스로어웨이 코드를 줄인다 |
| 4. `last_run_date` 시작 직후 기록 시 일시 장애가 당일 누락이 됨 | 중간 | **부분수용** | 취지(일시 장애 시 당일 재시도) 수용. 단 "성공 후 기록" 방식은 반복 실패 시 20초 루프마다 브라우저를 재구동(2GB RAM·계정 부하)해 위험하므로 반려하고, **함정 11의 기존 `retry_at` 패턴을 재사용**한다 — `last_run_date`는 시작 직후 기록(무한 재구동 방지 유지), 실패 시 `retry_at = 지금+30분`을 기록해 **당일 1회만 자동 재시도**. 재시도 실패 시 추가 재시도 없음(하루 최대 2회 시도). 키 분리(`last_success_date`) 없이 스키마 최소로 같은 효과를 얻는다 |
| 5. Notion 페이지 생성이 멱등적이지 않고 부분 페이지가 남음 | 중간 | **수용** | 기존 `news_notion.find_today_report()`를 그대로 재사용해 **브라우저 구동 전에** 같은 제목 페이지 존재를 검사, 있으면 "이미 업로드됨"으로 성공 처리(중복 생성 방지 + 재시도 경로의 낭비 방지). append 도중 실패 시 부분 페이지는 남기지 않고 `archived: true` PATCH로 정리(백업 `.md`가 있으므로 부분 페이지를 남길 이유가 약하다는 검토 의견 수용) |
| 6. 마크다운 변환 범위가 출처 보존 요구에 비해 약함 | 중간 | **수용** | 프롬프트에 "표 금지 + 출처는 `[매체명](URL), YYYY-MM-DD` 형식" 강제(§3.4), 변환기에 **bare URL 자동 링크화** 추가, rich_text 2,000자 분할은 서식 조각(세그먼트) 단위로 처리(§3.6). 표·코드블록 등 나머지 미지원은 유지(프롬프트로 금지했으므로 최소 구현 충분) |
| 7. 프롬프트에 웹 UI에서 보장되지 않는 도구 사용 지시 잔존 | 낮음 | **수용** | 원문 §6의 "도구(web_search, browse_page 등)를 적극 활용" 문구를 "웹에서 확인 가능한 최신 보도 기준으로 조사하고 출처 URL을 반드시 포함"이라는 결과 요구로 치환(§3.4) |
| 8. 상주 루프 대신 작업 스케줄러 `docker exec` 대안 | 낮음 | **반려** | ① 컨테이너 재시작 후 보정 실행(`>=` 비교)은 내부 루프만 가능 — 검토자 스스로 이 경우 루프 유지를 인정. ② 사용자는 GUI만 쓰는 초보라 DSM 작업 항목이 늘수록 운영이 복잡해진다. ③ 기존 `news_bot.py` 상주 패턴과 일관. 검토자가 단 조건("내부 루프면 공유 락 필수")은 지적 2 수용으로 충족 |

검토의 "미해결 질문 답변"도 반영했다 — 완료 감지에 **최소 생성 시간 90초** 추가와 품질
게이트에 **마지막 섹션 존재 확인** 추가(§3.3), 클립보드는 best-effort로 격하하고 innerText를
**정상 경로**로 승격(§3.3), 계정·세션 이식성은 미해결 질문으로 유지(§7).

## 1. 목표와 성공 기준

매일 07:00에 상주 프로세스가 grok.com에 저장 프로필로 접속해 프롬프트로 IT·클라우드·보안
뉴스 보고서를 생성시키고, 응답을 수집해 Notion 'AI NEWs' 페이지
(`396a0b7e561a80edbb31e60dcc86148a`) 하위에 `Grok news_{YYYY-MM-DD}` 페이지로 업로드한다.

**성공 기준**
1. probe(§5 0단계)로 grok.com 로그인 판별·프롬프트 전송·응답 추출이 실증된다.
2. 로컬 Docker 컨테이너에서 지정 시각에 자동 실행되어 Notion 하위 페이지가 실제 생성된다.
3. 세션 만료·Cloudflare 미통과·응답 미완성·Notion 오류 등 어떤 실패에도 프로세스가 멈추거나
   (`input()` hang) 죽지 않고, 실패 알림 1건 후 30분 뒤 1회 재시도, 그래도 실패면 다음 날을 기다린다.
4. 기존 `bot.py`·`news_bot.py` 동작에 영향이 없다(별도 프로세스·별도 프로필·별도 설정 파일).
   단 리디 수집과 Grok 실행은 공유 락으로 **동시에 브라우저를 띄우지 않는다**.

## 3. 상세 설계

### 3.1 실행 흐름 전체

```
[news_grok.py 상주 프로세스 — 시계만 보는 루프. getUpdates 절대 호출 안 함(409 무관)]
  20초마다 grok_settings.json 로드(검증 포함) 후,
  ① retry_at이 설정돼 있고 now >= retry_at:
       retry_at 제거·저장 → run_grok_report(record_retry=False)   ← 재실패해도 추가 재시도 없음
  ② 아니고 now.strftime("%H:%M") >= daily_time 이고 last_run_date != 오늘:
       last_run_date = 오늘 기록·저장(시작 직후 — 무한 재구동 방지)
       → run_grok_report(record_retry=True)                        ← 실패 시 retry_at = 지금+30분 기록

[run_grok_report(record_retry) — 1회 실행 본체. 모든 예외를 잡아 실패 알림 1건]
  1. api_keys.json에서 notion 토큰 확인 → 없으면 실패 알림 후 종료
  2. 멱등성 검사 — find_today_report(parent, token, f"Grok news_{오늘}")가 있으면
     "이미 업로드됨 — 스킵" 로그 후 성공 종료 (브라우저 구동 전, 비용 0)
  3. 공유 브라우저 락 획득 — browser_lock.acquire("grok", wait_sec=600)
     10분 내 실패 시 실패 알림("리디 수집 진행 중으로 추정") 후 종료
     (record_retry=True 경로면 retry_at이 기록되어 30분 뒤 재시도 — 그때는 보통 풀림)
  4. GROK_PROFILE의 Singleton* 잔여 락 삭제(함정 4)
  5. persistent context 구동 (프로필=GROK_PROFILE, headless=HEADLESS 해석은 리디와 동일,
     --no-sandbox --disable-dev-shm-usage, permissions=["clipboard-read"])
  6. https://grok.com 접속 → Cloudflare 대기(최대 90초)
  7. 로그인 상태 확인 → 미로그인이면
     · sys.stdin.isatty()이고 HEADLESS=0이면(맥 부트스트랩) "창에서 로그인 후 Enter" 대기
     · 아니면(무인) "이번 실행 건너뜀" + 실패 알림("맥에서 grok_profile 재로그인 후
       NAS grok_profile/에 복사 필요") 후 종료(함정 3 — input() hang 금지)
  8. grok_prompt.txt 로드 → {현재 날짜}를 YYYY-MM-DD로 치환 → 입력창 입력 → 전송
  9. 응답 완료 감지 (§3.3 — 최소 90초 + 안정화 60초 + 하드 타임아웃 20분)
 10. 응답 텍스트 수집 (innerText 정상 경로, 복사 버튼→클립보드는 best-effort 우선 시도)
 11. 품질 게이트(§3.3) 미달이면 실패 알림 후 종료(부분 보고서 업로드 안 함)
 12. 원문 백업 저장 — OUTPUT_DIR/grok_news_{YYYY-MM-DD}.md (Notion 실패 대비 유실 방지)
 13. 브라우저 닫기 (Notion 업로드 전에 닫아 메모리 반환 — 2GB RAM 배려)
 14. markdown_to_blocks() → create_report_page()로 "Grok news_{YYYY-MM-DD}" 페이지 생성
     append 도중 실패 시 해당 페이지를 archive_page()로 정리 후 예외 재발생
 15. 성공 로그(성공 알림은 보내지 않음). finally에서 락 해제·브라우저 확실히 닫기
```

### 3.2 `news_grok.py` — 상수·함수 시그니처·CLI

첫 줄에 한국어 헤더 주석을 단다(전역 규칙). 주석 문장은 콜론으로 끝내지 않는다.

```python
# Grok 웹 자동화 봇 — 매일 아침 grok.com에서 뉴스 보고서를 생성시켜 Notion에 업로드 (별도 상주 프로세스)

GROK_URL = "https://grok.com"
GROK_PROFILE = os.environ.get("GROK_PROFILE", "/app/.grok_profile")
OUTPUT_DIR = os.environ.get("OUTPUT_DIR", ".")
GROK_SETTINGS_FILE = os.path.join(OUTPUT_DIR, "grok_settings.json")
PROMPT_FILE = os.path.join(os.path.dirname(os.path.abspath(__file__)), "grok_prompt.txt")
NOTION_PARENT_PAGE_ID = "396a0b7e561a80edbb31e60dcc86148a"
HEADLESS = os.environ.get("HEADLESS", "1") != "0"   # 리디와 동일 규칙(0=실제 창)

def load_grok_settings() -> dict     # news_bot.load_news_settings와 동일 검증 패턴(§3.5)
def save_grok_settings(s: dict) -> None
def _clear_singleton_locks() -> None # GROK_PROFILE 안 SingletonLock/Cookie/Socket 삭제(함정 4)
def _new_grok_browser(p)             # persistent_context(GROK_PROFILE, HEADLESS, 안정화 플래그,
                                     #   permissions=["clipboard-read"])
def _wait_cloudflare(page, limit=90) -> bool  # 리디 것과 동일 로직 자체 구현(12줄 내외)
def _is_logged_in(page) -> bool      # 프롬프트 입력창 존재 여부로 판단(§3.3 셀렉터 후보)
def _submit_prompt(page, text) -> None
def _wait_response_done(page, hard_timeout=1200) -> bool   # §3.3 완료 감지
def _extract_report(page) -> str     # innerText 정상 경로 + 복사버튼 best-effort(§3.3)
def _quality_gate(text: str) -> str | None   # 통과 None, 미달이면 사유 문자열 반환(§3.3)
def _notify_failure(today: str, message: str) -> None  # notify_telegram으로 소유자 1건.
                                     # 전송 자체가 실패하면 로그만 남기고 계속(news_bot 패턴)
def run_grok_report(record_retry: bool = True) -> None  # §3.1의 1~15 오케스트레이션
def probe() -> None                  # §5 0단계 스파이크(아래 설명)
def main() -> None                   # 20초 시계 루프(§3.1). 함정 8 준수(>= 비교)
```

**CLI 모드**
- `python news_grok.py` — 상주 루프(main).
- `python news_grok.py --once` — 스케줄 무시하고 즉시 1회 `run_grok_report(record_retry=False)`.
  `last_run_date`·`retry_at`을 **기록하지 않음**(테스트가 정기 실행을 막지 않도록 — 함정 8 취지).
- `python news_grok.py --probe` — **스파이크 검증 전용**(검토 지적 3 수용).
  Notion을 호출하지 않고 다음만 수행한다.
  1) 브라우저 구동 → grok.com 접속 → Cloudflare 통과 → 로그인 판별 결과 출력
  2) 짧은 테스트 프롬프트("한 문장으로만 답해줘. 오늘 날짜는?") 전송
  3) 완료 감지 → 응답 추출 → `OUTPUT_DIR/grok_probe_response.txt` 저장
  4) 각 단계에서 `OUTPUT_DIR/grok_probe_{단계}.png` 스크린샷 저장(셀렉터 확정용 증거)
  probe도 락을 획득한다(맥 로컬에선 사실상 무경합이지만 코드 경로 통일).

### 3.3 grok.com 자동화 — 셀렉터 후보·완료 감지·텍스트 수집·품질 게이트

grok.com DOM은 미지다. **아래 후보는 probe(§5 0단계)에서 실 브라우저로 확인해 확정하고,
확정값을 코드 상수로 채운다** — "구현 시 실제 확인해 채울 지점"이다.
함정 10 교훈대로 CSS-in-JS 해시 클래스명에는 절대 의존하지 않는다
(태그·`aria-label`·`role`·`placeholder`·구조 위치만 사용).

| 대상 | 셀렉터 후보(우선순위순) | 폴백 |
|---|---|---|
| 프롬프트 입력창(=로그인 판별 겸용) | `textarea` → `div[contenteditable="true"]` → `[aria-label]`/`[placeholder]`에 "Ask"/"Grok" 류 문구 | 입력창이 하나도 없으면 미로그인으로 판정 |
| 전송 | 입력 후 `Enter` 키 | `button[type="submit"]`, `button[aria-label*="Send" i]`/`*="Submit" i` 클릭 |
| 생성 중 정지 버튼(보조 신호 B) | `button[aria-label*="Stop" i]` | 못 찾으면 신호 B 미사용(없어도 동작) |
| 응답 영역 | 프롬프트 전송 후 새로 생긴 마지막 대화 블록(전송 전 대비 DOM 변화로 특정) | `main` 등 본문 영역 innerText에서 프롬프트 원문 이후 부분 추출 |
| 복사 버튼 | 응답 말미 `button[aria-label*="Copy" i]` | 실패 시 innerText 경로 |

**완료 감지 — `_wait_response_done()` 확정 로직** (검토자 의견 반영으로 초안 대비 강화)
- 5초 간격으로 응답 영역 텍스트의 길이+해시를 폴링한다.
- 완료 판정 조건 3가지를 **모두** 만족해야 한다.
  1) 전송 후 **최소 90초 경과**(검색 준비 단계의 짧은 정지를 완료로 오판 방지)
  2) **60초 연속 무변화**(안정화 창)
  3) 현재 텍스트가 품질 게이트의 "마지막 섹션" 조건을 이미 만족하거나, 보조 신호
     (정지 버튼 소실 / 입력창 재활성화)가 확인됨 — 보조 신호는 probe에서 사용 가능이
     확인된 경우에만 켠다(없어도 1·2 + 마지막 섹션 검사로 동작)
- **하드 타임아웃 20분** 초과 시 실패(부분 텍스트 폐기).

**텍스트 수집 — `_extract_report()`** (검토자 의견 수용으로 우선순위 재정의)
- **정상 경로는 innerText**다. 응답 영역 innerText는 렌더링 텍스트라 `#` 마커가 소실되어
  헤딩이 paragraph로 강등될 수 있다 — 내용은 보존되므로 이 열화를 수용한다.
- 복사 버튼 → `navigator.clipboard.readText()`는 **best-effort 1차 시도**로만 쓴다
  (성공하면 마크다운 원문이라 변환 품질이 좋음). 클릭·권한·포커스 어느 하나라도 실패하면
  조용히 innerText로 넘어간다. 클립보드 실패가 전체 실패가 되어서는 안 된다.

**품질 게이트 — `_quality_gate()`** (함정 6과 같은 취지 — 불량 데이터 업로드 방지)
- 최소 길이 1,500자 미만 → 실패("응답이 너무 짧음 — 오류/거절 응답 의심").
- "개요" 또는 "Executive Summary" 문자열 미포함 → 실패.
- 마지막 섹션 표지("종합 트렌드" 또는 "권고") 미포함 → 실패(중간 절단 감지 — 검토자 의견).

### 3.5 설정 스키마 — `grok_settings.json` (OUTPUT_DIR = work 폴더)

### 3.10 브라우저 프로필·Dockerfile·compose

- **프로필 분리(필수)** — Grok은 `GROK_PROFILE=/app/.grok_profile`(신규 ENV), 리디는 기존
  `BROWSER_PROFILE=/app/.browser_data_dir`. 프로필 공유 시 Singleton 락 충돌(함정 4)·세션 오염.
- `Dockerfile` — 기존 ENV 블록에 `ENV GROK_PROFILE=/app/.grok_profile` 한 줄 추가.
- `docker-compose.yml` volumes에 `- /volume1/docker/bombam/grok_profile:/app/.grok_profile`
  한 줄 추가(폴더 마운트 — 함정 5 준수).
- 로컬 테스트는 `-v "$(pwd)/local_test/grok_profile:/app/.grok_profile"`.
- **최초 로그인 부트스트랩(사용자 확정 방식)** — 맥에서
  `GROK_PROFILE=./local_test/grok_profile OUTPUT_DIR=./local_test/work HEADLESS=0 python3 news_grok.py --probe`
  로 창을 띄우고, 미로그인 감지 시 isatty+HEADLESS=0 조건에서 "창에서 로그인 후 Enter"
  대기(§3.1의 7). 로그인하면 세션이 프로필 폴더에 저장되고, 완료 후 그 폴더를 NAS
  `/volume1/docker/bombam/grok_profile/`에 File Station으로 복사한다.
  컨테이너/NAS에서는 isatty가 아니므로 이 대기는 절대 걸리지 않는다.
- `shm_size: "1gb"`(기존)와 `--disable-dev-shm-usage` 플래그를 그대로 따른다.

## 4. 엣지케이스·함정 대응 (CLAUDE.md 함정 목록 전 항목 대조)

| # | 함정 | 이 설계의 대응 |
|---|---|---|
| 1 | HEADLESS=1이면 Cloudflare에 막힘 | `HEADLESS` 환경변수를 리디와 동일 해석(0=실제 창). compose의 기존 `HEADLESS=0` 상속. 시작 로그에 모드 출력 |
| 2 | `xvfb-run` 래퍼 무한 hang | entrypoint의 기존 Xvfb(:99) 직접 구동 공유. 래퍼 안 씀 |
| 3 | 무인 환경 `input()` hang | 미로그인 시 `sys.stdin.isatty() and not HEADLESS`일 때만 수동 로그인 대기(맥 부트스트랩 전용). 컨테이너에선 건너뜀+실패 알림 |
| 4 | 프로필 `Singleton*` 잔여 락 | Grok 전용 프로필 분리 + 매 실행 전 SingletonLock/Cookie/Socket 선제 삭제 |
| 5 | 단일 파일 볼륨 마운트 금지 | 신규 마운트는 `grok_profile/` 폴더만. `grok_settings.json`은 work 폴더 안, `grok_prompt.txt`는 이미지에 COPY |
| 6 | 빈 데이터가 캐시로 굳음 | 캐시 없음. 같은 취지의 품질 게이트(1,500자·필수 헤딩·마지막 섹션)로 불량 보고서 업로드 차단 |
| 7 | mem_limit 금지 | compose에 mem_limit 추가 안 함. 동시 브라우저는 **공유 락**으로 차단(운영 수칙 아님 — 검토 수용) |
| 8 | 자동 실행 기록·`>=` 비교 | `now >= daily_time and last_run_date != 오늘`. `--once`/`--probe`는 기록 안 함 |
| 9 | 알림 수신자 규칙 | 발신은 실패 알림뿐 — 소유자 1인(telegram.json chat_id)에게만. subscribers 배포 없음 |
| 10 | CSS-in-JS 클래스명 의존 금지 | grok.com 셀렉터는 태그·aria-label·role·placeholder·DOM 변화 기반만(§3.3). probe로 확정 |
| 11 | 네트워크 장애 복원력 | getUpdates 자체를 안 함(409 무관). Notion은 기존 1회 재시도 정책 재사용. Grok 실패 시 `retry_at`(+30분) 1회 재시도 — 기존 리디 패턴과 동형 |
| 12 | `.dockerignore` ASCII만 | 보강 패턴 전부 ASCII. 신규 파일명도 전부 ASCII |

**실패 시나리오별 동작**

| 시나리오 | 동작 |
|---|---|
| notion 토큰 없음 | 외부 호출 전 즉시 실패 알림 → 종료 |
| 오늘 페이지 이미 존재(멱등성) | 브라우저 구동 없이 "이미 업로드됨 — 스킵" 성공 처리 |
| 브라우저 락 10분 미획득 | 실패 알림("리디 수집 진행 중 추정") → retry_at으로 30분 뒤 재시도 |
| Cloudflare 90초 미통과 | 브라우저 닫고 실패 알림 → retry_at 재시도 |
| Grok 세션 만료 | 건너뜀 + 실패 알림("맥에서 grok_profile 재로그인 후 NAS 복사 필요") — 재시도해도 같은 결과지만 무해(브라우저 1회 구동 비용뿐) |
| 입력창/전송 실패(UI 개편) | 예외 캐치 → 실패 알림("grok.com 화면 구조 변경 의심") |
| 응답 20분 미완·품질 게이트 미달 | 부분 텍스트 업로드 안 함, 실패 알림 → retry_at 재시도 |
| Notion 생성/append 실패 | 원문 `.md` 백업은 이미 저장됨. append 도중 실패 시 부분 페이지 archive 정리 → 실패 알림(백업 파일 경로 포함) → retry_at 재시도(멱등성 검사가 archived 페이지는 무시하므로 안전) |
| 컨테이너 재시작(07:00 이후) | `>=` 비교로 재시작 직후 실행. 이미 실행한 날은 last_run_date로 차단. "기록 직후·run 진입 전" 크래시면 그날 건너뜀(중복 페이지 방지 우선) — retry_at이 기록된 채 재시작해도 루프가 이어받아 재시도 |
| 같은 날 `--once` 반복 | 멱등성 검사로 두 번째부터 "이미 업로드됨" 스킵(중복 페이지 안 생김) |
| 실패 알림 자체가 실패 | 로그만 남기고 계속 |

## 5. 테스트 계획 (스파이크 우선 — 검토 지적 3 반영한 구현·검증 순서)

전제 — grok.com 실계정 세션이 필요해 CI식 무인 테스트는 불가. 반복 비용이 큰 실접속은
0·2단계에서 최소 횟수만, 반복 검증은 1단계(Notion 단독)에서 한다.

### 0단계 — 스파이크 probe (맥, 컨테이너 밖) ← **가장 먼저. 이 단계 통과 전 본 구현 금지**
```bash
cd "/Users/hyundeok/Downloads/봄밤작가 작품 리스트와 댓글 평가-3/"
GROK_PROFILE=./local_test/grok_profile OUTPUT_DIR=./local_test/work HEADLESS=0 \
  python3 news_grok.py --probe
# 기대:
#  · 크롬 창 → grok.com → (첫 회) 로그인 화면 감지 → "창에서 로그인 후 Enter" → 사람이 로그인
#  · 테스트 프롬프트 전송 → 응답 감지 → local_test/work/grok_probe_response.txt 에 응답 저장
#  · 단계별 스크린샷 grok_probe_*.png 저장 → 이 증거로 §3.3 셀렉터·완료 신호를 코드에 확정
#  · 로그인 세션이 local_test/grok_profile 에 저장됨(이후 단계·NAS 복사에 재사용)
```
probe 실패 시 셀렉터 후보를 수정해 재시도하고, **확정된 셀렉터를 이 문서 §3.3 표에 채워
넣은 뒤** 본 구현(Notion·스케줄 결합)으로 진행한다.

### 1단계 — Notion 함수 단독 검증 (브라우저 없이, 빠른 반복)

#       105개 이상 블록짜리 긴 md로도 실행해 100개 분할 append 확인.

### 4단계 — 실패 경로·락 검증
```bash
# (a) 로그인 만료: 빈 grok_profile로 마운트한 컨테이너에서
docker exec bombam_test python -u /app/news_grok.py --once
#     기대 — "무인 환경 + 미로그인 → 건너뜀" 로그 + 실패 알림 1건, 프로세스 생존
# (b) Notion 토큰 제거(api_keys.json notion="") 후 --once → 외부 접속 전 즉시 실패 알림
# (c) --once 가 last_run_date/retry_at 을 안 건드리는지 grok_settings.json 확인
# (d) docker restart bombam_test → [grok] 루프 자동 재기동 확인
# (e) 락 동작: 컨테이너에서 mkdir /app/out/browser.lock 후 --once 실행
#     기대 — "락 대기" 로그 30초 간격 → 10분 후 포기 + 실패 알림(짧게 보려면 대기 중 rmdir로 해제
#     → 즉시 진행 확인). 반대로 grok --once 진행 중 텔레그램 /run → 리디가 락 대기 로그 후
#     grok 종료 뒤 진행되는지 확인
# (f) retry_at 검증: grok_settings.json 에 retry_at 을 1분 뒤로 수동 기록 → 그 시각에
#     자동 재실행되고 retry_at 이 비워지는지 확인
```

### grok.com 실접속의 한계와 대안
- 응답 생성이 수 분 걸리고 계정 사용량 제한 가능성이 있어 실접속 완주는 0·2·3단계에서
  각 1~2회로 제한한다. 변환기·업로드·락·설정 검증 등 반복이 필요한 부분은 전부
  브라우저 없는 경로(1단계, 4단계 b·c·e·f)에서 수행한다.
- NAS 실환경(다른 IP)에서 세션이 유지되는지는 로컬에서 검증 불가 — 첫 배포 후 NAS에서
  1회 완주 확인 전에는 완료로 간주하지 않는다(§7).

## 6. 구현 시 주의사항

1. **구현 순서 고정** — ① `.dockerignore` 보강 → ② `news_grok.py`의 probe 경로(브라우저
   구동·로그인 판별·전송·추출)만 먼저 작성해 0단계 통과 → ③ 셀렉터 확정값 반영 →
   ④ `news_notion.py` 추가 함수 + 1단계 → ⑤ 스케줄 루프·락·entrypoint·compose 결합.
2. 새 소스 파일(`news_grok.py`, `browser_lock.py`) 첫 줄에 한국어 헤더 주석. 주석·로그
   문장은 콜론으로 끝내지 않는다. `grok_prompt.txt`는 프롬프트 원문이므로 헤더 주석 제외.
3. `news_notion.py`·`bot.py`는 명시된 지점 외 한 줄도 건드리지 않는다(수술적 수정).
   특히 `_get_json`·`find_today_report`·`run_daily_job`의 기존 로직 유지.
4. `news_grok.py`는 `ridi_collector`를 import하지 않는다(import 부작용). `notify.py`의
   `notify_telegram`(또는 news_bot과 동일한 소유자 전송 최소 헬퍼)과 `news_notion`,
   `browser_lock`만 import.
5. 브라우저 닫기·락 해제는 반드시 `finally`에서 — 어떤 실패 경로에서도 Chromium 좀비와
   락 잔존이 없어야 한다(락 잔존은 stale 75분 자활이 있지만 정상 경로에선 즉시 해제).
6. Grok 응답 대기 중 5초 폴링마다가 아니라 **60초에 1번꼴로만** 진행 로그를 찍는다
   (20분 대기 시 로그 도배 방지 — bot.py의 로그 절제 패턴과 동일한 취지).
7. 작업 폴더의 미추적 `표준/` 디렉토리는 한글명이라 `.dockerignore`로 제외할 수 없다
   (함정 12). NAS 배포는 git 작업본 기준이라 무관하지만, **로컬 빌드 이미지에는 포함되므로**
   로컬 테스트 전 해당 폴더를 프로젝트 밖으로 옮기거나 내용을 확인할 것(구현 범위 밖 — 사용자 안내 사항).
8. 커밋 단위 제안 — ⓐ `.dockerignore` 보강(독립 보안 수정) ⓑ Grok 자동화+Notion 업로드
   본체 ⓒ 공유 락(browser_lock.py + bot.py 수정). 논리 단위가 다르므로 나눠 제안한다.
9. 배포는 CLAUDE.md 표준 절차 — 로컬 Docker 검증 통과 후에만 main push → NAS `봄밤_배포`.
   NAS에는 사전에 `grok_profile/` 폴더 생성 + 맥 프로필 복사가 필요하다(GUI 안내 필수).

## 7. 남은 미해결 질문 (구현 착수 전 확인)

- **[사용자] Grok 계정·요금제** — grok.com 계정이 준비돼 있는가? 로그인 수단(X/Google/이메일)은
  무엇인가? 무료 요금제면 웹검색 포함 장문 보고서에 사용량 제한이 걸릴 수 있다(검토자도
  운영용 무료 요금제 비권장). **probe 단계 전에 확인 필수.**
- **[사용자] 세션 이식성 실검증** — 맥→NAS 프로필 복사가 grok.com에서도 유지되는지는
  NAS 실환경에서만 확인 가능하다. 첫 배포 후 NAS 1회 완주 확인까지를 완료 조건으로
  수용하는가? 무효화 시 "실패 알림 → 맥 재로그인 → 재복사" 운영은 사용자 확정대로 수용.
- **[구현자] probe 결과로 확정할 것** — §3.3 표의 셀렉터 실측값, 보조 신호 B·C 사용 가능
  여부, 클립보드 경로 동작 여부, 완료 감지 파라미터(90초/60초/20분)의 실측 기반 조정치.
