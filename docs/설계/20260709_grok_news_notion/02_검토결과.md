# 설계 검토 결과

## [심각도: 높음] `local_test/grok_profile`이 Docker 이미지에 포함될 수 있음

문제 설명  
설계의 테스트 순서는 먼저 `local_test/grok_profile`에 Grok 로그인 세션을 만든 뒤 `docker build`를 수행한다. 그런데 현재 `.dockerignore`는 `local_test/`, `.git/`, `*.zip` 등을 제외하지 않는다. 현재도 `local_test/work/api_keys.json`, `local_test/telegram_test.env`, 브라우저 프로필 일부가 빌드 컨텍스트에 포함될 수 있고, Grok 프로필까지 추가되면 세션 쿠키가 이미지 레이어에 들어갈 위험이 커진다.

개선 제안  
설계의 “`.dockerignore` 수정 없음”은 철회해야 한다. ASCII 패턴만 사용해 최소한 아래 항목을 제외하도록 설계를 바꿔야 한다.

```dockerignore
.git/
local_test/
*.zip
profile_cookies.zip
*.xlsx
```

또는 로그인 부트스트랩을 `docker build` 이후로 옮겨도 되지만, 근본적으로 `.dockerignore` 보강이 필요하다.

## [심각도: 높음] 2GB NAS에서 브라우저 동시 실행을 운영 수칙만으로 막는 것은 부족함

문제 설명  
설계는 Grok 07:00, 뉴스봇 10:00, 리디 11:00이라 겹치지 않는다고 본다. 그러나 실제 `bot.py`에는 사용자의 `/run` 수동 실행이 있고, `4_매일알림.py`는 실패 시 `retry_at`으로 25분 뒤 자동 재시도를 예약한다. 즉 07:00~07:20에 리디 Chromium과 Grok Chromium이 동시에 뜰 수 있다. 현재 `_job_running`은 `bot.py` 내부 스레드용이라 `news_grok.py`와는 공유되지 않는다.

개선 제안  
운영 수칙 대신 `OUTPUT_DIR` 아래 공유 락을 둬야 한다. 예를 들어 `browser.lock`을 `os.open(..., O_CREAT|O_EXCL)` 또는 `mkdir` 원자 연산으로 잡고, 리디 수집과 Grok 실행 모두 같은 락을 사용하게 한다. 락이 있으면 Grok은 “브라우저 작업 진행 중” 실패 알림 후 건너뛰거나, 짧은 대기 후 1회 재시도하는 쪽이 안전하다.

## [심각도: 중간] Grok 웹 UI 자동화의 핵심 불확실성이 너무 늦게 검증됨

문제 설명  
설계는 입력창, 전송 버튼, 정지 버튼, 복사 버튼 셀렉터를 구현 단계에서 확정한다고 한다. 하지만 이 기능의 성패는 거의 전부 grok.com DOM, 로그인 상태 판별, Cloudflare 통과, 응답 완료 감지에 달려 있다. Notion 업로드 로직보다 먼저 “Xvfb + headful Chromium + 저장 프로필”에서 Grok 프롬프트 전송과 응답 추출이 실제로 되는지 검증해야 한다.

개선 제안  
구현 순서를 바꿔야 한다.

1. `news_grok_probe.py` 또는 `news_grok.py --probe` 수준의 최소 스파이크로 로그인 판별, 입력, 전송, 응답 텍스트 추출만 먼저 검증한다.
2. 그 결과로 안정 셀렉터와 완료 신호를 설계 문서에 확정한다.
3. 이후 Notion 업로드와 스케줄 루프를 붙인다.

현재 문서의 §3.4와 §6은 불확실성을 인지하고 있지만, 성공 기준에 비해 검증 순서가 뒤에 있다.

## [심각도: 중간] `last_run_date`를 시작 직후 기록하면 일시 장애가 곧 당일 누락이 됨

문제 설명  
설계는 `last_run_date`를 실행 시작 직후 저장한다. 이는 중복·비용 방지에는 좋지만, Grok 웹 자동화는 Cloudflare, 세션 만료, 응답 지연, Notion append 실패 등 실패 지점이 많다. 한 번의 일시 장애로 그날 보고서가 완전히 누락된다.

개선 제안  
`last_attempt_date`와 `last_success_date`를 분리하는 편이 낫다. 최소 구현을 유지하려면 `last_run_date`는 성공 후 기록하고, 중복 페이지 방지는 Notion에서 동일 제목 페이지를 먼저 찾는 방식으로 해결한다. 비용 반복이 우려되면 `retry_count`를 1회로 제한하면 된다.

## [심각도: 중간] Notion 페이지 생성이 멱등적이지 않고 부분 페이지가 남을 수 있음

문제 설명  
설계는 같은 날 `--once`를 반복하면 동일 제목 페이지가 중복될 수 있음을 수용하고, append 도중 실패하면 부분 페이지를 남긴다고 한다. 하지만 기존 `news_notion.find_today_report()`는 이미 중복 child page를 최신 편집본으로 고르는 패턴을 갖고 있다. 새 Grok 업로드도 같은 날짜 제목을 먼저 검색해 중복 생성을 피하는 편이 운영상 낫다.

개선 제안  
`create_report_page()` 전에 같은 parent 아래 `Grok news_{YYYY-MM-DD}`가 있는지 확인한다. 있으면 생성하지 말고 실패 처리하거나, 명시적으로 archive 후 재생성한다. append 도중 실패한 페이지는 그대로 두기보다 `archived: true` PATCH로 정리하거나 제목 앞에 `[업로드 실패]`를 붙여 사용자가 정상 보고서와 혼동하지 않게 해야 한다.

## [심각도: 중간] 마크다운 변환 범위가 출처 보존 요구에 비해 약함

문제 설명  
설계의 변환기는 헤딩, 불릿, 번호 목록, divider, `[텍스트](URL)`, `**굵게**`만 지원한다. 하지만 Grok은 표, bare URL, 괄호 안 URL, 중첩 목록, 긴 출처 문장을 쉽게 생성한다. 특히 “모든 항목에 원문 링크 + 보도 날짜 포함”이 핵심 요구인데, bare URL을 링크로 변환하지 않으면 Notion에서 출처 품질이 떨어질 수 있다.

개선 제안  
프롬프트에서 “표 금지, 모든 출처는 `[매체명](URL), YYYY-MM-DD` 형식”을 강제하고, 변환기에는 bare URL 자동 링크화 정도는 추가해야 한다. `**굵게**`와 링크가 섞인 rich_text 분할도 2,000자 제한을 넘지 않도록 토큰 단위로 처리해야 한다.

## [심각도: 낮음] 프롬프트에 웹 UI에서 보장되지 않는 도구 사용 지시가 남아 있음

문제 설명  
원문 프롬프트에는 `web_search`, `browse_page` 도구를 적극 활용하라는 지시가 있다. Grok 웹 UI가 내부적으로 웹검색을 할 수는 있지만, 사용자가 제어 가능한 “도구 호출”과는 다르다. 이 문구는 모델이 불필요한 도구 호출 흉내나 설명을 출력하게 만들 수 있다.

개선 제안  
`grok_prompt.txt` 생성 시 “도구 사용” 표현을 제거하고, “웹에서 확인 가능한 최신 보도 기준으로, 출처 URL을 반드시 포함”처럼 결과 요구로 바꾸는 편이 안정적이다.

## [심각도: 낮음] 더 단순한 운영 대안이 있음

문제 설명  
별도 상주 루프를 추가하면 컨테이너 안에 `bot.py`, `news_bot.py`, `news_grok.py` 세 프로세스가 계속 돈다. `news_grok.py`는 하루 한 번만 실행하면 되므로 상주 프로세스가 꼭 필요하지 않다.

개선 제안  
시놀로지 작업 스케줄러에서 매일 07:00에 `docker exec bombam python -u /app/news_grok.py --once`를 실행하는 방식이 더 단순하다. 다만 컨테이너 재시작 후 보정 실행이 필요하면 현재 설계처럼 내부 루프를 유지할 수 있다. 내부 루프를 택한다면 위의 공유 브라우저 락은 필수다.

## 미해결 질문 답변

- Grok 계정·요금제  
  필수 선행조건이다. 무료 요금제는 장문 웹검색 보고서 생성량 제한에 걸릴 가능성이 높으므로 운영용으로는 권장하지 않는다. 로그인 수단은 프로필 이식성이 좋은 방식으로 정하고, 최초 NAS 실환경 1회 완주 전에는 구현 완료로 보면 안 된다.

- 세션 이식성 리스크 수용  
  리디에서 검증됐다는 사실만으로 Grok도 된다고 보면 안 된다. Grok은 IP·기기 변화에 더 민감할 수 있으므로 “실패 알림 → 재로그인 → 프로필 재복사” 운영을 수용하되, 첫 배포 전 NAS에서 실제 세션 유지 검증이 필요하다.

- 07:00~07:20 사이 `/run` 자제 수칙  
  운영 수칙만으로 갈음하는 것은 반대다. 2GB RAM 환경에서는 공유 브라우저 락을 넣어야 한다.

- 응답 완료 감지 파라미터  
  `5초 폴링 + 60초 무변화 + 20분 하드 타임아웃 + 품질 게이트`는 시작점으로는 가능하다. 다만 60초 무변화만으로 완료 판정하면 검색 중 장기 정지에서 조기 완료될 수 있으므로 정지 버튼 소실, 입력창 재활성화, 최소 생성 시간, 마지막 heading 포함 여부를 함께 봐야 한다.

- 클립보드 수집 전략  
  클립보드는 1순위 best-effort로 두되 주 전략으로 믿으면 안 된다. Xvfb/headful Chromium에서 권한·포커스·사용자 제스처 조건에 흔들릴 수 있다. 복사 버튼 실패 시 innerText 또는 DOM 구조 기반 추출이 반드시 정상 경로로 동작해야 한다.

- append 도중 실패 시 부분 페이지 잔존  
  남겨 두는 설계보다 archive 처리 또는 `[업로드 실패]` 제목 변경이 낫다. 백업 `.md`가 있으므로 부분 페이지를 정상 보고서처럼 남길 이유가 약하다.

- grok.com DOM 확정 절차  
  설계 시점에서 더 줄일 수 있는 가장 큰 불확실성은 실 브라우저 스파이크다. 구현 전에 입력창·전송·완료·복사 버튼 locator를 실제 페이지에서 확인하고, 실패 시 사용할 innerText 추출 범위까지 로그와 스크린샷으로 남기는 절차를 먼저 수행해야 한다.
