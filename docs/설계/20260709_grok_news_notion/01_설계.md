# 설계: Grok 뉴스 보고서 자동 생성·Notion 업로드

## 1. 목표와 배경

### 무엇을 만드는가
매일 아침 07:00에 봇이 **grok.com 웹페이지에 직접 접속**(xAI API 키 사용 안 함)해서,
지정된 프롬프트로 IT·클라우드·보안 뉴스 보고서를 생성시키고, 그 응답 텍스트를 수집해
**Notion의 'AI NEWs' 페이지(`396a0b7e561a80edbb31e60dcc86148a`) 하위에
`Grok news_{YYYY-MM-DD}` 제목의 새 페이지로 업로드**하는 상주 프로세스를
기존 NAS 봄밤 컨테이너에 추가한다.

- 웹 자동화 방식은 기존 리디 수집(`ridi_collector.py`)과 동일한 Playwright 패턴을 따른다.
  저장된 로그인 세션(브라우저 프로필 폴더)으로 로그인 상태를 유지하고, Cloudflare
  (봇 차단 대기 화면) 통과 대기도 같은 방식이다.
- 원래 프롬프트는 Grok에게 Notion MCP 도구로 직접 업로드하라고 지시하지만, 자동화된
  웹 세션에서는 Grok이 MCP(외부 도구 연결)를 쓸 수 없으므로 **그 부분은 제거·치환**하고,
  Notion 업로드는 파이썬(`news_notion.py`의 신규 함수)이 담당한다.
- 범위는 **Notion 업로드까지**. 텔레그램으로 보고서를 중계하는 것은 이번 범위 밖이다.
  단, 기존 `news_bot.py` 패턴대로 **실패 시 소유자에게 텔레그램 실패 알림 1건**은 보낸다
  (무인 NAS에서 조용히 죽는 걸 막기 위해 — 성공 알림은 보내지 않음).

### 성공 기준 (무엇이 되면 완료인가)
1. 로컬 Docker 컨테이너에서 07:00(테스트 시엔 몇 분 뒤 시각)에 자동 실행되어,
   grok.com에서 보고서를 받아 Notion 하위 페이지가 실제로 생성된다.
2. 로그인 세션 만료·Cloudflare 미통과·응답 미완성·Notion 오류 등 어떤 실패에도
   프로세스가 멈추거나(`input()` hang) 죽지 않고, 실패 알림 후 다음 날을 기다린다.
3. 기존 봄밤 봇(`bot.py`)·뉴스봇(`news_bot.py`) 동작에 아무 영향이 없다
   (같은 컨테이너, 별도 프로세스·별도 브라우저 프로필·별도 설정 파일).

### 건드리지 않는 것 (명시)
- `ridi_collector.py`, `bot.py`, `news_bot.py`, `notify.py` — **수정 없음**.
  `news_grok.py`는 `notify.notify_telegram`과 `news_notion`의 신규 함수만 import한다.
  `ridi_collector`는 import하지 않는다(모듈 import 시점에 settings.json/account.json 로드,
  리디 전용 전역값 `PROFILE_DIR`/`HEADLESS` 바인딩 등 부작용이 있어, 필요한 소형 헬퍼
  두 개(Cloudflare 대기, persistent context 구동)는 `news_grok.py`에 자체 구현한다.
  중복은 각 12줄 내외로 결합도 제거 이득이 더 크다).
- `settings.json`, `news_settings.json` — **스키마 변경 없음**. Grok 설정은 별도 파일
  (`grok_settings.json`, §3.5에 근거 서술).
- `api_keys.json` — 스키마 변경 없음. 기존 `notion` 키만 사용(xAI 키 불필요 — 사용자 확정).
- 텔레그램 명령(`/run` 등)으로 Grok 수집을 수동 트리거하는 기능 — 범위 밖(최소 구현).

## 3. 상세 설계

### 3.1 실행 흐름 전체

```
[news_grok.py 상주 프로세스 — 시계만 보는 루프, getUpdates 절대 호출 안 함(409 방지)]
  20초마다:
    grok_settings.json 로드 (검증 포함)
    if 현재시각 >= daily_time(기본 07:00) and last_run_date != 오늘:
        last_run_date = 오늘 기록·저장   ← 실행 '시작 직후' 기록(news_bot과 동일,
                                          브라우저 재실행 반복 방지)
        run_grok_report() 실행

[run_grok_report() — 1회 실행 본체]
  1. api_keys.json에서 notion 토큰 확인 → 없으면 실패 알림 후 종료
  2. GROK_PROFILE 폴더의 Singleton* 잔여 락 삭제(함정 4 예방)
  3. Playwright persistent context 구동 (프로필=GROK_PROFILE, headless=HEADLESS 환경변수,
     리디와 동일한 안정화 플래그 --no-sandbox --disable-dev-shm-usage 등,
     clipboard-read 권한 부여 — §3.3 복사 버튼 전략용)
  4. https://grok.com 접속 → Cloudflare 대기(최대 90초, 리디 패턴 재사용)
  5. 로그인 상태 확인 → 미로그인(입력창 없음/로그인 화면)이면
     "이번 실행 건너뜀" + 실패 알림(함정 3 — input() 금지) 후 브라우저 닫고 종료
  6. grok_prompt.txt 로드 → {현재 날짜}를 오늘(YYYY-MM-DD)로 치환 → 입력창에 입력 → 전송
  7. 응답 완료 감지 (§3.4 — 안정화 감시 + 하드 타임아웃 20분)
  8. 응답 텍스트 수집 (복사 버튼→클립보드 우선, innerText 폴백)
  9. 최소 품질 검사(길이·필수 헤딩) → 미달이면 실패 알림 후 종료
 10. 원문 백업 저장 — OUTPUT_DIR/grok_news_{YYYY-MM-DD}.md
     (Notion 업로드가 실패해도 보고서가 유실되지 않도록. work 폴더는 NAS에 마운트됨)
 11. 브라우저 닫기 (Notion 업로드 전에 닫아 메모리 반환 — 2GB RAM 배려)
 12. markdown_to_blocks()로 변환 → create_report_page()로
     parent(396a0b7e…) 하위에 "Grok news_{YYYY-MM-DD}" 페이지 생성
 13. 성공 로그. 실패 시 어느 단계든 소유자에게 실패 알림 1건(원인 문구 포함)
```

### 3.3 프롬프트 처리와 응답 텍스트 수집

**프롬프트 파일(`grok_prompt.txt`)** — 원문(`grok_prompt_원문.md`) 기반으로 두 군데만 수정.

- 원문 "## 5. Notion 업로드 규칙 (매우 중요)" 섹션 **전체 삭제** 후 아래로 **치환**.

  ```
  ## 5. 출력 규칙 (매우 중요)
  - 보고서 전문을 마크다운 텍스트로만 출력한다. 업로드·도구 호출·파일 저장을 시도하지 않는다.
  - 보고서 외의 인사말·설명·후속 질문을 덧붙이지 않는다.
  - 첫 줄은 반드시 `# Grok news_{현재 날짜}` 형식의 제목으로 시작한다.
  - 섹션 제목은 `##`, 뉴스 항목 필드는 `* 제목:` 형태의 불릿으로 쓴다.
  ```

- 마지막 줄 "…위 Notion 페이지 하위에 업로드해줘." →
  "이제 {현재 날짜} 기준으로 보고서를 마크다운으로 작성해줘."

- `{현재 날짜}` 치환: 파이썬이 실행 시 `YYYY-MM-DD`(예: 2026-07-09)로 일괄 치환.
  Notion 페이지 제목은 Grok 출력에서 파싱하지 않고 **파이썬이 직접**
  `Grok news_{YYYY-MM-DD}`로 확정한다(모델 출력 흔들림에 무관하게 제목 보장).

**응답 텍스트 수집 — 2단계 전략.**
- 1순위: 응답 말미에 나타나는 **복사(Copy) 버튼 클릭 → `navigator.clipboard.readText()`**.
  Grok류 챗 UI의 복사 버튼은 통상 **마크다운 원문**을 복사하므로 헤딩(`#`)·불릿·링크가
  보존돼 Notion 블록 변환 품질이 좋다. context 생성 시 `permissions=["clipboard-read"]` 필요.
- 2순위(폴백): 응답 컨테이너의 `innerText` 수집. 이 경우 렌더링된 텍스트라 `#` 마커가
  사라져 헤딩이 문단(paragraph)으로 강등된다 — 내용은 보존되므로 "업로드는 되지만
  서식이 밋밋한" 수준으로 열화. 이 한계를 수용한다(§6 미해결 질문에 실 DOM 확인 필요 명시).

### 3.4 Grok 응답 완료 감지 전략 (DOM 미지 전제 — 후보와 리스크)

Grok은 웹검색을 하며 수 분간 스트리밍 생성하므로 "언제 끝났는가"가 이 설계의 최대 리스크다.
실제 DOM을 사전에 알 수 없으므로 **신호 후보를 조합**하고, 구현 단계에서 실 DOM으로 확정한다.

| 후보 신호 | 방법 | 리스크 |
|---|---|---|
| **A. 텍스트 안정화(주 전략)** | 5초 간격으로 응답 영역 텍스트 길이+해시를 폴링. **60초 연속 무변화**면 완료로 간주 | Grok이 중간에 '생각/검색 중'으로 60초 이상 멈추면 조기 완료 오판. 안정화 창을 60초로 길게 잡고 B·C로 교차 확인 |
| B. 정지(Stop) 버튼 소실 | 생성 중 표시되는 정지 버튼(`aria-label` 등)이 사라지면 완료 | 셀렉터가 CSS-in-JS 해시일 수 있음 — 함정 10 교훈대로 **클래스명 의존 금지**, aria-label·role·svg 형태 등 안정 속성만 후보 |
| C. 입력창 재활성화 | 생성 중 비활성화됐던 textarea/전송 버튼이 다시 활성화 | UI에 따라 생성 중에도 활성일 수 있음 |
| D. 하드 타임아웃 | **20분** 경과 시 무조건 중단 | 20분 내 미완이면 그 시점 텍스트를 버리고 실패 처리(부분 보고서 업로드 안 함) |

완료 판정 후 **품질 게이트**를 추가로 통과해야 업로드한다.
- 최소 길이(예: 1,500자 미만이면 실패 — 오류 메시지·거절 응답 걸러냄)
- 필수 헤딩 존재(예: "개요" 또는 "Executive Summary" 문자열 포함)

구현 원칙: A(안정화)+D(타임아웃)를 **확정 로직**으로 하고, B·C는 실 DOM 확인 후
사용 가능하면 "안정화 창 단축용 보조 신호"로만 쓴다(없어도 동작하게).

### 3.5 설정 스키마 — 별도 파일 `grok_settings.json` (OUTPUT_DIR = work 폴더)

**기존 `news_settings.json`에 통합하지 않는 근거(중요).**
`news_bot.py`의 `load_news_settings()`는 `DEFAULT_SETTINGS`에 있는 키만 남기고
(`raw = {k: v … if k in DEFAULT_SETTINGS}`) 교정 시 그 결과로 **파일을 덮어쓴다**.
즉 grok 키를 같은 파일에 넣으면 news_bot이 저장하는 순간 **grok 키가 삭제**된다.
이를 피하려면 news_bot을 수정해야 하는데(수술적 수정 원칙 위반), 두 상주 프로세스가
한 파일을 20초마다 읽고 쓰는 경합(race)도 생긴다. → **파일 분리가 최소 변경**이다.

```json
{
  "daily_time": "07:00",
  "last_run_date": ""
}
```

- 검증 규칙은 `news_bot.load_news_settings`와 동일(HH:MM 제로패딩 강제, 아니면 07:00 교정,
  last_run_date는 YYYY-MM-DD 아니면 "" 처리, 파일 없으면 기본값으로 자동 생성).
- 파일 위치가 work 볼륨이므로 NAS에서 File Station으로 직접 수정 가능
  (단일 파일 마운트 금지 함정 5와 무관 — '폴더' 마운트 안의 파일).
- `api_keys.json`은 그대로 사용, `notion` 키만 읽는다.

### 3.6 `news_notion.py` 신규 함수 — Notion 페이지 생성 API

Notion API(버전 2022-06-28, 기존 상수 재사용)로 다음 두 호출을 쓴다. urllib만 사용(기존 원칙).

**① 페이지 생성 — `POST https://api.notion.com/v1/pages`**

```json
{
```

### 3.7 마크다운→Notion 블록 변환 범위 (최소 구현)

| 마크다운 | Notion 블록 | 비고 |
|---|---|---|
| `# ` / `## ` / `### ` | heading_1 / 2 / 3 | `####` 이하는 heading_3으로 승격 |
| `- ` `* ` 불릿 | bulleted_list_item | 들여쓰기(중첩)는 무시하고 평탄화 |
| `1. ` 등 숫자 불릿 | numbered_list_item | |
| `---` | divider | |
| 그 외 비어있지 않은 줄 | paragraph | |
| 인라인 `[텍스트](URL)` | rich_text의 `link` | 출처 링크 보존(핵심 요구) |
| 인라인 `**굵게**` | rich_text `annotations.bold` | 정규식 1개로 처리 가능 — 포함 |

**지원하지 않는 것(한계 명시)**: 표, 코드블록, 이미지, 인용(`>`), 체크박스, 중첩 목록 구조,
기울임·취소선. 해당 줄은 원문 그대로 paragraph로 들어간다. 프롬프트가 요구하는 보고서
형식(헤딩+불릿+링크)에는 위 범위로 충분하다.

### 3.8 `entrypoint.sh` — 상주 프로세스 추가 (별도 프로세스 방식 채택)

news_bot에 통합하지 않고 **별도 프로세스**로 둔다. 근거 — news_bot 루프는 20초 주기의
가벼운 API 호출인데 Grok 실행은 브라우저를 띄워 최장 20분을 점유한다. 통합하면 Grok
실행 중 뉴스 중계가 지연되고, 한쪽 크래시가 다른 쪽을 죽인다. 기존 news_bot 블록과
동일한 패턴으로 한 블록만 추가한다.

```bash
# Grok 뉴스 보고서 봇 — 죽으면 30초 후 자동 재기동
(
  while true; do
    python -u /app/news_grok.py || true
    echo "[grok] Grok봇 프로세스 종료 감지 — 30초 후 재기동"
    sleep 30
  done
) &
```

Xvfb(:99)는 entrypoint가 이미 띄워 두므로(함정 2 — `xvfb-run` 래퍼 금지 방식) Grok
브라우저도 같은 DISPLAY를 그대로 쓴다. 리디 브라우저와 같은 X 서버를 공유해도 문제없다
(서로 다른 창일 뿐이며, 시간대 분리로 동시 실행 자체가 드묾 — §3.9).

### 3.9 브라우저 프로필 분리·메모리·시간대

- **프로필 분리(필수)**: Grok 세션은 `GROK_PROFILE=/app/.grok_profile`(신규 ENV),
  리디는 기존 `BROWSER_PROFILE=/app/.browser_data_dir`. 같은 프로필을 두 프로세스가
  공유하면 Chromium `Singleton*` 락 충돌(함정 4)과 세션 오염이 나므로 **디렉토리·볼륨
  모두 분리**한다.
  - docker-compose 추가 볼륨: `- /volume1/docker/bombam/grok_profile:/app/.grok_profile`
  - 로컬 테스트: `-v "$(pwd)/local_test/grok_profile:/app/.grok_profile"`
  - 매 실행 시작 시 `GROK_PROFILE` 안의 `SingletonLock/SingletonCookie/SingletonSocket`을
    선제 삭제(그 프로필은 news_grok만 쓰므로 안전).
- **최초 로그인 부트스트랩(1회 수동)**: 리디와 같은 방식 — 맥에서
  `GROK_PROFILE=./local_test/grok_profile HEADLESS=0 python news_grok.py --once`로 창을 띄워
  사람이 grok.com에 로그인하면 세션이 프로필 폴더에 저장되고, 그 폴더를 NAS
  `grok_profile/`에 복사한다. (단, `--once`는 무인 분기라 로그인 대기를 안 하므로,
  부트스트랩 편의를 위해 리디의 함정 3 패턴 그대로 **`sys.stdin.isatty()`이고
  `HEADLESS=0`일 때만** "창에서 로그인 후 Enter" 대기를 넣는다. 컨테이너/NAS에서는
  isatty가 아니므로 절대 걸리지 않는다.)
- **메모리(2GB RAM)**: Chromium 1개가 순간 수백 MB를 쓰므로 리디 수집과 동시 구동을
  피해야 한다. 스케줄이 이를 보장한다 — Grok **07:00**(최장 07:20경 종료),
  뉴스봇 중계 10:00(브라우저 없음), 리디 자동 수집 **11:00**. 겹침 없음.
  잔여 리스크: 사용자가 07:00~07:20 사이에 텔레그램 `/run`(리디 수동 수집)을 누르면
  Chromium 2개가 동시에 뜬다. 상호 잠금(lock) 장치는 최소 구현 원칙상 넣지 않고,
  운영 수칙("아침 7시대에는 /run 자제")으로 처리한다(§6에 검토 요청).
  추가로 Grok 브라우저는 **Notion 업로드 전에 닫아** 점유 시간을 최소화한다.
- `shm_size: "1gb"`(기존 compose 설정)와 `--disable-dev-shm-usage` 플래그를 그대로 따른다.

| # | 함정 | 이 설계의 대응 |

| 8 | 자동 실행 기록·`>=` 비교 | `now.strftime("%H:%M") >= daily_time and last_run_date != 오늘` (news_bot과 동일). `--once` 수동 실행은 last_run_date를 기록하지 않음 |

| 시나리오 | 동작 |
|---|---|
| notion 토큰 없음 | 외부 호출 전에 즉시 실패 알림 → 종료(다음 날 재시도). news_bot과 동일 패턴 |
| Cloudflare 90초 미통과 | 브라우저 닫고 실패 알림("Cloudflare 미통과 — 내일 자동 재시도") |

| 컨테이너 재시작(07:00 이후) | `>=` 비교 덕에 재시작 직후 루프에서 실행됨. 이미 실행한 날이면 `last_run_date`가 같아 중복 실행 없음. 단 "기록 직후·업로드 전" 크래시면 그날은 건너뜀(과금·중복 페이지 방지 우선 — news_bot §3.4와 같은 선택) |
| 같은 날 `--once` 반복 실행 | Notion에 동일 제목 페이지가 중복 생성될 수 있음(Notion은 중복 제목 허용). 테스트 후 수동 삭제 — 자동 중복 검사 없음(최소 구현) |
| 텔레그램 실패 알림 자체가 실패 | 로그만 남기고 계속(news_bot 패턴) |

- [사용자] **Grok 계정·요금제** — grok.com에 사용할 계정이 준비돼 있는가? 로그인 수단이
  무엇인가(X 계정 / Google / 이메일)? 무료 요금제면 웹검색 포함 장문 보고서 생성에
  사용량 제한이 걸릴 수 있는데 허용 가능한가?
- [사용자] **세션 이식성 리스크 수용** — 맥에서 만든 로그인 세션을 NAS로 복사하는 방식은
  리디에서는 검증됐지만, grok.com이 IP·기기 변화에 더 민감해 세션이 무효화될 수 있다.
  그 경우 "실패 알림 → 맥에서 재로그인 → 프로필 재복사" 운영을 수용하는가?
  (리디와 동일한 운영 부담 수준)
- [사용자] **07:00~07:20 사이 `/run` 자제 수칙** — 이 시간대에 리디 수동 수집을 누르면
  브라우저 2개 동시 구동으로 2GB RAM이 빠듯해진다. 코드 잠금 없이 운영 수칙으로
  갈음해도 되는가?
- [검토자] **응답 완료 감지 파라미터** — "5초 폴링 + 60초 무변화 + 20분 하드 타임아웃 +
  품질 게이트(1,500자·필수 헤딩)" 조합이 적절한지, 안정화 창(60초)이 Grok의 '검색 중'
  멈춤 대비 충분한지 의견 요청.

  안정적으로 동작한다고 보는지, 아니면 처음부터 innerText 폴백을 주 전략으로 삼고
  마크다운 서식 열화를 수용하는 편이 나은지.

  PATCH)하는 정리 로직을 넣을지, 백업 .md 존재를 근거로 남겨 두는 현 설계가 나은지.
- [검토자] **grok.com DOM 확정 절차** — 구현 단계에서 셀렉터(입력창·전송·정지·복사 버튼)를
  실 브라우저로 확인해 확정하는 것 외에, 설계 시점에 더 줄일 수 있는 불확실성이 있는지.
